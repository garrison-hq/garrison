# M2.1 context — Claude Code invocation

Binding constraints for the M2.1 milestone: swap the M1 fake subprocess for real Claude Code invocation, with Postgres MCP wiring, shipping end-to-end functional.

**Status**: active milestone. Spike complete at `docs/research/m2-spike.md`. Spec not yet drafted; this context file is binding input to `/speckit.specify`.

**Precedence**: this file is binding for M2.1 operational work. RATIONALE.md and ARCHITECTURE.md remain higher authorities (see AGENTS.md). Where this file and the spike doc describe the same tool behavior, this file is operational rule and the spike is ground truth — if they conflict, the spike is correct and this file has a bug.

---

## What M2.1 produces

A supervisor binary that, on receiving `work.ticket.created.engineering.todo` via pg_notify:

1. Spawns a real Claude Code subprocess (not the M1 fake subprocess) with:
   - The engineering department's `agent.md` as `--system-prompt`
   - A Postgres MCP server wired via `--mcp-config` and `--strict-mcp-config` (read-only)
   - `--output-format stream-json --verbose`, `--no-session-persistence`, a `--model` flag, and a `--max-budget-usd` cap
   - `SysProcAttr.Setpgid = true` (process group isolation per AGENTS.md concurrency rule 7)
2. Parses the `system`/`init` NDJSON event and confirms `mcp_servers[0].status == "connected"` for the Postgres MCP; bails otherwise
3. Parses subsequent NDJSON events: `assistant`, `user` (tool_result), `rate_limit_event`, and the terminal `result`
4. Lets the agent read its ticket from Postgres via the MCP, write `hello.txt` containing the ticket ID to the engineering workspace, and exit cleanly
5. After the Claude subprocess exits with a successful `result`, the supervisor writes the ticket transition (`column_slug = 'done'`) and the `agent_instances` row (`status = 'succeeded'`, `total_cost_usd`, `exit_reason = 'completed'`)

End-to-end acceptance: one `INSERT INTO tickets` at column `todo` for the engineering department produces, within a timeout, one `hello.txt` file in the engineering workspace with the ticket ID as its content, one `agent_instances` row with `status='succeeded'`, one `ticket_transitions` row for `todo → done`, and cost data captured.

## What is explicitly out of scope

Listed so that agents reading this file don't drift into them:

- **MemPalace integration** — that's M2.2. M2.1 agents do not read from or write to MemPalace. No `mempalace_*` tools in the MCP config. No wake-up context injection. No diary entries. No KG triples.
- **Agents writing to Postgres state** — M2.1 Postgres MCP is read-only. The supervisor is the sole writer of `tickets`, `ticket_transitions`, and `agent_instances`. If the agent needs to "mark done," it signals success by exiting with a successful `result` event; the supervisor interprets that and writes the state.
- **Multiple departments, multiple roles** — M2.1 has exactly one department (`engineering`) and exactly one agent role (the engineer writing `hello.txt`). The CTO role, the frontend/backend split, the QA role — all M3+ or later.
- **Hygiene checks** — M2.1 has no expected-writes contract yet because there are no MemPalace writes to check. The `hygiene_status` column on `ticket_transitions` stays NULL for M2.1 transitions.
- **Agent-spawned tickets** — the M2.1 agent produces a file and exits. It does not create further tickets.
- **Dashboard** — M3.
- **CEO chat** — M5.
- **skills.sh** — the engineering agent's `agent.md` does not reference any skills. Skills arrive in M7.
- **Rate-limit backoff logic** — M2.1 parses the `rate_limit_event` and logs it, but does not implement supervisor-side backoff. That's M6 alongside cost accounting.
- **Cost-based throttling** — M2.1 captures `total_cost_usd` per invocation; it does not make dispatch decisions based on aggregated cost. M6.
- **Multi-model support** — M2.1 uses a single configured model (operator choice, probably `claude-haiku-4-5-20251001` for cost during M2.1 validation). Per-agent or per-spawn model overrides are M4+.
- **Secret management** — M2.1 uses Claude Code's default OAuth path (`~/.claude/` keychain). `--bare` mode and `ANTHROPIC_API_KEY` injection are deferred to M6.
- **Tool permission confirmation** — M2.1 uses `--permission-mode bypassPermissions` (or the current equivalent) so the agent can write `hello.txt` without a prompt. Fine-grained permission policies are M4+.

## The stack (locked)

Additions to the M1 locked dependency list:

- `claude` binary on `$PATH` (the operator's Claude Code install). M2.1 does not vendor or wrap the binary; the supervisor invokes it via `exec.CommandContext`.
- One Postgres MCP server. The spec must pick which implementation — candidates include the official Anthropic Postgres MCP, community forks, or a minimal in-tree server. This is a binding decision for `/speckit.specify` to resolve; see "Binding questions for the spec" below.

## Spawn contract (binding)

Every Claude Code subprocess spawned by the M2.1 supervisor is launched with the following flags and env. This is binding; the spec may not deviate without updating this file.

**Argv**:
```
claude -p "<task description>"
  --output-format stream-json --verbose
  --no-session-persistence
  --model <model from agent config>
  --max-budget-usd <per-invocation cap>
  --mcp-config <per-invocation mcp-config-<instance_id>.json>
  --strict-mcp-config
  --system-prompt "<contents of agents/engineer.md>"
  --permission-mode bypassPermissions
```

**`exec.Cmd` configuration**:
- `Dir` = the engineering department's `workspace_path` (from the department config)
- `SysProcAttr.Setpgid = true` (AGENTS.md concurrency rule 7)
- `Stdin` unused; inherit `/dev/null`
- `Stdout` piped to the supervisor's line-parser goroutine (NDJSON)
- `Stderr` piped to slog with `stream="claude_stderr"`

**Per-invocation MCP config file**: the supervisor writes a temporary file at spawn time containing the Postgres MCP server definition, keyed to the agent_instance ID. Deleted on subprocess exit. The file is not shared across invocations — each spawn gets its own to prevent cross-invocation config contamination.

**Env**: inherit the supervisor's environment. Do not inject secrets into argv or env beyond what Claude Code naturally expects (`HOME`, `PATH`, etc.).

## Stream-json event routing (binding)

The supervisor parses Claude's stdout as NDJSON, one event per line. Route by `type` (and `subtype` where present):

| Event | Action |
|---|---|
| `system`/`init` | Record `session_id`, `tools[]`, `mcp_servers[]`, `cwd`. **Verify every entry in `mcp_servers[]` has `status == "connected"`.** Any other value (including `"failed"`, `"needs-auth"`, or an unknown string) causes immediate process-group SIGTERM and a failed agent_instance row with `exit_reason = "mcp_<name>_<status>"`. See AGENTS.md and spike §2.5 / §A. |
| `assistant` | Log content blocks (thinking, text, tool_use) to slog with structured fields (`agent_instance_id`, `turn_index`). Do not parse or interpret — observational logging only. |
| `user` | Log tool_result blocks. If `is_error == true`, log at warn level with the error detail. Do not bail on tool errors; the agent may recover on the next turn. |
| `rate_limit_event` | Log at warn level with all fields. Emit a slog entry that can be grep'd later. No dispatch behavior change in M2.1 — just visibility. |
| `system`/`task_started` | Log at debug level. Internal to Claude's own task system; no supervisor action. |
| `result` | Terminal event. Record `duration_ms`, `total_cost_usd`, `terminal_reason`, `permission_denials`, full result text. This marks the end of a successful run. |
| Unknown event `type` | Log at warn level with the raw line. Do not crash. Forward compatibility matters; future Claude versions may add event types. |

The supervisor bails on the first NDJSON parse error (line is not valid JSON). Malformed Claude output is treated as a subprocess failure.

## Termination contract (binding)

The supervisor kills a Claude subprocess in three scenarios:

1. **MCP health failure** (from init event): SIGTERM to process group immediately, 2-second grace, SIGKILL to group. `exit_reason = "mcp_<server>_<status>"`.
2. **Supervisor timeout** (subprocess context expires): SIGTERM to process group, 5-second grace (longer than MCP bail to allow in-flight turn to emit a `result`), SIGKILL to group. `exit_reason = "timeout"`.
3. **Supervisor shutdown** (root context cancelled): same as timeout but `exit_reason = "supervisor_shutdown"`. Terminal writes use `context.WithoutCancel(ctx)` per M1 retro §2 and AGENTS.md concurrency rule 6.

Signal the group: `syscall.Kill(-pgid, SIGTERM)` then `syscall.Kill(-pgid, SIGKILL)` after grace. Never signal the PID alone. See spike §2.7 and AGENTS.md concurrency rule 7.

## Agent-instance state machine (M2.1 additions to M1)

Fields on `agent_instances` that matter in M2.1:

- `status`: `running`, `succeeded`, `failed`, `timeout` (M1 values; semantics unchanged)
- `exit_reason`: M1 values plus M2.1 additions:
  - `completed` (Claude emitted a `result` with `terminal_reason == "completed"` and `is_error == false`)
  - `claude_error` (Claude emitted a `result` with `is_error == true`)
  - `mcp_<server>_<status>` (MCP health check failed on init event)
  - `parse_error` (malformed NDJSON from Claude's stdout)
  - `timeout` (subprocess context expired)
  - `supervisor_shutdown`
- `total_cost_usd`: new column, added in the M2.1 migration. Nullable. Populated from the terminal `result` event.
- `pid`: populated at `cmd.Start()` completion (resolves the M1 retro §4 dead column).

Migration: a single forward migration adds `total_cost_usd NUMERIC(10,6)` to `agent_instances` and optionally renames the column to match. The M1 retro noted the `pid` column was dead; M2.1 also backfills the spawn logic so `pid` is written. Both changes share one migration file.

## Postgres MCP (binding shape, not binding implementation)

The Postgres MCP server exposes **read-only** query access to the Garrison database. M2.1 agents use it to read their ticket; they do not use it to mutate state.

Required tool behavior (whatever implementation is chosen must provide this):
- A `query` or `execute_query` tool accepting a SQL string
- Read-only enforcement: the server must reject any statement that is not `SELECT` or `EXPLAIN`. If the chosen implementation doesn't enforce read-only, wrap it in a supervisor-authored restriction layer (Postgres role with SELECT-only permissions is sufficient).
- Parameter binding: either via the tool's parameter array or via the agent writing parameterized SQL with literal values. M2.1 doesn't need sophisticated binding.

The engineering agent's `agent.md` (located at `examples/agents/engineer.md` or similar) instructs the agent to:
1. Read the ticket via `SELECT id, objective, acceptance_criteria, metadata FROM work.tickets WHERE id = '<ticket_id>'`
2. Write `hello.txt` with the ticket ID as its contents
3. Exit

The `<ticket_id>` is injected into the task description (the `-p` argument), not a secret. Agents see the ticket ID in their prompt and use it to query the row.

## Workspace (M2.1-minimal)

The engineering department has one workspace directory (e.g. `/workspaces/engineering`). M2.1 does not touch git; the workspace is a plain directory the supervisor and Claude can both write to.

- Supervisor creates the workspace if missing on startup (idempotent `mkdir -p`)
- Claude runs with `cmd.Dir = workspace_path`
- The engineering agent writes `hello.txt` into the workspace
- No git init, no commits, no `.gitignore` management — all deferred to later milestones

## Acceptance criteria for M2.1

Each is a concrete, verifiable step. All ten must pass against a freshly-migrated Postgres and the release Docker image before M2.1 ships.

1. Fresh DB migration produces `departments`, `agents`, `agent_instances` tables with M2.1 columns; seeded engineering department and engineer agent rows exist.
2. Supervisor starts cleanly; `/health` returns 200; slog shows supervisor is LISTENing on `work.ticket.created.engineering.todo`.
3. `INSERT INTO work.tickets (..., department_id='<engineering-id>', column_slug='todo', origin='ui', objective='write hello', acceptance_criteria='file exists')` fires `pg_notify`.
4. Within 2 seconds, supervisor logs show a `Spawn` event for the engineer agent, a Claude subprocess is running, and `agent_instances` has a `running` row with a non-null `pid`.
5. Supervisor logs show the parsed `system`/`init` event, `mcp_servers` contains one entry `{name: "postgres", status: "connected"}`, Postgres MCP tools (at least `query`) present in the init event's `tools[]`.
6. Within the subprocess timeout (30 seconds default), Claude emits a `result` event with `terminal_reason: "completed"`, `is_error: false`, non-null `total_cost_usd`.
7. File `<workspace>/hello.txt` exists and contains the ticket ID (exact string match, allowing trailing newline).
8. `agent_instances` row updated: `status='succeeded'`, `exit_reason='completed'`, `finished_at` set, `total_cost_usd` populated.
9. `ticket_transitions` row exists: `from_column='todo'`, `to_column='done'`, `triggered_by_agent_instance_id` set, `hygiene_status` is NULL (M2.1 has no hygiene contract).
10. Supervisor graceful shutdown: SIGTERM to the supervisor while an agent is in-flight causes the process group to be signaled, the in-flight Claude exits cleanly or after grace, terminal writes complete via `WithoutCancel`, no stale `running` rows remain.

Chaos tests inherited from M1 still pass (reconnect catches missed events, external SIGKILL on the subprocess marks status=failed, graceful shutdown with in-flight cleanly). One new chaos test:

11. **Broken MCP config chaos test**: write an `mcp-config-bad.json` with a nonexistent command, force a spawn using it (either via a test-only hook or a dedicated test department with a bad config), verify the supervisor kills the Claude process group within 2 seconds of the init event, `agent_instances` row has `status='failed'`, `exit_reason='mcp_postgres_failed'` (or similar depending on what the bad server is named).

## Binding questions for `/speckit.specify`

Questions the spec must decide, not defer to clarify or plan. Each one has a default the spec may use if there's no strong reason to choose otherwise.

1. **Which Postgres MCP server implementation?** Candidates: the reference Postgres MCP from Anthropic's MCP reference implementations; a community fork (e.g. one of the popular ones on skills.sh or GitHub); a minimal in-tree Go server. Criteria: read-only support, stdio transport, active maintenance, Fedora+Linux compatibility. **Default**: Anthropic's reference Postgres MCP if it exists and supports read-only; otherwise a minimal in-tree Go server (~150 lines).

2. **How is Postgres read-only enforced?** Options: the MCP server's own read-only flag; a Postgres role with `GRANT SELECT ... TO garrison_agent` and revoke on everything else; supervisor-side SQL validation. **Default**: Postgres role. Role-based auth is robust, independent of MCP server behavior, and gives the supervisor a defensible story when M2.2 adds MemPalace (MemPalace writes use a different role/auth).

3. **Subprocess timeout default**: how long can a Claude Code agent run before the supervisor kills it? **Default**: 60 seconds for M2.1. The hello-world task completes in <10s empirically; 60s leaves plenty of slack without letting runaways cost money. Per-agent overrides are deferred.

4. **Per-invocation budget cap** (`--max-budget-usd`): **Default**: $0.05 per invocation. M2.1's hello-world task costs ~$0.001 with cache hits; $0.05 is 50x headroom without being reckless. Per-role budgets are deferred.

5. **Where does the per-invocation MCP config file live?** Options: `/tmp/garrison-mcp-<instance_id>.json`; the workspace; a supervisor-owned state dir. **Default**: a supervisor-owned state dir (e.g. `/var/lib/garrison/mcp/` or an operator-config path), cleaned up on subprocess exit via `defer os.Remove`. Workspaces are for agent artifacts, not supervisor scratch.

6. **How is the engineer agent's `agent.md` delivered?** Options: inline in the database `agents.agent_md` column; a file at a fixed path in the repo (`examples/agents/engineer.md`); both. **Default**: inline in the database (matches the architecture's "configs are first-class data in Postgres"). The `examples/` version is for operator bootstrap and open-source replication.

7. **Concurrency cap for engineering in M2.1**: **Default**: 1. M2.1 is proving the spawn path works, not testing concurrency under load. A cap of 1 makes test timing deterministic. Bumped to the eventual target (probably 3) in M3 or M4.

## Testing requirements

Unit tests:
- NDJSON event router: correct dispatch for each event type, warn-on-unknown-type behavior, bail-on-parse-error behavior
- MCP health check: `connected` passes, `failed`/`needs-auth`/unknown all trigger bail with correct `exit_reason`
- Termination: signal-to-group happens on context cancel, timeout, and MCP-bail paths
- Cost parsing: `total_cost_usd` captured correctly from `result` event; null if `result` malformed
- Config substitution for the per-invocation MCP config file: correct agent_instance ID, correct Postgres connection string

Integration tests via testcontainers-go (M1 harness reused):
- End-to-end happy path (acceptance criteria 1–10 automated)
- Broken MCP config triggers bail (acceptance criterion 11)
- Concurrent cap enforcement (engineering cap=1, two rapid tickets, second one queued)
- Rate-limit event logged correctly (inject a synthetic `rate_limit_event` via a mock Claude binary)

Chaos tests:
- All M1 chaos tests still pass unchanged
- External SIGKILL on the Claude subprocess marks `status='failed'` with signal in `exit_reason`
- Graceful shutdown while an agent is in-flight: process group receives SIGTERM, terminal writes succeed via WithoutCancel

One new chaos test worth calling out: **what happens if the Postgres MCP server dies mid-run?** The spike didn't characterize this. If it's cheap to test (drop the MCP process, observe Claude's behavior), add it. If Claude's behavior under mid-run MCP failure is itself ambiguous, flag it as an M2.1 spec open question and handle the discovery in the retro.

## Spike findings that bind this milestone

From `docs/research/m2-spike.md`, these are the binding observations:

- §2.1 — output format and event types (binding for the event router)
- §2.3 — `--model` flag fails fast on bad model names (binding for config validation)
- §2.4 — `total_cost_usd` only in final `result` event (binding for cost capture)
- §2.5 — `--strict-mcp-config` does NOT fail on broken servers (binding: init-parse is mandatory)
- §2.6 — `--bare` requires `ANTHROPIC_API_KEY` (informs: don't use `--bare` in M2.1)
- §2.7 — SIGTERM to PID is insufficient (binding: process group termination)
- §2.8 — `--no-session-persistence` prevents filesystem side effects (binding: always on)
- §A — init event honestly reports MCP server status (binding: check `status == "connected"`)

If the spec contradicts any of these without a documented reason, the spec is wrong.

## What we're watching for at ship

When M2.1 ships, these are the questions the retro must answer:

1. Was the init-event parsing reliable under real load? Did Claude ever emit a malformed init event, a missing `mcp_servers` field, or unexpected status values?
2. Was the process-group termination reliable? Any zombie processes, any children surviving, any grace-period edge cases?
3. Did `total_cost_usd` match what Anthropic's billing dashboard showed? Discrepancies suggest the cost capture has a gap.
4. Did the chosen Postgres MCP implementation hold up? Any crashes, any auth failures, any read-only bypasses?
5. What surprises emerged from real Claude Code that the spike didn't find? The spike ran a handful of invocations; M2.1 will run many more in testing. The retro documents what changed.
6. Did the spike pay off? Specifically: how many of M2.1's issues were prevented by spike findings vs. discovered during implement? If prevention rate is >50%, the spike discipline is validated.
