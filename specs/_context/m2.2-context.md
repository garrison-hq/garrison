## Milestone objective

Wire MemPalace into every agent spawn. The supervisor makes MemPalace available as an MCP tool to every Claude Code subprocess, provides wake-up context at spawn time, and checks hygiene writes asynchronously after transitions.

This is the milestone that validates or falsifies Garrison's memory thesis (RATIONALE §3). If the palace produces useful cross-agent memory, the downstream roadmap is extension. If it doesn't, the architecture needs revision before M3+.

## What M2.2 produces

1. **MemPalace bootstrapped at a dedicated path** (default `~/.garrison/palace/`, configurable via `GARRISON_PALACE_PATH`). Bootstrapped via `mempalace init --yes` at supervisor startup if not already present. Never in a git-tracked directory.

2. **MemPalace MCP server registered in every agent's per-invocation MCP config**, alongside the existing Postgres MCP server from M2.1. When the supervisor spawns a Claude subprocess for the engineering department, the subprocess sees two MCP servers: `postgres` (read-only) and `mempalace` (read + write).

3. **Wake-up context injection at spawn time.** Before spawning a Claude subprocess, the supervisor runs `mempalace wake-up --wing <role_wing>` and injects the ~170-token result into the Claude subprocess's prompt. The wake-up context is rendered into the `--system-prompt` argument between the agent.md and a separator. See "Wake-up injection contract" below.

4. **Expanded engineer agent.md** — the M2.1 agent.md wrote `hello.txt`. The M2.2 version executes the full completion protocol from `ARCHITECTURE.md` "MemPalace write contract" — reads the ticket from Postgres, searches the palace wing, implements, writes a diary entry, writes KG triples, transitions the ticket.

5. **Hygiene checker** — an async background process that, for every row inserted into `ticket_transitions`, verifies:
   - Was `mempalace_add_drawer` called with the ticket ID in the body during the agent_instance's run window?
   - Was at least one KG triple written that mentions this ticket?
   - Is the diary body substantive (>100 chars)?
   - Result written to `ticket_transitions.hygiene_status` as one of `'clean'`, `'missing_diary'`, `'missing_kg'`, `'thin'`, `'pending'`.

6. **A `garrison_agent_mempalace` Postgres role** following the `garrison_agent_ro` pattern from M2.1 (see "Database roles" below). Used by the hygiene checker to read MemPalace metadata without requiring full supervisor credentials.

7. **Structured logging of MemPalace tool calls.** Every `tool_use` event in stream-json with `name == "mempalace_*"` gets logged at `info` level with ticket_id, agent_instance_id, tool name, and outcome. This produces the data stream M3's hygiene dashboard will later consume.

8. **End-to-end acceptance criterion**: a single ticket routed through the engineering workflow produces a diary entry in `wing_frontend_engineer`, at least one KG triple, a `ticket_transitions` row with `hygiene_status = 'clean'`, and a completed agent_instance with `total_cost_usd` captured.

## What M2.2 does not produce

Scope boundaries. Violations are scope creep and must be rejected:

- **Dashboard UI for hygiene** — M3. M2.2 produces the `hygiene_status` data and a read-only SQL query; the UI comes later.
- **Secrets injection** — M2.3. The hygiene checker uses `garrison_agent_mempalace` credentials from supervisor env vars, same pattern as M2.1's `AgentROPassword`.
- **Multi-department workflows** — M2.2 wires engineering + one additional role (see below). Full multi-department orchestration is additive in future milestones.
- **CEO agent** — M5. The CEO's palace access is a future concern.
- **Hiring flow** — M7.
- **Agent-spawned tickets** — M8.
- **MemPalace scaling** — the palace is single-process, single-writer in M2.2. Concurrent writes from multiple agents are sequenced by MCP's protocol-level synchronization. Real multi-agent concurrency characterization happens via M2.2's chaos tests; architectural scaling is not an M2.2 concern.
- **Wake-up context personalization beyond wing** — the wake-up command accepts only `--wing`. Personalizing by ticket metadata or role history is a later optimization.
- **Rate-limit backoff** — M2.1 logs `rate_limit_event`; M2.2 does not act on it. M6.
- **Cost-based throttling** — M2.2 captures per-invocation cost (M2.1 already does this) but does not make spawn decisions based on aggregate cost. M6.

If you find yourself implementing any of these, stop and surface it as an M2.2 scope violation.

## Real-world cost baseline (from M2.1)

M2.1 shipped with a per-invocation real cost of ~$0.04 for trivial runs (vs. $0.003 in mockclaude fixtures — 13× delta). This is operative for M2.2's budget reasoning.

**Per-invocation budget cap for M2.2: $0.10 USD.** Justified as follows:
- Real M2.1 baseline: $0.04
- MemPalace overhead estimate: +$0.02 (wake-up context read, diary write, KG triples, 1-2 palace searches)
- Headroom for occasional longer runs: +$0.04
- Total cap: $0.10

The cap is enforced via Claude Code's `--max-budget-usd 0.10` flag at spawn. Exceeding the cap terminates the subprocess with `exit_reason = "budget_exceeded"`.

**Acceptance check**: M2.2's test suite must verify that a representative engineering ticket (read-palace → implement → write-palace → transition) completes within the $0.10 cap. If it doesn't, either the cap is wrong or the agent flow is inefficient; both are findings for the M2.2 retro.

**Cross-check discipline**: like M2.1, M2.2's retro must include the cost cross-check against Anthropic's billing dashboard. M2.1 deferred this; M2.2 completes it as part of the retro.

## Spawn contract (binding, extended from M2.1)

M2.1's spawn contract is preserved. M2.2 adds:

### MCP config file extensions

The per-invocation MCP config file written by `internal/mcpconfig` gains a `mempalace` entry:

```json
{
  "mcpServers": {
    "postgres": {
      "command": "/usr/local/bin/supervisor",
      "args": ["mcp", "postgres"],
      "env": {
        "GARRISON_AGENT_RO_DSN": "..."
      }
    },
    "mempalace": {
      "command": "/usr/local/bin/mempalace",
      "args": ["serve"],
      "env": {
        "MEMPALACE_PATH": "/home/garrison/.garrison/palace"
      }
    }
  }
}
```

The `mcpServers` object is additive; the shape was already multi-server-ready in M2.1 (confirmed by M2.1 retro). M2.2 just adds one key.

### Wake-up injection contract

Before spawning the subprocess, the supervisor runs:

```bash
mempalace wake-up --wing <role_wing> --max-tokens 200
```

Captures stdout. Template for the `--system-prompt` argument becomes:

```
<agent.md content>

---

## Wake-up context

<wake-up stdout>

---

## This turn

You have been spawned to work ticket <ticket_id>. Read it, then execute your completion protocol.
```

The wake-up command's timeout is 2 seconds. If it exits non-zero or times out, the spawn proceeds without wake-up context (logged at warn level, `wake_up_status = 'failed'` on the agent_instance row). The spawn does NOT bail — missing wake-up is not a hard failure.

### MCP health check (unchanged from M2.1, now applies to MemPalace too)

The supervisor's init-event parser checks `mcp_servers[]` for both `postgres` and `mempalace`. Either one having `status != "connected"` causes the supervisor to kill the process group and record `exit_reason = "mcp_mempalace_failed"` (or `"mcp_postgres_failed"` as in M2.1).

Fail-closed discipline unchanged. Unknown status values are treated as failures.

### Subprocess management (unchanged rules)

AGENTS.md concurrency rule 7 (process group) and rule 8 (pipeline-drain-before-Wait, from M2.1 retro §1) apply without modification. M2.2 adds no new subprocesses directly managed by the supervisor; the hygiene checker runs in-process as a goroutine, not as a subprocess.

## Database roles

### `garrison_agent_ro` (existing, unchanged from M2.1)

Read-only access to the M2.1 read surface. Password set post-migrate per `docs/ops-checklist.md`. Used by `internal/pgmcp` for Postgres MCP queries.

### `garrison_agent_mempalace` (new)

Read-write access to MemPalace-related tables only. Follows the same pattern as `garrison_agent_ro`:

1. Created by M2.2 migration: `CREATE ROLE garrison_agent_mempalace LOGIN;`
2. Granted permissions on MemPalace-related tables (the hygiene checker reads `ticket_transitions`, `agent_instances`, and any MemPalace-metadata tables introduced by M2.2's schema additions).
3. Password NOT set in the migration. Operators run `ALTER ROLE garrison_agent_mempalace PASSWORD '...'` post-migrate, per `docs/ops-checklist.md`.
4. Password consumed by the supervisor via env var `GARRISON_AGENT_MEMPALACE_PASSWORD`, composed into a DSN at startup via `config.AgentMempalaceDSN()` (parallel to M2.1's `config.AgentRODSN()`).

This pattern is deliberately parallel to M2.1's. A future milestone introducing another scoped role follows the same three-step discipline. M2.3's Infisical bootstrap secrets (`ENCRYPTION_KEY`, `AUTH_SECRET`) will follow the same shape — ops checklist documents the post-deploy step, supervisor reads from env, config composes the secret at startup.

## Hygiene checker (binding shape)

Runs as a goroutine inside the supervisor process. Listens on `pg_notify('work.ticket.transitioned.*.*')`. For each notification:

1. Wait 5 seconds (gives the agent's post-transition writes time to land in MemPalace).
2. Query MemPalace for:
   - Drawers in `<role_wing>` where the body mentions the ticket ID and `completed_at` is within the agent_instance's run window.
   - KG triples where the object or subject is `ticket_<id>` or similar.
3. Evaluate:
   - No matching drawer → `hygiene_status = 'missing_diary'`
   - Matching drawer with body < 100 chars → `hygiene_status = 'thin'`
   - No matching KG triple → `hygiene_status = 'missing_kg'`
   - Both present, substantive → `hygiene_status = 'clean'`
   - Evaluation failed (palace query error, timeout) → `hygiene_status = 'pending'`
4. `UPDATE ticket_transitions SET hygiene_status = $1 WHERE id = $2`.

The checker runs at-most-once per ticket_transition. Failures (palace unreachable, query timeout) produce `hygiene_status = 'pending'` rather than an infinite retry. A periodic sweep reprocesses `pending` rows older than 1 minute.

The 5-second wait is a soft-gate artifact: agents are supposed to write before transitioning, but the palace write might still be in flight when the transition notification fires. A short delay keeps the checker's window aligned with reality without becoming a bottleneck.

## Engineering workflow for M2.2

M2.2 ships engineering + one additional role: `qa-engineer`.

**Engineer role** (the existing M2.1 role, expanded):
- Listens on `work.ticket.created.engineering.todo`
- On spawn: reads ticket, searches `wing_frontend_engineer`, writes a minimal code change (M2.2's acceptance ticket is "create `changes/hello-$TICKET_ID.md` with one paragraph about what you did"), writes diary, writes KG triples, transitions to `qa_review`.
- agent.md path: `migrations/seed/engineer.md` (committed, seeded by migration).

**QA engineer role** (new for M2.2):
- Listens on `work.ticket.transitioned.engineering.in_dev.qa_review`
- On spawn: reads ticket, searches `wing_qa_engineer`, verifies the engineer's output exists and reads sensibly, writes a diary entry noting review outcome, writes KG triples, transitions to `done`.
- agent.md path: `migrations/seed/qa-engineer.md` (committed, seeded by migration).

Two roles (not one) because the full end-to-end hygiene check requires an actual second-agent handoff. With only the engineer, the hygiene checker has nothing cross-agent to verify — the whole point of the thesis (RATIONALE §3) is that Agent B reads Agent A's memory. Testing that in M2.2 means Agent B must exist.

## Acceptance criteria for M2.2

The milestone ships when all of these pass:

1. `mempalace init --yes` runs on supervisor startup against a clean `~/.garrison/palace/` (or the configured path), completes without error, and leaves a valid palace behind.
2. A ticket inserted into `tickets` with `column_slug = 'todo'` and `department_id = engineering` triggers an engineer spawn via the M2.1 event bus.
3. The engineer's Claude subprocess has both `postgres` and `mempalace` MCP servers listed in its `--mcp-config` and both show `status: "connected"` in the init event.
4. The engineer's system prompt includes the wake-up context from `mempalace wake-up --wing wing_frontend_engineer`.
5. The engineer writes exactly one diary entry to `wing_frontend_engineer` with the ticket ID in the body, writes at least two KG triples mentioning the ticket, and transitions the ticket to `qa_review`.
6. The hygiene checker, within 10 seconds of the transition, writes `hygiene_status = 'clean'` to the `ticket_transitions` row.
7. The QA engineer spawns on the `qa_review` transition, reads the engineer's diary entry from the palace (`mempalace_search` returns it), writes its own diary entry to `wing_qa_engineer`, transitions to `done`.
8. Both agent_instances complete with `total_cost_usd` populated. Combined cost for the two-agent run is under $0.20 ($0.10 per-invocation cap × 2).
9. Chaos test: MemPalace MCP server dies mid-run. Supervisor detects (stream-json `result` event with non-zero exit or init event with `status != "connected"` on a new spawn) and records `exit_reason = "mcp_mempalace_failed"` on the affected agent_instance. The supervisor does not crash.
10. Chaos test: broken MemPalace MCP config (invalid JSON in the mcpconfig file, unreadable palace path). Init event shows `mcp_servers[1].status = "failed"`. Supervisor bails within 2 seconds. Agent_instance row shows `exit_reason = "mcp_mempalace_failed"`.
11. Chaos test: hygiene checker's palace query times out. Status correctly set to `'pending'`. The next sweep cycle reprocesses and updates to `'clean'` or the appropriate failure.
12. Real-world cost cross-check: 5-10 M2.2 runs against Anthropic's billing dashboard, aggregate `total_cost_usd` matches within 5%. Discrepancy is a bug to report in the retro.
13. All M1 and M2.1 tests still pass.

## Binding questions for `/garrison-specify m2-2`

The spec must commit to each of these (with the listed defaults if no strong reason exists to choose otherwise). The spec is incomplete if any is deferred.

1. **Palace path default**: `~/.garrison/palace/` (expanded via `os.UserHomeDir()`). Override via `GARRISON_PALACE_PATH` env var. Justification required if another default is chosen.

2. **MemPalace MCP server invocation**: `mempalace serve` as a subprocess that Claude spawns via the mcp-config. Default assumes `mempalace` is on `$PATH` in the supervisor's container; Dockerfile must install it. Install method: binary release from MemPalace's repo, verified by SHA256, same discipline as M2.1's Claude Code install. Spec commits to a specific MemPalace version.

3. **Wake-up token budget**: `--max-tokens 200`. Empirical basis: spike observed ~170 tokens for typical wing content. 200 provides headroom.

4. **Wake-up failure policy**: non-blocking (spawn proceeds without wake-up). Alternative: bail the spawn with `exit_reason = "wake_up_failed"`. Default is non-blocking because wake-up is an optimization, not a correctness concern. Justification required if bail is chosen.

5. **Hygiene checker wait delay**: 5 seconds between transition notification and hygiene evaluation. Tunable via `GARRISON_HYGIENE_DELAY`. Justification required if a different default is chosen.

6. **Hygiene checker concurrency**: single goroutine, sequential processing of notifications. Justification required if concurrent evaluation is chosen (it complicates reasoning about at-most-once semantics).

7. **QA engineer listens_for pattern**: `work.ticket.transitioned.engineering.in_dev.qa_review`. Spec commits to the exact pattern (M2.1's trigger emits fully-qualified channels, see migration).

8. **Per-invocation budget cap**: $0.10 USD. Justification as described in "Real-world cost baseline" section above. Applied via Claude Code's `--max-budget-usd 0.10` flag.

9. **`garrison_agent_mempalace` role permissions**: SELECT on `ticket_transitions`, `agent_instances`, `tickets`; plus SELECT/INSERT/UPDATE on any MemPalace-metadata tables M2.2 introduces. Spec commits to the exact grants by migration.

10. **Seed agent.md content**: the spec commits to the exact content of `migrations/seed/engineer.md` and `migrations/seed/qa-engineer.md`. These files embody the completion protocol from `ARCHITECTURE.md`'s MemPalace write contract.

## Open questions the spec may defer to `/speckit.clarify`

These are genuinely unknown until investigated:

- **MemPalace version to pin**: latest stable as of M2.2 spec time. Check the MemPalace repo for current release and commit to the exact version. If the latest is pre-1.0, flag the stability concern.
- **MemPalace MCP server's authentication model**: does `mempalace serve` require any authentication (API key, token)? The spike covered basic usage but may have missed auth details. Clarify against the MemPalace docs.
- **Palace bootstrap on existing installations**: if `~/.garrison/palace/` already exists from a prior run (e.g. developer testing), does `mempalace init --yes` silently succeed, error out, or re-initialize? Clarify against MemPalace behavior.
- **KG triple shape for the "two-agent handoff" case**: the QA engineer needs to reference the engineer's triple. What's the idiomatic KG pattern? (`kg.query(subject='ticket_<id>', predicate='completed_by')`?) Clarify by reading MemPalace's KG query API.
- **Wing creation timing**: are wings created on first write (via `add_drawer`) or do they need to be explicitly created? Affects whether the spec needs a "create wings at bootstrap" step.

These notes inform the plan; the plan may diverge with justification.
