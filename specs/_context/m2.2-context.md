# M2.2 context — MemPalace MCP wiring

Binding constraints for the M2.2 milestone: wire MemPalace into every agent spawn, run the bootstrap in a controlled way, validate the memory thesis that the whole Garrison architecture depends on.

**Status**: planned milestone, follows M2.1. Spike complete at `docs/research/m2-spike.md`. Spec not yet drafted; this context file is binding input to `/speckit.specify` when M2.2 activates.

**Precedence**: same as M2.1. RATIONALE.md and ARCHITECTURE.md higher; the spike is ground truth for tool behavior.

---

## What M2.2 produces

After M2.2 ships, the system has:

1. A MemPalace installation at `~/.garrison/palace/` (operator-configurable), bootstrapped via `mempalace init --yes` against a dedicated empty directory at `~/.garrison/palace-init/`. Neither path is inside a git-tracked directory.
2. Every supervisor-spawned Claude Code subprocess has **two** MCP servers configured, both health-checked via the init event:
   - The Postgres MCP from M2.1 (read-only ticket access)
   - The MemPalace MCP (`python -m mempalace.mcp_server --palace ~/.garrison/palace`)
3. The engineering agent's `agent.md` carries the full completion protocol from `ARCHITECTURE.md`: diary entry to `wing_engineer`, KG triples for completion and artifacts, optional hall_discoveries write. Agents obey it.
4. A read-only hygiene query: `SELECT * FROM ticket_transitions WHERE hygiene_status != 'clean'` returns transitions where expected writes are missing. This query powers the M3 hygiene dashboard; M2.2 produces the data and the query, not a UI.
5. The memory thesis from RATIONALE §3 is validated or falsified. M2.2's retro records the verdict.

End-to-end acceptance expands on M2.1: one ticket → one Claude subprocess → one `hello.txt` file → **one diary entry in `wing_engineer`** → **one KG triple `(agent_instance_<id>, completed, ticket_<id>, valid_from=now)`** → supervisor writes the transition → hygiene check runs async → `hygiene_status = 'clean'` (or whatever the check records).

## What is explicitly out of scope

- **Full hygiene UI** — M3 builds the dashboard. M2.2 produces the query and the data. An operator can run `psql` to see hygiene issues.
- **Multiple wings beyond `wing_engineer` and `wing_company`** — M2.2 only exercises the engineering role. More roles arrive as M3+ adds departments.
- **`mempalace mine` bulk ingest** — the spike found `mine` is O(files) and unusable at scale. M2.2 does not use `mine` for any live path. Bulk ingesting ARCHITECTURE.md, RATIONALE.md, AGENTS.md into the palace is a separate one-off task the operator runs manually; it is not a milestone task.
- **Long-running MemPalace daemon** — M2.2 uses per-invocation stdio spawning (each Claude subprocess gets its own MemPalace server via `--mcp-config`). The ~500ms startup overhead is accepted per the spike.
- **Concurrent write stress testing** — the spike tested 2 concurrent writers successfully; M2.2 runs with the engineering cap of 1 (or 2-3), which is well below the tested load. Stress testing for 10+ concurrent writers is deferred until a milestone's workload demands it.
- **AAAK dialect compression** — the spike surfaced `mempalace_get_aaak_spec` and the `compress` subcommand but did not characterize when AAAK is applied or who writes in it. M2.2 agents write plain prose drawers. AAAK compression is a future concern.
- **Tunnel creation** — `mempalace_create_tunnel` and cross-wing tunneling are not part of M2.2's scope. Engineers write to `wing_engineer`; no cross-wing writes yet.
- **Dashboard surfacing of hygiene** — M3.
- **Per-agent diary via `mempalace_diary_write`** — the spike surfaced this tool but its relationship to the drawer-based diary pattern from ARCHITECTURE.md's "MemPalace write contract" section is unclear. M2.2 uses **drawers** for diary entries (per the architecture doc's written contract). The `mempalace_diary_write` tool is a future consideration; the spec may explore whether it offers anything drawers don't, but M2.2 does not adopt it.

## The stack (locked)

Additions to the M2.1 stack:

- `mempalace` Python package (currently 3.3.2 per the spike). Installed in a venv at a supervisor-known path (e.g. `~/.garrison/mempalace-venv/` or inside the Dockerfile for the release image).
- A small MemPalace bootstrap script or `garrison bootstrap-palace` subcommand that idempotently:
  1. Creates `~/.garrison/palace-init/` if missing (empty or with a minimal seed file)
  2. Runs `mempalace init --yes --palace ~/.garrison/palace ~/.garrison/palace-init/` once
  3. Verifies the palace at `~/.garrison/palace/` is reachable via `mempalace status --palace ~/.garrison/palace`
  4. Is safe to re-run (no-op on second call)

No new Go dependencies expected. JSON parsing for MemPalace responses is the same `encoding/json` M2.1 already uses.

## Spawn contract (binding additions to M2.1)

Every Claude Code subprocess in M2.2 adds one MCP server entry to the per-invocation MCP config:

```json
{
  "mcpServers": {
    "postgres": { "command": "<postgres mcp cmd>", "args": [...] },
    "mempalace": {
      "command": "<path to venv>/bin/python",
      "args": ["-m", "mempalace.mcp_server", "--palace", "<palace path>"]
    }
  }
}
```

Both entries appear in the init event's `mcp_servers[]`. Both must have `status == "connected"` before the supervisor lets the turn proceed. Either failure is a spawn abort with `exit_reason = "mcp_<name>_<status>"`.

The palace path is resolved at supervisor startup from the config (env var `GARRISON_PALACE_PATH`, default `~/.garrison/palace/`). The path is the same across all spawns within a single supervisor run — MemPalace is a shared org-level resource, not per-agent or per-department.

## Bootstrap contract (binding)

The first time Garrison runs on a new host, the palace must be created before any agent spawn. This is a required step, not an optional one.

**Binding rules** from the spike:

1. **Never run `mempalace init` against a git-tracked directory.** The palace init target (`~/.garrison/palace-init/`) must be outside any git repository. The supervisor verifies this on startup — if `~/.garrison/palace-init/` is inside a git repo (detected via walking up looking for `.git`), startup fails with a clear error before any init attempt. This rule is non-negotiable; see AGENTS.md and spike §3.2.

2. **Always pass `--yes`.** `mempalace init` is interactive by default and crashes in non-interactive contexts with `EOFError`. The bootstrap script passes `--yes`. Non-interactive is the only supported mode.

3. **Bootstrap is idempotent.** Running the bootstrap twice must succeed. The script detects an existing palace at `~/.garrison/palace/` and skips the init step.

4. **Bootstrap is a deliberate operator action**, not something that happens automatically on `supervisor start`. This is because the palace is expensive (~tens of MB with embedding models loaded) and creating one silently on process start violates the principle that the supervisor manages processes, not installs. The operator runs `garrison bootstrap-palace` once per host; the supervisor refuses to start until that has succeeded.

## Wing and room conventions (binding for M2.2)

Per spike §3.3 and §3.7, wings and rooms are created on write via `mempalace_add_drawer`. M2.2 establishes conventions that downstream milestones inherit.

**Wings used in M2.2**:
- `wing_engineer` — all drawers written by any engineer agent instance (shared across instances, per ARCHITECTURE.md)
- `wing_company` — CEO-level decisions. M2.2 does not write here (no CEO yet); the wing exists for future use and the M1 retro (from M2.2 onwards) will write retros here.
- `wing_engineering` — the engineering department manager's wing. M2.2 does not have a CTO/manager role, so this wing remains unpopulated in M2.2.

**Rooms within `wing_engineer`** — the agent writes to these:
- `hall_events` — one drawer per completed ticket, body is the diary entry
- `hall_discoveries` — optional drawer if the agent surfaces a pattern worth remembering (the spike suggests most M2.2 tickets will not produce a discovery; that's fine)
- `hall_facts`, `hall_preferences`, `hall_advice` — not used by M2.2's minimal engineer agent; reserved for future agent roles that need them

**Agent.md completion protocol** (the engineer's `agent.md` changes materially from M2.1):

```markdown
## Completion protocol (MANDATORY)
Before your turn ends, you MUST:

1. Write a diary entry to wing_engineer via mempalace_add_drawer:
   wing: "wing_engineer"
   room: "hall_events"
   content: |
     ticket_id: <the ticket id from your prompt>
     outcome: <one line — what you produced>
     artifacts: [<paths>]
     rationale: |
       <one paragraph — why you did what you did>
     blockers: [<list of short strings or empty>]
     discoveries: [<list of short strings or empty>]
     completed_at: <iso timestamp>

2. Write one KG triple via mempalace_kg_add:
   subject: "agent_instance_<your instance id>"
   predicate: "completed"
   object: "ticket_<ticket id>"
   valid_from: now

3. If you discovered a pattern worth remembering, write to hall_discoveries:
   wing: "wing_engineer"
   room: "hall_discoveries"
   content: <short paragraph>

Only after these writes should you exit.
```

The agent's instance ID is injected into the `-p` prompt alongside the ticket ID (as a separate field, not as the ticket ID).

## Hygiene check (binding shape)

After every `ticket_transitions` insert triggered by an agent success, the supervisor runs an async hygiene check. This is new in M2.2 (M2.1 had no hygiene contract).

**The check** (runs against Postgres + MemPalace, roughly 200ms post-transition):

```sql
-- Was there a drawer write in wing_engineer/hall_events mentioning this ticket
-- during the agent_instance's run window?
SELECT EXISTS (
  SELECT 1 FROM mempalace_drawers d
  WHERE d.wing = 'wing_engineer'
    AND d.room = 'hall_events'
    AND d.content LIKE '%' || $ticket_id || '%'
    AND d.created_at BETWEEN $agent_instance.started_at AND $agent_instance.finished_at
);
```

(Depending on what MemPalace exposes queryably — the spike did not characterize direct SQL access to the palace. If MemPalace's storage is opaque, the hygiene check uses `mempalace_search` via MCP from the supervisor's own MemPalace client, not direct SQL.)

**Status values**:
- `clean` — both a drawer and a KG triple exist, drawer body passes a thin-write length check (>100 chars)
- `missing_diary` — no drawer found
- `missing_kg` — drawer exists but no KG triple for this ticket
- `thin` — drawer exists but body is shorter than 100 characters

**What the check does NOT do in M2.2**: block transitions, retry, notify, or surface to a UI. It writes to `ticket_transitions.hygiene_status` and stops. M3 displays the results; M6 adds operator action on hygiene issues.

## Acceptance criteria for M2.2

All must pass against a freshly-migrated Postgres, a freshly-bootstrapped palace, and the release Docker image before M2.2 ships.

1. `garrison bootstrap-palace` against a clean host creates `~/.garrison/palace/` with Chroma index files and SQLite DB, non-empty.
2. Re-running `garrison bootstrap-palace` on the same host succeeds without re-initializing (idempotent check).
3. Attempt to bootstrap with `~/.garrison/palace-init/` inside a fake git repo fails with a clear error before any MemPalace call (the "don't init git-tracked dirs" rule is enforced at supervisor start).
4. Supervisor starts; `/health` returns 200; the init event of a test spawn shows both `postgres` and `mempalace` in `mcp_servers[]`, both with `status == "connected"`.
5. Insert one engineering ticket. Within subprocess timeout, the Claude agent:
   - Reads the ticket via Postgres MCP
   - Writes `hello.txt` (M2.1 acceptance still passes)
   - Writes one drawer to `wing_engineer/hall_events` matching the ticket ID (verified via `mempalace_search --wing wing_engineer` post-run)
   - Writes one KG triple for completion (verified via `mempalace_kg_query` post-run)
   - Exits with `result.terminal_reason == "completed"`
6. Supervisor writes the ticket transition and `agent_instances` row as in M2.1. Async hygiene check runs within 1 second and writes `hygiene_status = 'clean'` to `ticket_transitions`.
7. Inject a test where the agent's `agent.md` is modified to SKIP the diary write (simulating a misbehaving agent). The hygiene check writes `hygiene_status = 'missing_diary'`. The transition itself still succeeds (soft gate, per RATIONALE §5).
8. Broken MemPalace MCP (missing venv, corrupted palace) triggers the init-event bail: `exit_reason = 'mcp_mempalace_failed'` (or similar; exact string depends on what `status` MemPalace reports under failure — the spike observed `failed` for a bogus command, so that is expected).
9. Two concurrent engineering spawns (cap temporarily raised to 2 for this test) both write their drawers successfully; the palace has two entries tagged to each respective ticket.
10. M2.1 acceptance criteria 1-11 all still pass. No regression.

One new chaos test:
11. **MemPalace MCP dies mid-run**: external `kill` on the MemPalace subprocess while the agent is mid-turn. Observe Claude's behavior from stream-json events. Expected: tool calls return errors, agent either adapts or exits with `is_error == true`. Record the result in the retro — this is a behavior the spike did not characterize.

## Binding questions for `/speckit.specify`

1. **Where does the MemPalace venv live?** Options: a fixed path (`~/.garrison/mempalace-venv/`); bundled in the Docker image; operator-configurable. **Default**: fixed path for dev, bundled for Docker. The spec may choose to make it configurable with the fixed default as the fallback.

2. **How does the supervisor know the agent_instance ID to inject into the agent's prompt?** The ID is generated by the supervisor before spawn (to write the `running` row). It's passed alongside the ticket ID in the `-p` argument. **Default**: the supervisor constructs a prompt like `"Ticket: <ticket_id>\nInstance: <instance_id>\nFollow your agent.md."` where the agent.md's completion protocol references `<your instance id>` as a placeholder the agent substitutes at runtime.

3. **Where does the hygiene check run?** Options: a separate goroutine in the supervisor; a separate subsystem; a post-transition callback. **Default**: a goroutine spawned from the event dispatcher for each `ticket.transitioned.*` event, with its own timeout context and errgroup parent so it cannot block the main dispatch loop.

4. **What's the thin-write threshold?** The architecture doc mentions "<100 characters" as a hint. **Default**: 100 characters on the drawer body (excluding YAML frontmatter). Operator-configurable via env var for future tuning.

5. **How are MemPalace failures classified in `exit_reason`?** The spike saw `failed` for a bogus command. Other states: `needs-auth` (which MemPalace itself won't hit), and potentially `disconnected` mid-run. **Default**: `mcp_<server>_<status>` for init-time failures; `mcp_<server>_disconnected_midrun` for mid-run failures (if detectable). The specific mid-run detection mechanism is an M2.2 spec concern.

6. **Does the bootstrap command live in the supervisor binary (as `supervisor bootstrap-palace`) or a separate binary/script?** **Default**: a subcommand of the supervisor binary. One binary to install, consistent with the single-binary deployment ethos.

7. **What's in `~/.garrison/palace-init/`?** It's the target `mempalace init` scans to auto-detect wings and rooms. **Default**: empty directory. Empty means MemPalace creates only its default `general` room and no auto-detected wings. This is fine because M2.2 creates wings on write via `add_drawer`.

## Spike findings that bind this milestone

From `docs/research/m2-spike.md`:

- §3.1–§3.2 — mempalace 3.3.2 install, CLI surface, `init --yes` requirement (binding for bootstrap)
- §3.2 — init mutates `.gitignore` (binding: dedicated non-git-tracked palace path)
- §3.4 — `mine` is unsuitable for live writes (binding: agents use MCP `add_drawer`, not `mine`)
- §3.6 — 29 MCP tools available; the important ones for M2.2 are `add_drawer`, `kg_add`, `kg_query`, `search`, `check_duplicate`, `list_wings`
- §3.7 — wings and rooms created on write (binding: no pre-creation step needed)
- §3.8 — 2-writer concurrency clean (sufficient for M2.2's engineering cap)
- §3.9 — per-invocation ~500ms startup overhead (accepted; daemon deferred)

All open questions from §5 of the spike remain open after M2.2 ships and should be revisited when they become blocking.

## What we're watching for at ship

The M2.2 retro must answer:

1. **The memory thesis**: did agent writes produce useful memory? Specifically, if you read the 5-10 drawers written during M2.2 testing, do they read as "an engineer's working notebook" or as "agents dutifully filling required fields"? The difference matters enormously — the first validates the thesis; the second suggests the completion protocol is being performed rather than inhabited.
2. **Hygiene signal quality**: how many transitions ended up `clean` vs `missing_*` vs `thin`? If the hygiene check fires false positives (flagging thin writes that are actually fine, or missing writes that exist but couldn't be matched), M3's dashboard will be noisy.
3. **Bootstrap pain**: how many tries did it take a fresh install to produce a working palace? Any operator-facing surprises?
4. **Mid-run MCP failure behavior**: the chaos test discovers what actually happens. Record it for future reference.
5. **Concurrency under realistic load**: were 2-3 concurrent spawns actually clean in practice, or did the spike's "2 writers worked cleanly" finding mask a real-world failure mode?
6. **AAAK and diary tools revisited**: now that we've been writing drawers, was the spike's decision to ignore `mempalace_diary_write` and AAAK compression correct, or did we find a reason to revisit?

If answer to #1 is "agents dutifully filling required fields," M3 and beyond slow down. The completion protocol needs redesign before scaling to more roles. This is the thesis-validation moment RATIONALE §3 explicitly flags.
