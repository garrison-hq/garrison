# M2.2.2 — Compliance calibration

**Binding context for `/garrison-specify m2-2-2`, `/garrison-plan m2-2-2`, `/garrison-tasks m2-2-2`, `/garrison-implement m2-2-2`.**

This is a **focused patch milestone** addressing the compliance gaps M2.2.1's live-run append surfaced: haiku reaching for `finalize_ticket` but unable to recover from schema errors, and opus burning budget on exploration before reaching the completion tool. M2.2.2 does not change architecture; it tunes the mechanism M2.2.1 built so real models can actually use it.

**Last updated**: 2026-04-23.

---

## Why this milestone exists

M2.2.1 validated that the tool-as-attractor pattern works (haiku reached for `finalize_ticket` unprompted), but revealed two gaps:

1. **Schema errors are not actionable on weaker models.** Haiku emitted a malformed payload, received `error_type="schema", field=""` (decode-level rejection), and gave up on the tool — treating schema error as "try something else" rather than "fix and retry."
2. **Exploration before completion burns budget on stronger models.** Opus spent 25+ seconds on palace searches before reaching any completion tool, hitting its budget cap at $0.2564 before `finalize_ticket` was ever called.

Neither is an architectural failure. Both are operationalization problems — how the existing mechanism communicates with real models under real constraints. M2.2.2 fixes the communication.

The non-goal: do not revise RATIONALE §3 further. M2.2.1's thesis (supervisor writes from structured agent output) remains correct. M2.2.2 is about making the mechanism effective, not about replacing it.

## What M2.2.2 produces

### 1. Richer schema error responses (retro item 2)

The `finalize_ticket` tool's error response is extended to pinpoint exact failures:

**For JSON decode failures** (current `error_type="schema", field=""`):
- Parse position: line and column of the failing token
- Excerpt: the ~40 character window around the failure
- Hint: human-readable description of what went wrong ("expected `}` before `]`", "invalid escape sequence")

**For schema validation failures** (current `error_type="schema", field="<json-pointer>"`):
- Precise field path (already present)
- Constraint that failed: `"min_length"`, `"required"`, `"type_mismatch"`, etc.
- Expected value: `"string with min length 50"`, `"array with min length 1"`, etc.
- Actual value received (truncated to 100 chars if long)

The error response shape becomes:

```json
{
  "error_type": "schema",
  "failure": "decode" | "validation",
  "field": "<json-pointer or empty>",
  "line": 12,
  "column": 45,
  "excerpt": "...context...",
  "constraint": "min_length",
  "expected": "string with min length 50",
  "actual": "ok",
  "hint": "the rationale field must be at least 50 characters; you sent 2"
}
```

Purpose: give the agent enough specific information to fix and retry, not just "something was wrong."

### 2. Fix Adjudicate precedence (retro item 6)

In `internal/spawn/spawn.go`'s `Adjudicate` function, the precedence order is revised so that when `ResultSeen=true`:

- **Before**: `IsError` checked first, then `isBudgetTerminalReason`. Result: Claude's own budget-cap signal (`is_error=true + terminal_reason="error_max_budget_usd"`) classifies as `claude_error`, hiding the cost root cause.
- **After**: `isBudgetTerminalReason` checked first when ResultSeen=true, then `IsError`. Budget-cap events classify as `budget_exceeded`.

Single-line swap plus regression tests. The M2.2/M2.2.1 test suite covers the existing paths; M2.2.2 adds one new test (`TestAdjudicateBudgetExceededTakesPrecedenceOverIsError`) pinning the new behavior.

### 3. Front-load the finalize directive in agent.md (retro item 5)

The engineer and qa-engineer agent.md's work loop is restructured so `finalize_ticket` is named at the top of the work description, not at the end. Current M2.2.1 shape:

> 1. Read the ticket. 2. Search palace. 3. Implement. 4. Call finalize_ticket.

New M2.2.2 shape:

> Your goal is to call `finalize_ticket` with a valid payload describing the work you did on ticket `$TICKET_ID`. Everything below is in service of producing that payload.
>
> 1. [optional] Read relevant palace context ONLY if ticket complexity warrants it (see palace-search calibration below).
> 2. Do the work.
> 3. Call `finalize_ticket`.

This makes the completion mechanism the first thing the agent is primed on, not the final consideration after a long work loop.

### 4. Example-based agent.md (retro item 1)

Both seed agent.md files gain a concrete, realistic `finalize_ticket` payload example in a fenced block. The example uses a plausible ticket shape (not the literal current ticket_id — a placeholder the agent should not copy verbatim), demonstrates the full required schema, and is annotated to explain the why of each field.

Target: haiku's first attempt produces a well-formed payload because the template is visible in the prompt. The example is ~400-600 chars, fitting within the 3000-4000 byte agent.md target.

Critical property: the example must NOT be the literal payload the agent should emit. It's a template with placeholder values the agent adapts. Confusion between "follow the example" and "copy the example" is a real risk; the annotation addresses this explicitly.

### 5. Prompt-level retry framing (retro item 3)

The completion section's language is revised:

- **Before** (M2.2.1): "You have up to 3 attempts."
- **After** (M2.2.2): "If `finalize_ticket` returns a schema error, read the `hint` field carefully, fix the specific issue named, and call `finalize_ticket` again. You have up to 3 attempts; retrying with corrections is expected."

This reframes retry from "you might fail twice, which is bad" to "retry with corrections is part of the normal flow." Combined with the richer error responses from item 1, this should produce actionable retry behavior rather than haiku's observed "try a different tool" pattern.

### 6. Palace-search calibration (retro expansion, adjacent to item 5)

The engineer and qa-engineer agent.md's palace-search guidance is calibrated with ticket-complexity heuristics. The M2.2.1 language ("Check `hall_discoveries` in your wing for patterns worth knowing before you start") is too unconditional — it prompted opus to spend 25+ seconds on exploration for a trivial ticket.

New guidance (template):

> Before starting work, decide whether palace context is needed:
>
> - **Skip palace search if**: the ticket's objective is straightforward, well-defined, and similar to routine work you've done before. Small diary entries, hello.txt-style tasks, one-line changes. These consume <5% of the budget cap; palace search is not cost-effective.
> - **Search palace if**: the ticket mentions cross-cutting concerns (multiple components, integration work), references prior tickets or ongoing threads, or the objective is ambiguous enough that prior context would help. Budget up to 3 tool calls (one `mempalace_search` + one `mempalace_list_drawers` + one targeted read).
> - **In doubt, skip.** Finalize without context is recoverable (operator can enrich the diary after). Hitting the budget cap mid-exploration is not.

This is the most prompt-engineering-heavy item in M2.2.2 and the one most likely to need iteration. Its success metric: opus reaches `finalize_ticket` within 60 seconds on a trivial ticket (down from never-reached in M2.2.1's 2.8x budget-burning observation).

### 7. Regression tests and compliance re-run

New integration test: `TestM222ComplianceMatrix` runs the same happy-path ticket against both haiku-4-5-20251001 and opus-4-7, asserts:

- Both models call `finalize_ticket` at least once
- Both models produce at least one well-formed payload (via the new richer errors + example)
- Both models complete with `hygiene_status='clean'` on at least one run out of 3

The test is gated by a `live_acceptance` build tag (carry-over from M2.2.1) and requires operator-supplied API credentials. Run manually at acceptance time; cost ~$1-2 for the full matrix.

Criterion: both models reach clean completion. Partial passes (e.g. haiku clean, opus still fails) are documented in the retro as follow-up, not ship blockers.

## What M2.2.2 does not produce

Explicit non-scope:

- **Flexible-first-attempt schema mode** (retro item 4). Dropped deliberately. If M2.2.2's richer errors (item 2) don't produce reliable retry behavior, item 4 becomes a future milestone. Don't build both; observe whether item 2 suffices first.
- **Changes to the finalize_ticket schema itself.** Strict shape from M2.2.1 is preserved. Richer errors tell the agent what's wrong; the schema doesn't weaken.
- **Changes to the retry cap.** Still 3 attempts, enforced supervisor-side.
- **Changes to the atomic write transaction.** M2.2.1's semantics remain.
- **Dashboard work.** M3 concern.
- **Secret vault.** M2.3 concern.
- **MemPalace version bumps, container changes, socket proxy changes.** Everything stays on M2.2's deployment topology.
- **CI automation of the compliance matrix.** M2.2.1's retro deferred this; M2.2.2 does not take it on. The live_acceptance tag pattern remains operator-run.
- **Persistent retry counter.** The in-memory supervisor counter from M2.2.1 remains. Persistence is a follow-up concern if crashes mid-retry become observable.
- **`hall_discoveries` searchability fix.** M2.2.1's retro noted this as deferred; remains so.

If the spec phase tries to include any of these, stop and surface the scope violation.

## Architectural decisions (binding)

### Decision 1: Retain strict schema, invest in error feedback

M2.2.2 does not weaken the `finalize_ticket` input schema. The fix for compliance is better error communication, not a looser contract. Rationale: a strict schema forces reflection; weakening it defeats the M2.2.1 mechanism. Rich errors give the agent enough information to comply with the strict schema.

If after M2.2.2 ships haiku still cannot recover from schema errors even with detailed feedback, the follow-up question (M2.2.3 or rolled into M3) becomes whether strict schemas are fundamentally incompatible with weaker models. That evidence doesn't exist yet; don't pre-commit to its conclusion.

### Decision 2: Prompt-level changes do not modify the tool contract

All agent.md rewrites in M2.2.2 are prompt changes only. The `finalize_ticket` tool's schema, response shape (aside from richer errors), and supervisor-side handling remain identical to M2.2.1. An agent compiled against M2.2.1 behaves identically to one compiled against M2.2.2 — only the seed prompts differ.

Consequence: rolling back M2.2.2's prompt changes is a seed-data revert, not a code revert. If an M2.2.2 prompt change degrades behavior, revert is cheap.

### Decision 3: Palace-search calibration via prompt, not code

The palace-search heuristics (when to search, when to skip) are expressed as prompt guidance, not enforced by the supervisor. Rationale: the supervisor has no way to know a ticket's complexity heuristically without running an LLM pass of its own, which adds cost. The agent itself is the right place to make this judgment given the ticket text and its own role context.

Trade-off: the agent may still over-explore on specific models. That's what the compliance matrix measures. If prompt-only calibration fails for opus specifically, M3 may need a supervisor-enforced "skip palace on first 3 turns" mechanism. Not now.

### Decision 4: Precedence fix is surgical

The Adjudicate precedence swap is one line. Not a broader refactor of the precedence table. Rationale: Adjudicate is load-bearing for every terminal classification; broader changes risk regressions on M1/M2.1/M2.2 paths the tests cover. Surgical fix with targeted regression test is the right scope.

## Acceptance criteria for M2.2.2

The milestone ships when:

1. `finalize_ticket` tool returns schema errors with all extended fields present (line, column, excerpt, constraint, expected, actual, hint for validation errors; appropriate subset for decode errors).
2. `internal/finalize/tool_test.go` includes at least 6 new test cases covering each new error scenario (decode at various positions, validation for each required field, validation for each constraint type).
3. `Adjudicate` correctly returns `(status=failed, exitReason=budget_exceeded)` when `IsError=true + terminal_reason="error_max_budget_usd" + ResultSeen=true`. Regression test pins this.
4. `engineer.md` opens with "Your goal is to call `finalize_ticket`..." language; the example payload is present as a fenced block; palace-search calibration bullets are present; retry framing is present.
5. `qa-engineer.md` has the parallel structure appropriate for the QA role.
6. Both agent.md files are within 3500-4500 bytes.
7. Mockclaude fixture test: an agent that emits a malformed JSON payload on attempt 1 and a schema-valid payload on attempt 2 produces `hygiene_status='clean'` and completes successfully. This verifies the retry path works end-to-end with the new error shape.
8. Mockclaude fixture test: an agent that emits 3 consecutive malformed payloads produces `hygiene_status='finalize_failed'` as in M2.2.1. No regression.
9. **Live-acceptance test** (`TestM222ComplianceMatrix`, gated by `live_acceptance` build tag): the happy-path ticket runs 3 times against haiku-4-5-20251001 and 3 times against opus-4-7. At least 2/3 runs per model produce `hygiene_status='clean'`. Aggregate cost across all 6 runs: under $3 USD. Headline acceptance — this is where M2.2.2 either validates or fails to.
10. All M1, M2.1, M2.2, M2.2.1 tests pass unchanged (aside from the M2.2.1 schema-error fixture updates enumerated in binding question 7).
11. Zero new Go dependencies. Fifth consecutive milestone.
12. Agent.md rollback verified: reverting `migrations/seed/*.md` to M2.2.1 versions produces M2.2.1 behavior exactly (compliance gaps return). This confirms decision 2 — prompt changes are isolated from code.

Criterion 9 is the headline. The others are hygiene. If criterion 9 doesn't pass for both models, the retro documents this and M2.2.2 ships as "partial compliance improvement" rather than "compliance validated" — the same honest-ship pattern M2.2.1 used.

## Implementation notes (non-binding, save plan time)

- **Schema error extension is mechanical.** Go's `json.Unmarshal` returns `*json.SyntaxError` with `Offset` field; converting offset to line/column requires walking the payload once. 30-40 lines of helper code. Extraction of field context from offset is straightforward.

- **The example payload placeholder format matters.** If the example uses realistic-looking UUIDs or realistic string content, haiku may copy them verbatim (observed failure mode in other LLM-coding contexts). Use obviously-placeholder syntax: `<ticket_id>`, `<your one-line outcome>`, etc. The curly-brace mustache-style is also fine: `{{ticket_id}}`. Avoid JSON-valid but realistic-looking strings.

- **Palace-search calibration is the most iteration-prone item.** The first draft will likely be wrong. Budget explicit rework time; don't expect the spec to land it perfectly on pass 1. The compliance matrix is how we know.

- **Front-loading changes agent.md structure materially.** The M2.2.1 work loop has "Read ticket → Search palace → Implement → Finalize" as roughly equal-weight bullets. M2.2.2 demotes the first three to "in service of producing the finalize payload." This is a semantic shift, not just reordering. Reviewers should read for whether the new shape coherently explains what the agent is doing and why.

- **The Adjudicate fix is the smallest change in the milestone.** One line swap. Don't overthink it. The regression test is the thing that matters.

- **Consider batching the prompt changes in one commit.** Items 3, 5, 6 (front-loading, example, retry framing, palace calibration) all touch the same two files. Splitting them into separate commits creates artificial churn. Single commit per agent.md file, ordered engineer.md then qa-engineer.md, with a clear message explaining the four concerns addressed.

- **The compliance matrix test is operator-owned.** Don't write it as a normal test that runs in CI. Same pattern as M2.2.1's `TestM221ComplianceModelIndependent` — build-tag gated, documented invocation, cost budget named.
