## Scope deviation from threat model — READ FIRST

The threat model at `docs/security/vault-threat-model.md` (dated 2026-04-22) scopes M2.3 to include both the vault plumbing AND the full operator-facing UI (CRUD screens, audit log viewer, rotation UI, access control UI, secret usage view). The threat model explicitly states "M2.3's UI work includes the rotation view, not just CRUD" (Rule 7) and quotes "two to four weeks of solo frontend work" as part of the M2.3 budget (§5).

**This M2.3 context deviates from that scope.** The deviation is:

- **M2.3**: supervisor-side vault integration + data model (this milestone)
- **M3**: dashboard read-only surfaces for secrets (browsing, audit log viewer, stale-secret surfacing, role-to-secret mapping viewer)
- **M4**: dashboard write surfaces for secrets (CRUD, grant editing, rotation initiation, environment/path management)

Rule 7's "rotation is a first-class concern" intent is preserved: rotation metadata (cadence, last_rotated_at) is first-class in M2.3's data model, so M3's read surface and M4's write surface can both build on it without retrofitting. What changes is that the rotation UI ships in M4, not M2.3.

## Why M2.3 now

The M2.2.x compliance arc validated the mechanism Garrison uses to complete tickets: finalize_ticket, atomic transitions, hygiene checks, per-role agent spawning. Agents can now do work reliably on the trivial ticket level.

But agents can't yet do *any* work that requires credentials. A supervisor that spawns agents without secrets can never exercise the Peppol integration, never call Stripe, never authenticate to a customer's Google Workspace. The compliance mechanism is necessary but insufficient for real Hey Anton work.

M2.3 addresses this: the supervisor reads secrets from Infisical at spawn time and injects them as environment variables into the agent subprocess. Agents get credentials without the credentials ever entering the LLM's context window.

This is not a product surface. This is plumbing that makes product surfaces possible.

## In scope for M2.3

All scope below is supervisor-side Go work, data model work (migrations), or Infisical deployment. No operator-facing UI in this milestone.

### Supervisor-side vault client (Go package)

A new `internal/vault` package that:

1. Authenticates to Infisical as a Machine Identity using Universal Auth. Credentials (client ID + client secret for Infisical's ML auth) come from environment variables set by the orchestration layer — this is the bootstrap-secret chicken-and-egg problem (§6 open question 7 in the threat model).

2. Fetches secrets by path at spawn time. The supervisor determines which secrets to fetch based on the spawning agent's role and the `agent_role_secrets` grants.

3. Injects secret values as environment variables into the Claude Code subprocess. The agent's `process.env.STRIPE_KEY` or equivalent returns the raw value; the value never enters the LLM context window, the agent.md, the system prompt, or any arg to `claude`.

4. Writes dual audit records: one to Infisical (via the Infisical API — automatic) and one to Garrison's own `vault_access_log` Postgres table with ticket/instance context that Infisical's log doesn't have.

5. Never logs secret values. Enforced at the type level in Go: the logger type for the vault package has no method that accepts a `SecretValue`. Compiler-enforced, not convention.

6. Handles every Infisical API failure mode with supervisor-facing errors:
   - `vault_unavailable` — Infisical is down, network error, TLS failure
   - `vault_auth_expired` — supervisor's machine identity token expired mid-run
   - `vault_permission_denied` — supervisor's Infisical-side role doesn't have access to the requested secret (Infisical-level denial, distinct from Garrison's `agent_role_secrets` denial)
   - `vault_rate_limited` — Infisical API rate limit hit
   - `vault_secret_not_found` — secret path doesn't exist in Infisical

   Each failure mode maps to a supervisor adjudicator outcome. Agent spawns that require secrets and cannot get them fail with `exit_reason = "vault_unavailable"` (or similar) per Rule 5. Hygiene sees these as new failure categories.

### Data model (migrations)

Three new tables. All tables carry `customer_id` from M2.3 even though M2.3 is single-tenant; the column defaults to the operating entity's identifier. This matches the threat model's multi-tenancy posture.

**`agent_role_secrets`** — the per-role grant table (Rule 2 enforcement).
- Columns: `role_slug` (FK to `agents.role_slug`), `secret_path` (the Infisical path like `/production/stripe/api_key`), `env_var_name` (the env var the supervisor injects, like `STRIPE_KEY`), `customer_id`, `granted_at`, `granted_by` (operator user ID, or `system` for operator-entered grants), plus standard audit columns.
- Constraint: unique on `(role_slug, env_var_name)` — you can't grant the same env var twice to the same role (ambiguity on which secret value wins).
- No default grants. A new agent role with no rows in this table receives zero secrets at spawn time.

**`vault_access_log`** — the Garrison-side audit table (Rule 6).
- Columns: `agent_instance_id` (FK to `agent_instances`), `ticket_id` (FK to `tickets`, nullable for non-ticket-bound operations), `secret_path`, `outcome` (`granted` / `denied_no_grant` / `denied_infisical` / `error_fetching` / `error_injecting`), `timestamp`, plus standard audit columns.
- **Does NOT contain secret values.** Enforced at the vault client's type level (Rule 6).
- Indexed on `agent_instance_id` and `ticket_id` for context-joined queries M3's dashboard will issue.

**`secret_metadata`** — Garrison-side metadata mirror of what's in Infisical, for query performance (Rule 4).
- Columns: `secret_path`, `customer_id`, `provenance` (enum: `operator_entered`, `oauth_flow`, `environment_bootstrap`, `customer_delegated`), `rotation_cadence` (interval: 30d / 90d / never), `last_rotated_at`, `last_accessed_at`, `allowed_role_slugs` (array, denormalized for fast lookups), plus audit columns.
- Kept in sync with Infisical via hooks on secret create/update/delete (supervisor-side, triggered by API actions from M4's write dashboard later; in M2.3 this table is populated by the initial secret seeding migration and by the OAuth flow handler below).
- Exists specifically so M3 can answer "which secrets are stale?" and "which roles can access this secret?" without hitting Infisical's API for every dashboard page load.

### Spawn-time enforcement

Three new spawn-time checks in `internal/spawn`:

1. **Rule 1 enforcement (secret-value leakage)**: before spawning, the supervisor scans the agent's resolved `agent.md` content against the set of secret *values* it is about to inject. If any secret value appears verbatim in agent.md, the spawn blocks with `exit_reason = "secret_leaked_in_agent_md"`. This is a bug in the agent.md, not in the agent; the operator needs to fix the agent.md before the agent can run.

2. **Rule 2 enforcement (per-role scoping)**: the supervisor issues a single SQL query (`SELECT env_var_name, secret_path FROM agent_role_secrets WHERE role_slug = ? AND customer_id = ?`) to determine what secrets to fetch. Agents with no grants get no secrets. The query returns zero rows → the supervisor skips the Infisical call entirely.

3. **Rule 3 enforcement (vault opacity)**: the supervisor's MCP config generation does NOT expose any vault-related MCP tool to the agent. No `vault_get`, no `secrets_list`, nothing. Spawn-time check: if the `agents.mcp_config` JSONB ever contains a vault MCP server registration, the spawn blocks. (This prevents a future bug where someone adds vault-as-MCP and breaks the threat model accidentally.)

### Infisical deployment

Infisical runs as a service in the same Coolify project as Garrison. Specifics:

- **Container**: `infisical/infisical:latest`, pinned to a specific digest at production time.
- **Backing services**: Infisical's own PostgreSQL and Redis, deployed alongside Infisical — NOT sharing Garrison's Postgres (per threat model §5 deployment shape).
- **Network**: bound to the internal Coolify network, NOT exposed publicly. Garrison's supervisor and dashboard connect via internal hostname.
- **Machine identity**: two separate identities — `garrison-supervisor` (read-only on secrets needed at spawn time) and `garrison-dashboard` (read+write on secrets, needed by M4). M2.3 creates both identities during Infisical bootstrap; M2.3 only uses the supervisor identity.
- **Bootstrap secret handling**: Infisical's own `ENCRYPTION_KEY` and `AUTH_SECRET` are set via Coolify environment variables at deploy time. M2.3 documents the generation procedure (use `openssl rand -base64 32` per Infisical docs) and the operator-side discipline (store in a password manager, never in a `.env` file committed to git).

### Secret-pattern scanning of agent outputs

Before any agent output is written to MemPalace, the supervisor scans it for suspected-secret patterns:

- `sk-` prefix (OpenAI, Stripe)
- `xoxb-` prefix (Slack bot tokens)
- `AKIA` prefix (AWS access keys)
- `-----BEGIN ` (PEM headers, private keys)
- `ghp_` prefix (GitHub personal access tokens)
- `gho_`, `ghu_`, `ghs_`, `ghr_` (other GitHub token types)
- Bearer-token-shaped strings in obvious contexts

On detection: the value is redacted to `[REDACTED:<pattern>]` in the MemPalace write, AND a hygiene warning is emitted on the ticket. This is best-effort, not bulletproof — a secret that doesn't match any known prefix will slip through. That's acceptable for M2.3; §6 open question 8 in the threat model explicitly accepts this limit.

## Out of scope for M2.3 (explicit deferrals)

To keep scope honest:

### Deferred to M3 (dashboard read-only)

- Secret browsing view: "show me all secrets in the vault, their paths, provenance, last rotation"
- Audit log viewer: "show me `vault_access_log` filtered by agent instance, ticket, time range, outcome"
- Stale-secret surfacing: "show me secrets past rotation deadline" — the data is in `secret_metadata.rotation_cadence` + `last_rotated_at`; M3 renders it
- Role-to-secret mapping viewer: "which secrets can role X access?" and "which roles can access secret Y?"

### Deferred to M4 (dashboard write)

- Secret CRUD (create, edit, delete with "this secret is used by roles X, Y, Z" warning)
- Role-to-secret grant editing (add/remove rows in `agent_role_secrets`)
- Rotation initiation UI (operator clicks "rotate this secret"; Garrison calls Infisical's rotation API or prompts for new value)
- Environment and path tree management
- Error handling UI (operator-facing messages for the failure modes the supervisor enumerates in M2.3)

### Explicitly out of scope for M2.3/M3/M4 entirely

- **Workspace sandboxing**: agents with vault-injected secrets can still write those secrets to arbitrary host filesystem paths. This is a real concern, tracked at `docs/issues/agent-workspace-sandboxing.md`, planned for post-M2.3 Docker-per-agent work. NOT a gate on M2.3 shipping.
- **Multi-tenancy**: the data model supports it (every table has `customer_id`) but no per-customer isolation is enforced at M2.3. Customer-delegated OAuth tokens are still operationally the operator's; true multi-tenancy is a future additive migration.
- **Intentional operator leakage mitigations**: if the operator pastes a secret into a chat message, commits a `.env`, or otherwise bypasses Garrison's vault surface, the vault cannot save them. Per threat model §3 explicitly accepted threats.

## Open questions the spec must resolve

Questions that should not be pre-decided in context and belong in `/speckit.specify`:

### From the threat model (§6), still open

1. **Environment model**: one Infisical environment for all of Garrison, or Infisical environments (dev/staging/prod) for Garrison's own split? Default lean: single environment, path conventions carry Garrison's semantics.

2. **Machine identity scope**: one identity per supervisor *instance* (what M2.3 currently assumes), or one identity per agent_instance *spawn*? Per-spawn is more secure but increases Infisical API load. The spec decides after measuring current supervisor spawn rate from M2.2.x data.

3. **Secret value injection mechanism**: env vars only (what M2.3 currently assumes), or env vars plus a mounted `/secrets/` directory for multi-line secrets (PEM keys, certificates)? Env vars have OS-level size limits that PEM-format values can hit.

4. **Audit log retention**: Infisical community edition's audit log has a retention cap. Garrison's `vault_access_log` in Postgres has no such cap but costs storage. Spec decides retention policy and whether Garrison actively exports from Infisical to Postgres for longer retention.

5. **Secret discovery at spawn time**: SQL join on `agent_role_secrets` at spawn time (M2.3 default), pre-computed denormalized column on `agents`, or Infisical's own access policy? Default commits to SQL join; spec can override.

6. **Bootstrap secret delivery**: environment variables set by Coolify (M2.3 default), read from a restricted file at startup, or a different mechanism? Spec documents the exact mechanism and the operator discipline around it.

7. **Secret-pattern scanning patterns and action**: the list of regexes to scan for, and what the scanner does on match (redact + hygiene warning is M2.3 default, but alternatives include blocking the MemPalace write entirely).

### New questions for M2.3 specifically

8. **Infisical version pinning policy**: major-version upgrades need review per threat model §3 (explicitly accepted risk: Infisical supply-chain compromise). Spec commits to a pinning discipline — specific digest vs. minor-version range vs. latest.

9. **Dual-logging failure mode**: what happens if the Garrison-side `vault_access_log` write fails but the Infisical access succeeded? The secret IS fetched, the agent IS about to get it, but Garrison has no record. Options: fail closed (roll back the secret fetch, fail the spawn), fail open with a critical error (proceed but emit a hygiene-critical alert), fail open with retry (proceed and retry the log write async).

10. **Per-customer secret namespacing at the Infisical path level**: even though M2.3 is single-tenant, should the Infisical path convention include `customer_id` as a prefix to reduce future migration pain? Default lean: yes, paths like `/<customer_id>/operator/stripe_key`. Spec confirms.

11. **Test fixtures for Infisical**: integration tests need a real-ish Infisical to exercise against. Options: testcontainer Infisical (adds deps and boot time), mock Infisical (risk of mock drift), separate test-only Infisical instance in CI. Spec decides test infrastructure.

12. **Migration path for agent_role_secrets**: M2.2's seed migrations created the engineer and qa-engineer agent roles. Those roles currently require no secrets. M2.3's migrations need to establish both the empty grant rows AND the pattern for future roles to add grants. Spec commits to the convention.

## Acceptance criteria framing

The M2.3 spec will commit to specific success criteria, but the context-level framing is:

1. **Vault plumbing works**: supervisor reads a secret from Infisical, injects it as an env var, agent uses it successfully in a tool call. Tested end-to-end.

2. **Rules 1-6 are provable**: automated tests demonstrate each rule. Rule 1 (value not in context window) is testable by inspecting the `claude` invocation args. Rule 2 (per-role scoping) is testable by running an agent without a grant and confirming the env var is absent. Rule 3 (no vault MCP tool) is testable by inspecting the MCP config passed to `claude`. Rule 4 (provenance tagging) is testable by querying `secret_metadata`. Rule 5 (Infisical handles crypto) is a deployment assertion. Rule 6 (audit everything, log no values) is testable by grepping `slog` output for secret values after a run.

3. **Failure modes are graceful**: every Infisical API failure maps to a supervisor adjudicator outcome. No unhandled errors. Hygiene sees the new failure categories as specific `exit_reason` values.

4. **Data model supports M3/M4**: the three new tables are populated end-to-end during M2.3's acceptance runs, so M3's dashboard work has real data to consume.

5. **Zero new Go dependencies except Infisical SDK**: Garrison's locked-deps discipline holds. The Infisical Go SDK is the one addition; all other work uses existing stdlib + pgx + slog patterns.
