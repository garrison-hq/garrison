# Feature specification: M2.2 — MemPalace MCP wiring

**Feature branch**: `004-m2-2-mempalace`
**Created**: 2026-04-22
**Status**: Draft
**Input**: `/garrison-specify m2-2` — "Wire MemPalace into every agent spawn; validate or falsify the cross-agent-memory thesis."

This spec is bound by `specs/_context/m2.2-context.md`. Decisions settled there — the palace path default, the wake-up injection template, the MCP config extension shape, the hygiene checker shape, the $0.10 per-invocation budget cap, the two-role scope (engineer + qa-engineer), the scope exclusions, and every "Binding question" answer — are inputs, not questions. Where this spec cites a constraint, the authoritative text lives in that file; consult it rather than the paraphrase here.

Tool behaviour of MemPalace (CLI surface, MCP tool inventory, init idempotency characterization, concurrent-write semantics) is characterized in `docs/research/m2-spike.md` Part 2. Where this spec references specific behaviour (e.g. "wings are created lazily by `add_drawer`"), the citation is into the spike by section number. The spec does not re-derive behaviour; it translates the context file's binding constraints into testable, user-story-shaped requirements.

M2.2 builds on M2.1 (see `specs/003-m2-1-claude-invocation/spec.md` and `docs/retros/m2-1.md`). The supervisor, stream-json event routing, init-event MCP health-check, per-invocation MCP config file, process-group termination, pipeline-drain-before-Wait discipline, exit-reason vocabulary, and `total_cost_usd` capture all remain as shipped in M2.1. M2.2 extends the MCP config to include a second server, adds wake-up context injection, expands the engineer's agent.md to execute the full MemPalace write contract, adds a QA engineer role, and adds an async hygiene checker. Nothing else about M2.1's architecture is revisited.

This is the milestone that validates or falsifies the memory thesis (RATIONALE §3). If the palace produces useful cross-agent memory under real work, the downstream roadmap is extension. If it doesn't, the architecture is reconsidered before M3+.

## Clarifications

### Session 2026-04-22

- Q: MemPalace MCP server stdio invocation for MemPalace 3.3.2? → A: `mempalace mcp` — the binary subcommand confirmed by spike §3.1. Exact args (e.g. `--palace <path>`) resolved against MemPalace 3.3.2 docs at implementation time; the supervisor does not embed a `python -m …` fallback.
- Q: MemPalace version pin for M2.2? → A: **3.3.2** (spike-validated; no "or newer" hedge). Upgrades are a future-milestone decision.
- Q: KG triple shape for the two-agent handoff? → A: triples are keyed on `ticket_<id>` as entity (per ARCHITECTURE.md "KG triple conventions"). Engineer writes at minimum `(agent_instance_<id>, completed, ticket_<id>)` and `(<artifact_path>, created_in, ticket_<id>)`. QA engineer and hygiene checker both query via `mempalace_kg_query(entity='ticket_<id>')` — direction-agnostic; returns all triples mentioning the ticket.
- Q: Palace bootstrap strategy when `~/.garrison/palace/` already exists? → A: **detect first, init only if missing**. The supervisor checks for `<palace_path>/chroma.sqlite3` (the canonical on-disk marker per spike §3.5); if present, skip init and log `palace_already_initialized`; if absent, run `mempalace init --yes` and fail startup loudly on any non-zero exit. The supervisor does not depend on MemPalace's own idempotency. *(**Superseded 2026-04-23 F1**: `mempalace init --yes` IS idempotent in 3.3.2; `chroma.sqlite3` is a post-first-write marker, not post-init. Unconditional `init --yes` at startup is simpler and correct. See Session 2026-04-23 F1.)*
- Q: Does the M2.2 migration introduce any Postgres tables for palace metadata? → A: **No.** All palace state lives in MemPalace's own SQLite + Chroma store (spike §3.5); adding a Postgres mirror would duplicate state across stores and violate RATIONALE §2. The `garrison_agent_mempalace` role is therefore SELECT-only on existing M2.1 tables (`ticket_transitions`, `agent_instances`, `tickets`). No INSERT/UPDATE grants.

### Session 2026-04-23

- Q: How does a successful agent (engineer or qa-engineer) effect the ticket-column transition from `in_dev → qa_review` or `qa_review → done`? → A: **Supervisor-side adjudication** (matches M2.1). The agent completes its MemPalace write contract and emits a terminal `result` event with `is_error=false`; the supervisor's terminal transaction then inserts the `ticket_transitions` row and updates `tickets.column_slug` atomically, in the same tx as `agent_instances` terminal + `event_outbox.processed_at`. Agents do NOT issue Postgres writes for column transitions. The Postgres MCP server exposed to every agent (via the M2.1 `garrison_agent_ro` role) stays SELECT-only; `garrison_agent_mempalace` (hygiene checker reads) is also SELECT-only. No new writable Postgres role is introduced in M2.2.
- Q: Plan-phase correction — MemPalace MCP server stdio invocation for 3.3.2. → A: **`python -m mempalace.mcp_server --palace <path>`.** Session 2026-04-22 Q1's answer (`mempalace mcp`) is factually wrong for MemPalace 3.3.2: `mempalace mcp` is a help-text printer that prints setup instructions to stdout and exits; it is not an MCP server. Verified by direct check of the 3.3.2 package source and by running `mempalace mcp --help` against the spike's installed version. The supervisor's MCP config `mempalace` entry therefore invokes `python -m mempalace.mcp_server --palace <path>` (the module entry point documented by `mempalace mcp`'s own output). This supersedes Session 2026-04-22 Q1.
- Q: Does `mempalace.mcp_server` require authentication beyond `--palace`? → A: **No.** The module's `--help` exposes only `--palace`; no API key, no token, no env-based auth. Filesystem scope of the palace path is the sole trust boundary. Closes Clarify item C in §Open questions.
- Q: Deployment shape for MemPalace (sidecar container vs. local binary on supervisor host)? → A: **Sidecar container (`garrison-mempalace`) + socket-proxy (`garrison-docker-proxy`).** Supervisor stays Go-only; MemPalace's Python runtime is isolated in its own container. All supervisor-side invocations (palace bootstrap, wake-up, MCP server for Claude, hygiene palace queries) go through `docker exec` against the sidecar, via a filtered `linuxserver/socket-proxy` endpoint (POST=1 EXEC=1 CONTAINERS=1 only; everything else default-deny). Claude's per-invocation MCP config's `mempalace` entry has `command: "docker"`, `args: ["exec", "-i", "garrison-mempalace", "python", "-m", "mempalace.mcp_server", "--palace", "/palace"]`, `env: {DOCKER_HOST: "tcp://garrison-docker-proxy:2375"}` *(socket-path detail superseded 2026-04-23 F5 — linuxserver/socket-proxy is a TCP HAProxy at :2375, not a unix-socket proxy)*. The palace path (`/palace`) is a Docker-named-volume mount inside the MemPalace container. FRs 203/204/207/213 and NFRs 208/213 — which were written assuming a local `mempalace` binary on the supervisor's `$PATH` — retain their *behavioural* requirements (wake-up returns ~170 tokens; MCP serves 29 tools; init marker detected before init; per-invocation MCP config has both entries) but their *implementation mechanism* shifts to docker-exec as described above. See `plan.md` §"Deployment topology" for the concrete call shapes and §"Spec-level findings" for the full list of affected FRs.
- Q: Engineer's `listens_for` value for M2.2 acceptance. → A: **`["work.ticket.created.engineering.in_dev"]`.** Per FR-228's "scope simplicity" default — M2.2 acceptance tests insert tickets directly at column `in_dev`; no CTO agent is introduced to move tickets from `todo → in_dev`. The engineer's handler therefore registers against the `created.engineering.in_dev` channel (M1's `emit_ticket_created` trigger fires on INSERT to `tickets`, qualified by the inserted `column_slug`). The `todo` column remains in the seeded workflow so human/test operators can still use it; no agent is registered against `created.engineering.todo` in M2.2.
- Q: `mempalace_add_drawer` parameter shape. → A: **`wing`, `room`, `content`** — per spike §3.6 (signature is `wing,room,content`, all required). The `room` argument is the hall designation (e.g. `hall_events`, `hall_discoveries`) per ARCHITECTURE.md "wings and halls" taxonomy. Seed `agent.md` content writes drawers to `room="hall_events"` by default; other halls (e.g. `hall_discoveries` for M3+ when future-learning patterns are recorded) remain available to agents but are not required by M2.2's completion protocol.
- Q: How does an agent learn its own `agent_instance_id` at turn start, so it can emit the required KG triple `(agent_instance_<instance_id>, completed, ticket_<ticket_id>)`? → A: **Injected into the `--system-prompt`'s "This turn" block** alongside the ticket_id. FR-207a's template is extended to read `"You have been spawned as agent_instance <instance_id> to work ticket <ticket_id>. Read it, then execute your completion protocol."` The supervisor substitutes both placeholders at spawn time from the `agent_instances.id` it just inserted (step 1 of `spawn.runRealClaude`'s dedupe transaction, committed before `cmd.Start`). The `-p` task description is NOT duplicated; no new MCP tool (e.g. `whoami`) is introduced; the agent does NOT query Postgres to discover its id.
- Q: T001 spike finding F1 — palace bootstrap marker file. → A: **Drop detect-first; always run `mempalace init --yes` at startup.** T001 observed that `mempalace init --yes` produces only `mempalace.yaml`, NOT `chroma.sqlite3` (the ChromaDB file appears only after the first write via `add_drawer` or `mine`). Session 2026-04-22 Q4's marker assumption is factually wrong for 3.3.2. T001 also observed that `mempalace init --yes` is **idempotent** — running it a second time against an initialized palace exits 0 and re-writes the identical yaml. The simpler bootstrap is therefore unconditional `mempalace init --yes /palace` at every supervisor startup; the supervisor logs `palace_initialized` on success and exits startup on any non-zero. This supersedes Session 2026-04-22 Q4 and FR-201's marker-check-then-conditional-init logic.
- Q: T001 spike finding F2 — `mempalace wake-up` flag shape in 3.3.2. → A: **`mempalace --palace <path> wake-up --wing <wing>`.** T001 observed that `mempalace wake-up --help` in 3.3.2 exposes only `--wing`; there is NO `--max-tokens` flag and `--palace` is NOT accepted by the subcommand (it must be at the top level). Wake-up output is implicitly bounded by MemPalace's internal L0+L1 renderer (~79 tokens observed for a 1-drawer wing; spike §3.1 saw ~170 tokens for typical content). The supervisor accepts stdout verbatim and injects it into `--system-prompt`. This supersedes FR-207's `--max-tokens 200` wording and deletes NFR-209 entirely.
- Q: T001 spike finding F3 — `Dockerfile.mempalace` base image. → A: **`python:3.11-slim`**, not `python:3.11-alpine`. T001 observed that chromadb's Rust-compiled transitives (tokenizers et al.) have no musl wheels on PyPI; `pip install mempalace==3.3.2` inside alpine fails in a source build step that needs the Rust toolchain the alpine base doesn't carry. `python:3.11-slim` (debian-glibc) installs from prebuilt wheels directly. Measured image size: 478 MB. Plan §"`supervisor/Dockerfile.mempalace` (new)" is amended accordingly.
- Q: T001 spike finding F4 — pip hash-pinning discipline for `Dockerfile.mempalace`. → A: **Committed `requirements-lock.txt` generated via `pip-compile`**, installed via `pip install --no-cache-dir --require-hashes -r requirements-lock.txt`. T001 observed that `pip install --require-hashes mempalace==3.3.2 --hash=sha256:...` fails because `--require-hashes` requires ALL transitive deps pinned with `==`; mempalace's `chromadb<2,>=1.5.4` range doesn't satisfy that. The lockfile approach produces a ~40-line `requirements-lock.txt` with every transitive `==`-pinned and hash-locked. Upgrading MemPalace regenerates the lockfile. The `--require-hashes` rigor from NFR-213 is preserved; only the mechanism shifts (lockfile vs. single-line). Canonical mempalace 3.3.2 wheel SHA256 (for pinning `mempalace` itself): `sha256:cb288e8028d26dfb384125baecfcf584aa2ba5a30a216ff82d9745af070d5e45`.
- Q: T001 spike finding F5 — socket-proxy transport shape. → A: **TCP `:2375` via compose network, not a unix socket.** T001 observed that `linuxserver/socket-proxy:latest` is a HAProxy-based HTTP proxy listening on TCP port 2375 inside its container; it does NOT expose a unix-domain socket. The Session 2026-04-23 Deployment clarification's `DOCKER_HOST=unix:///var/run/docker-proxy.sock` wording is factually wrong for this image. **Corrected shape**: supervisor env carries `DOCKER_HOST=tcp://garrison-docker-proxy:2375`; MCP config's mempalace-entry `env` block carries the same; compose exposes port 2375 over the project network (no published host port); the `docker-proxy-sock` shared named volume is dropped entirely. Proxy filter behaviour is unchanged — `POST=1 EXEC=1 CONTAINERS=1` with everything else default-deny, verified end-to-end in T001 (EXEC allowed; VOLUMES/IMAGES denied with HTTP 403). This supersedes the Session 2026-04-23 Deployment clarification's socket-path detail and cascades into the plan's deployment topology, T002 compose, T009 `GARRISON_DOCKER_HOST` default, T011 mcpconfig env, and T015 env propagation.
- Q: T001 spike finding F6 — `procps` in the MemPalace sidecar image. → A: `Dockerfile.mempalace` adds `apt-get install -y procps` so human operators running `docker exec garrison-mempalace ps aux` get sensible output. Test code (chaos and hygiene) continues to use `/proc/[0-9]*/cmdline` directly — tool-agnostic, robust against future image-slimming. Neither `ps` nor `pgrep` is in `python:3.11-slim` by default; this is a one-line fix. Adds ~1 MB to the image.

## User scenarios and testing (mandatory)

### User story 1 — engineer agent writes MemPalace during ticket completion (priority: P1)

The operator inserts an engineering ticket. The supervisor spawns a Claude Code subprocess with both the Postgres MCP server (from M2.1) and the MemPalace MCP server (new) wired into the per-invocation config. The engineer receives a wake-up context injection read from `wing_frontend_engineer` ahead of the turn. It reads the ticket via Postgres MCP, searches its wing for prior related work via MemPalace MCP, writes `changes/hello-<ticket_id>.md` to its workspace, writes a diary entry to `wing_frontend_engineer` via `mempalace_add_drawer`, writes at least two KG triples mentioning the ticket via `mempalace_kg_add`, and transitions the ticket from `in_dev` to `qa_review`. The supervisor captures the terminal `result` event, records `total_cost_usd` (≤ $0.10), structured-logs every `mempalace_*` `tool_use` event, and writes the terminal `agent_instances` row.

**Why this priority**: this is the smallest slice that proves (a) a second MCP server can be added to the per-invocation config without regressing M2.1's init-parse contract, (b) wake-up context injection actually reaches the subprocess, (c) the MemPalace write contract from `ARCHITECTURE.md` can be executed by a real Claude subprocess in a real workspace, and (d) the $0.10 per-invocation cap is realistic for the palace-enabled flow. If this story ships, the mechanical half of the memory thesis (writes land, cost is bounded) is proven. User Story 2 proves the reads-land half.

**Independent test**: against a freshly-migrated Postgres with M2.2 seeds applied, a bootstrapped palace at the configured path, the `claude` and `mempalace` binaries on `$PATH`, and the supervisor running, insert one ticket targeting `engineering` at column `todo`. Progress the ticket to `in_dev` (or seed a ticket directly at `in_dev` to exercise the engineer's trigger). Observe: a single `agent_instances` row with `status='succeeded'`, `exit_reason='completed'`, `total_cost_usd` populated and ≤ $0.10, `wake_up_status='ok'`; exactly one `ticket_transitions` row for `in_dev → qa_review`; exactly one diary entry in `wing_frontend_engineer` whose body mentions the ticket id; at least two KG triples mentioning the ticket; and a structured slog line at `info` level for each `mempalace_*` `tool_use` event emitted on stream-json.

**Acceptance scenarios**:

1. **Given** a clean database with M1+M2.1+M2.2 migrations applied, the engineering department, engineer role (with `palace_wing='wing_frontend_engineer'`), and qa-engineer role seeded, the `claude` and `mempalace` binaries on `$PATH`, and a writable palace path (default `~/.garrison/palace/`), **When** the supervisor starts, **Then** on startup it ensures `mempalace init --yes` has run against the configured path (idempotent if already initialized; see Edge Cases), and thereafter the path contains a valid palace.
2. **Given** a ticket at column `in_dev` for `engineering`, **When** the supervisor spawns the engineer invocation, **Then** (a) the per-invocation MCP config file contains both `postgres` and `mempalace` entries per "Spawn contract — MCP config file extensions" in `m2.2-context.md`, (b) the engineer's `--system-prompt` concatenates `agent.md` content, the `mempalace --palace <path> wake-up --wing wing_frontend_engineer` stdout (flag shape corrected per clarification 2026-04-23 F2), and the "This turn" ticket hint per "Wake-up injection contract" in `m2.2-context.md`, and (c) the `--max-budget-usd` flag is set to `0.10` per "Binding question 8".
3. **Given** the Claude subprocess has started, **When** the supervisor reads the `system`/`init` event, **Then** it iterates `mcp_servers[]` and confirms both `postgres` and `mempalace` entries have `status == "connected"` (per M2.1's init-parse contract, extended by FR-208). Either one not connected causes the supervisor to signal the process group per M2.1's termination contract and record a failed `agent_instances` row with an `exit_reason` of the form `mcp_<server>_<status>`.
4. **Given** MCP health passed and the agent executes the completion protocol from `ARCHITECTURE.md` "MemPalace write contract" and `migrations/seed/engineer.md`, **When** the agent emits `tool_use` events whose `name` starts with `mempalace_`, **Then** the supervisor logs each at `info` level with structured fields `ticket_id`, `agent_instance_id`, `tool_name`, and `outcome` (from the subsequent `tool_result` event). These events are observational; they do not drive dispatch decisions.
5. **Given** the agent completes its completion protocol and emits the terminal `result` event with `is_error == false` and `terminal_reason == "completed"`, **When** the supervisor writes the terminal transaction, **Then** it updates `agent_instances` to `status='succeeded'`, `exit_reason='completed'`, sets `finished_at`, populates `total_cost_usd`, and sets `wake_up_status='ok'` (or `'failed'` / `'skipped'` per US5); inserts a `ticket_transitions` row for `in_dev → qa_review` with `triggered_by_agent_instance_id` set and `hygiene_status` initially NULL (annotated later by the hygiene checker per US3); and sets `event_outbox.processed_at`.
6. **Given** `total_cost_usd` for this invocation, **When** compared to the per-invocation cap from "Binding question 8" of `m2.2-context.md`, **Then** the captured cost is strictly less than $0.10 for the representative "engineer does one palace-enabled ticket" workflow. A value above the cap is a finding for the retro (either the cap is wrong or the agent flow is inefficient).

### User story 2 — QA engineer reads the engineer's MemPalace writes and completes the handoff (priority: P1)

The transition from `in_dev` to `qa_review` fires `pg_notify('work.ticket.transitioned.engineering.in_dev.qa_review', <ticket_id>)`. The supervisor spawns a QA engineer invocation. The QA agent receives a wake-up from `wing_qa_engineer`, reads the ticket via Postgres MCP, searches MemPalace for the engineer's diary entry (`mempalace_search` returns it), verifies the engineer's output exists and reads sensibly, writes its own diary entry to `wing_qa_engineer`, writes its own KG triples, and transitions the ticket from `qa_review` to `done`. The supervisor captures the terminal row. Both agents' `total_cost_usd` together are under the combined cap.

**Why this priority**: the entire point of the memory thesis (RATIONALE §3) is that Agent B reads Agent A's memory. A single-agent run does not exercise this. Until a second agent successfully reads the first agent's writes, the palace is unvalidated institutional memory — code that could be broken and we wouldn't know. This story is what makes the thesis testable. It is equal in priority to US1; functionally both are ship-blockers.

**Independent test**: pre-populate `wing_frontend_engineer` with a plausible diary entry + KG triples for a pre-seeded ticket (bypassing the engineer spawn). Manually transition the ticket to `qa_review`. Observe: the QA engineer spawns, its `mempalace_search` call returns the pre-populated diary entry (visible in structured logs as a `tool_use` + `tool_result` pair), the agent writes a diary to `wing_qa_engineer`, writes at least one KG triple, and transitions the ticket to `done`. For the full end-to-end flow, combine with US1: one ticket routed through `todo → in_dev → qa_review → done` produces two agent_instances with the correct diary/KG writes in two different wings, and `SUM(total_cost_usd)` over both rows is strictly less than $0.20.

**Acceptance scenarios**:

1. **Given** a ticket transition from `in_dev` to `qa_review` with the engineer's diary entry and KG triples already in the palace, **When** the supervisor's dispatcher receives the notification on `work.ticket.transitioned.engineering.in_dev.qa_review`, **Then** it spawns a QA engineer invocation (per the seed row for `role_slug='qa-engineer'`) with both MCP servers configured identically to US1's scenario 2.
2. **Given** the QA subprocess is running with both MCP servers connected, **When** the agent executes the search step of `migrations/seed/qa-engineer.md`'s completion protocol, **Then** one `tool_use` event with `name='mempalace_search'` is emitted, its `tool_result` returns the engineer's prior diary entry (body contains the ticket id), and the QA agent proceeds to the implement-then-write step.
3. **Given** the QA agent has verified the engineer's output and writes its own diary, **When** it emits `mempalace_add_drawer` with `wing='wing_qa_engineer'`, **Then** the write lands in `wing_qa_engineer` and the body mentions the ticket id.
4. **Given** the QA agent has completed the write contract, **When** it issues the Postgres transition to `done`, **Then** the supervisor's terminal transaction writes the second `agent_instances` row and the `qa_review → done` `ticket_transitions` row exactly as in US1's scenario 5.
5. **Given** both `agent_instances` rows for the two-agent run have populated `total_cost_usd`, **When** summed, **Then** the combined cost is strictly less than $0.20 (two invocations × $0.10 cap).

### User story 3 — hygiene checker annotates `ticket_transitions.hygiene_status` (priority: P2)

For every inserted `ticket_transitions` row, an async checker running as a single goroutine inside the supervisor waits 5 seconds, queries MemPalace for the expected writes (a diary drawer in the agent's wing referencing the ticket id, within the agent_instance's run window, with body ≥ 100 chars; at least one KG triple mentioning the ticket), evaluates a status, and updates the `ticket_transitions.hygiene_status` column to one of `'clean'`, `'missing_diary'`, `'missing_kg'`, `'thin'`, or `'pending'`. Rows that resolve to `'pending'` (palace unreachable, query timeout) are retried by a periodic sweep at 60-second cadence. The checker writes at-most-once per row per resolution attempt; the periodic sweep is the retry mechanism.

**Why this priority**: the hygiene dashboard is M3, but the *data* that dashboard consumes must be produced starting M2.2 — otherwise M3 has nothing to render. Soft gates (RATIONALE §5) become operative only when there is a persisted signal to surface. This story makes that data real. It is P2 (not P1) because the ticket flow completes correctly even when `hygiene_status` is NULL or stuck at `'pending'` — per soft-gates, transitions never block on hygiene — but a milestone that ships without producing the column values is a milestone that hasn't validated its own thesis.

**Independent test**: directly insert a `ticket_transitions` row with a known ticket id, with an associated `agent_instances` row and a manually-crafted MemPalace state (no diary / thin diary / diary but no KG / clean / etc.). Observe: within 10 seconds of the insert, `ticket_transitions.hygiene_status` has been updated to the expected value. For each of the five status values, there is at least one test configuration that reliably produces it. Induce a palace-unreachable condition (stop the MemPalace MCP server, block the checker's DSN, or similar) and confirm `'pending'` is written; restore the palace and confirm the next sweep cycle updates the row to its resolved status.

**Acceptance scenarios**:

1. **Given** the supervisor is running and the hygiene checker goroutine is listening on `work.ticket.transitioned.*.*`, **When** a `ticket_transitions` row is inserted (by an agent transition or by a test fixture), **Then** the checker receives the notification, waits 5 seconds (tunable via `GARRISON_HYGIENE_DELAY`), queries MemPalace against the `agent_instances` run window for the transition's `triggered_by_agent_instance_id`, evaluates status per "Hygiene checker (binding shape)" in `m2.2-context.md`, and `UPDATE`s `ticket_transitions.hygiene_status`.
2. **Given** the checker's query finds a diary drawer whose body mentions the ticket id (length ≥ 100 chars) and at least one KG triple mentioning the ticket, **When** the checker evaluates, **Then** `hygiene_status='clean'`.
3. **Given** the checker's query finds no matching diary drawer, **When** evaluated, **Then** `hygiene_status='missing_diary'`.
4. **Given** the checker's query finds a matching diary drawer with body length < 100 chars, **When** evaluated, **Then** `hygiene_status='thin'`.
5. **Given** the checker's query finds a matching diary drawer but no matching KG triple, **When** evaluated, **Then** `hygiene_status='missing_kg'`.
6. **Given** the checker's palace query errors or times out, **When** evaluated, **Then** `hygiene_status='pending'` and the periodic sweep reprocesses the row on its next cycle (default 60 s, tunable via `GARRISON_HYGIENE_SWEEP_INTERVAL`), resolving to a terminal status or remaining `'pending'` until the palace returns.
7. **Given** the checker processes a row to a non-`'pending'` terminal status, **When** the same ticket_transition row is observed again (replay, duplicate notification), **Then** the status is NOT overwritten. At-most-once-to-terminal per row.

### User story 4 — supervisor bails fast on MemPalace MCP failure (priority: P2)

The MemPalace MCP server fails to start for an invocation (binary missing, palace path unreadable, malformed MCP config JSON, or any other pre-`"connected"` condition). The supervisor detects this from the `system`/`init` event's `mcp_servers[]` array — extended from M2.1's single-server check to cover both `postgres` and `mempalace` — signals the Claude process group within 2 seconds of init-event receipt, and writes a failed `agent_instances` row with `exit_reason='mcp_mempalace_<status>'` (e.g. `mcp_mempalace_failed`). Similarly, if the MemPalace MCP dies mid-run, the subprocess's next MCP tool call fails and the run terminates with the M2.1 adjudication path (`mcp_mempalace_failed` if a new invocation observes `status != "connected"`; the run's terminal result dictates otherwise).

**Why this priority**: the M2.1 spec (US2 / SC-103) established the init-parse contract. M2.2 adds a second required server. If the supervisor only health-checks `postgres`, a broken `mempalace` would silently allow the run — the engineer would burn tokens, MCP tool calls would fail, and the ticket state would end up in an undefined place. The fail-closed extension is what preserves the M2.1 correctness invariant when there are now two required servers. It is P2 (not P1) only because US1/US2 must work first; functionally it is a ship blocker equal to US1.

**Independent test**: configure the supervisor to inject a deliberately-broken MemPalace MCP config (e.g. `command` points at `/bin/does-not-exist`, or `MEMPALACE_PATH` points at an unreadable directory) for a test ticket. Insert the ticket. Separately, arrange for the MemPalace MCP subprocess to die mid-run (signal it externally between the init event and the terminal result). Observe: the run terminates (either via a non-zero subprocess exit, a parse error, or a subsequent tool-use error) and the adjudication path writes a failed `agent_instances` row.

**Acceptance scenarios**:

1. **Given** a spawn wired with a broken MemPalace MCP config, **When** Claude emits the `system`/`init` event and the `mempalace` entry has `status != "connected"`, **Then** the supervisor sends SIGTERM to the Claude process group (`syscall.Kill(-pgid, SIGTERM)`) within 2 seconds of init-event receipt (NFR-106 from M2.1 applies unchanged, extended to the mempalace server), waits the 2-second grace, then escalates to SIGKILL on the same group.
2. **Given** the process group has been terminated on MemPalace MCP health failure, **When** the supervisor writes the terminal transaction, **Then** the `agent_instances` row has `status='failed'`, `exit_reason` of the form `mcp_mempalace_<status>` (where `<status>` is the verbatim string from the init event, e.g. `failed`, `needs-auth`, or any unknown value per M2.1's fail-closed discipline), `finished_at` set, and `total_cost_usd` NULL.
3. **Given** the supervisor has checked both `postgres` and `mempalace` during init-parse, **When** both are `"connected"`, **Then** the subprocess is allowed to proceed (preserving M2.1's happy path).
4. **Given** the MemPalace MCP subprocess dies mid-run after a successful init (signalled externally; disk unmounted; the process exits non-zero), **When** the next MCP tool call on the agent's side fails and propagates up to a terminal state, **Then** the supervisor's existing adjudication path (`result` event, or `no_result` on missing result, or `parse_error` on malformed JSON) writes an appropriate failed `agent_instances` row. No new adjudication path is required; the M2.1 terminal-state machine covers this.

### User story 5 — wake-up context injection is non-blocking on failure (priority: P3)

Before spawning a Claude subprocess, the supervisor runs `mempalace --palace <path> wake-up --wing <wing>` (flag shape per clarification 2026-04-23 F2 — `--palace` is a top-level flag; `--max-tokens` does not exist in 3.3.2) with a 2-second timeout. If the command exits non-zero, times out, or cannot be executed, the spawn proceeds without wake-up context (the `--system-prompt` contains the agent.md content followed by the "This turn" ticket hint, with no wake-up block in between). The supervisor logs the failure at `warn` level and records `wake_up_status='failed'` on the `agent_instances` row. A successful wake-up records `wake_up_status='ok'`.

**Why this priority**: wake-up is an optimization, not a correctness concern (per "Binding question 4" of `m2.2-context.md`). The agent.md's completion protocol remains authoritative; the wake-up adds context density at the margin. Failing the spawn on wake-up error would introduce a non-essential coupling between the optimization layer and the spawn contract. Non-blocking is the principled choice. Priority P3 because the failure path is exceptional; US1/US2/US3/US4 cover the common case.

**Independent test**: point `GARRISON_PALACE_PATH` at a dir the `mempalace` binary cannot read (or replace the `mempalace` binary on `$PATH` with `/bin/false` for the test run). Insert a ticket. Observe: the supervisor logs a warn-level line `wake_up_failed` with the underlying error and the elapsed time; the Claude subprocess spawns regardless; the resulting `agent_instances` row has `wake_up_status='failed'`; the agent completes the ticket normally (US1 acceptance criteria still hold, excluding criterion 6 about wake-up context being visible to the agent). Separately, run a normal invocation with a healthy palace and confirm `wake_up_status='ok'`.

**Acceptance scenarios**:

1. **Given** `mempalace --palace <path> wake-up --wing <wing>` returns exit 0 within the 2-second timeout, **When** the supervisor builds `--system-prompt`, **Then** the stdout is inserted between the `agent.md` content and the "This turn" block per the template in "Wake-up injection contract" of `m2.2-context.md`, and `wake_up_status='ok'` is persisted on the terminal `agent_instances` row.
2. **Given** the wake-up command exits non-zero or times out, **When** the supervisor builds `--system-prompt`, **Then** no wake-up block is included (the `agent.md` content is followed directly by the "This turn" block), the failure is logged at `warn` level with the underlying error and the elapsed time, the spawn proceeds, and `wake_up_status='failed'` is persisted on the terminal `agent_instances` row.
3. **Given** the `mempalace` binary is missing from `$PATH`, **When** the supervisor attempts to exec the wake-up command, **Then** the same non-blocking path runs and the warn log carries the `exec: "mempalace": executable file not found in $PATH` error.
4. **Given** a wake-up completes after the 2-second timeout, **When** the supervisor has already proceeded with the spawn, **Then** the late-arriving stdout is discarded (the command's context has been cancelled and the process reaped). No split-state where part of the wake-up made it into the prompt.

### Edge cases

- **Palace path already initialized from a prior run**: supervisor startup unconditionally runs `mempalace init --yes <palace_path>` per clarification 2026-04-23 F1 (superseding 2026-04-22 Q4's marker-check heuristic). Because `mempalace init --yes` is idempotent in 3.3.2 — a second call exits 0 and re-writes the identical `mempalace.yaml` — the palace-already-initialized case is indistinguishable from first-time initialization at the supervisor level; the supervisor logs `palace_initialized=true` either way. If the palace is in a half-initialized state (corrupt `chroma.sqlite3`, etc.), the operator runs `rm -rf` on the palace path per ops checklist guidance, and the supervisor's next startup re-initializes cleanly.
- **Palace path not writable at supervisor startup**: supervisor fails startup loudly (exit non-zero; error log identifies the path and the permission failure). This is a configuration problem the operator must fix; the supervisor does not attempt to chown/chmod the path.
- **Palace path co-located with a git-tracked directory**: prevented by the default path choice (`~/.garrison/palace/` is outside any Garrison checkout) and by ops checklist guidance. If the operator overrides `GARRISON_PALACE_PATH` to a git-tracked path, the supervisor does not prevent this — but AGENTS.md "what agents should not do" flags this as an operator error, and `mempalace init` will auto-mutate `.gitignore` of the target directory per spike §3.2.
- **Wake-up stdout is large**: per clarification 2026-04-23 F2 there is no `--max-tokens` flag; MemPalace's internal L0+L1 renderer bounds output size (~79–170 tokens observed). The supervisor inserts stdout verbatim regardless of length; no trimming. Budget impact is absorbed by the $0.10 per-invocation cap.
- **Wake-up returns empty string**: treated as `wake_up_status='ok'` with an empty wake-up block in the `--system-prompt`. No warning; empty wings are a legitimate steady state (first invocation of a role, recently-pruned wing).
- **MemPalace `add_drawer` creates a wing that does not exist yet**: succeeds per spike §3.7 (wings are created on first write). No explicit bootstrap wing-create step is required. Clarify item F tracks confirming this invariant against MemPalace 3.3.2.
- **Concurrent MemPalace writes from two agent subprocesses**: 2-writer concurrency characterized clean in spike §3.8. M2.2's default engineering concurrency cap is 1, so this case does not arise in the acceptance flow; if the operator bumps the cap, behaviour is inherited from MemPalace's SQLite + Chroma sequencing. Chaos tests do not cover higher concurrency in M2.2.
- **Agent writes a diary entry to the wrong wing**: the hygiene checker searches the agent's `palace_wing` (from the `agents` row) and will not find the drawer; `hygiene_status='missing_diary'`. The write still succeeded in the palace; future hygiene UI (M3/M6) can surface orphaned drawers separately.
- **Agent writes a KG triple but no diary drawer**: `hygiene_status='missing_diary'` (diary is the primary signal; KG triple without diary is insufficient per "Hygiene checker" in `m2.2-context.md`).
- **Agent writes a diary drawer but no KG triples**: `hygiene_status='missing_kg'`.
- **Agent writes both, diary body length < 100 chars**: `hygiene_status='thin'`.
- **MemPalace MCP call inside the agent run takes longer than the agent's remaining budget**: Claude emits a terminal `result` event with `terminal_reason='budget_exceeded'` (or equivalent — see clarify item on Claude Code's budget semantics in M2.1's spec). The supervisor records `exit_reason='budget_exceeded'` on the `agent_instances` row; no `ticket_transitions` row is written.
- **Hygiene checker receives a notification for a ticket_transition whose `triggered_by_agent_instance_id` is NULL** (test fixture, manual insert): the checker cannot determine the run window. Default behaviour: skip the evaluation (`hygiene_status` remains NULL). Such rows do not appear on the hygiene dashboard.
- **Hygiene checker's dedicated `pgx.Conn` disconnects**: the checker reconnects with exponential backoff (100 ms → 30 s cap) and resumes LISTEN. Missed notifications are re-discovered by the periodic sweep on `ticket_transitions` rows with `hygiene_status IS NULL` older than the delay window. Same pattern as M1's dispatcher.
- **Hygiene checker is processing a row when the supervisor receives SIGTERM**: the checker's in-flight UPDATE completes via `context.WithoutCancel(ctx)` + `TerminalWriteGrace`, same as M2.1 terminal writes (M1 retro §2, AGENTS.md concurrency rule 6). No row is left in a partial state.
- **Structured log of `mempalace_*` tool_use event when the tool_result is not yet observed**: the initial `tool_use` event is logged at `info` with `outcome='pending'`. The subsequent `tool_result` is logged as an update with the resolved outcome. The supervisor does not block on tool_result arrival; if the subprocess exits before a tool_result arrives (crash case), the outcome remains `'pending'` in the last log line — observational only; no dispatch consequence.
- **`mempalace` binary version mismatch between install and palace state**: out of scope for M2.2. If MemPalace introduces a migrations path, the operator runs `mempalace migrate` per ops checklist guidance.
- **Two M2.2 migrations applied in the wrong order**: prevented by `goose` monotonic ordering (M1 pattern, unchanged).
- **`GARRISON_HYGIENE_DELAY` set to 0 or negative**: supervisor validates at startup; values ≤ 0 are rejected and startup fails.
- **`GARRISON_HYGIENE_SWEEP_INTERVAL` set to 0 or negative**: same validation; startup fails.

## Requirements (mandatory)

### Functional requirements

M2.2 inherits every functional requirement from the M1 and M2.1 specs. The requirements below are additions and refinements specific to M2.2.

#### Palace bootstrap and configuration

- **FR-201**: The supervisor MUST bootstrap the MemPalace palace at the path given by `GARRISON_PALACE_PATH` at startup by unconditionally running `mempalace init --yes <palace_path>` (per clarification 2026-04-23 finding F1 — `mempalace init --yes` is idempotent on 3.3.2; a second invocation against an initialized palace is a no-op that re-writes the identical `mempalace.yaml`). The supervisor MUST log at `info` level with structured field `palace_initialized=true` on success. Any non-zero exit, timeout, or unexpected error — including, but not limited to, permission errors or the unexpected-interactive-prompt case from spike §3.2 — MUST cause the supervisor to exit startup with a non-zero code and a log line identifying the path and the underlying error. The marker-file check and the "already_initialized" vs "initialized" state distinction from Session 2026-04-22 Q4 are both superseded: there is no marker file in 3.3.2 that faithfully distinguishes "palace initialized and never written to" from "palace uninitialized" (`chroma.sqlite3` appears only after the first MCP write); the correct discipline is therefore to lean on MemPalace's own idempotency, not on a supervisor-side marker heuristic.
- **FR-202**: The configured palace path MUST NOT be a git-tracked directory. The supervisor does not enforce this at code level (operator-owned), but documentation MUST warn against it per AGENTS.md "what agents should not do" and spike §3.2. The default `~/.garrison/palace/` is outside any Garrison checkout.
- **FR-203**: The supervisor MUST read `MempalaceBin` (default `mempalace`), `PalacePath` (default resolved from `GARRISON_PALACE_PATH` or `os.UserHomeDir()`), and `AgentMempalacePassword` (from `GARRISON_AGENT_MEMPALACE_PASSWORD`) from environment via the existing `internal/config` package at startup, and MUST compose `AgentMempalaceDSN()` identically to M2.1's `AgentRODSN()` pattern (see "Database roles" in `m2.2-context.md`). **Note (superseded by Session 2026-04-23 deployment-shape clarification):** under the approved 2b + socket-proxy topology, MemPalace runs in a sidecar container (`garrison-mempalace`) and is invoked through filtered `docker exec`; the supervisor's local-binary references in this requirement (`MempalaceBin`, `~/.garrison/palace/`, in-supervisor MemPalace install) are superseded by `DockerBin` + `MempalaceContainer` + a container-internal `PalacePath` (default `/palace`) plus a dedicated `Dockerfile.mempalace`. The *behavioural* requirement is unchanged; only the implementation mechanism shifts. See Clarifications → Session 2026-04-23.

#### MCP config extension

- **FR-204**: The per-invocation MCP config file written by `internal/mcpconfig` MUST contain both a `postgres` entry (unchanged from M2.1) and a `mempalace` entry, per the JSON shape in "MCP config file extensions" of `m2.2-context.md`. The `mempalace` entry's `command` MUST be the `mempalace` binary (resolved via `MempalaceBin`, default `mempalace` on `$PATH`), its `args` MUST invoke the `mcp` subcommand (per clarification 2026-04-22 Q1 — the binary subcommand confirmed by spike §3.1), and its `env` MUST set `MEMPALACE_PATH` to the configured palace path. The exact `args` form (e.g. `["mcp"]` vs. `["mcp", "--palace", "<path>"]`) is resolved against MemPalace 3.3.2's `mempalace mcp --help` at implementation time; the `python -m mempalace.mcp_server` fallback from spike §3.6 is NOT used in production. **Note (superseded by Session 2026-04-23 deployment-shape clarification):** under the approved 2b + socket-proxy topology, MemPalace runs in a sidecar container (`garrison-mempalace`) and is invoked through filtered `docker exec`; the supervisor's local-binary references in this requirement (`MempalaceBin`, `~/.garrison/palace/`, in-supervisor MemPalace install) are superseded by `DockerBin` + `MempalaceContainer` + a container-internal `PalacePath` (default `/palace`) plus a dedicated `Dockerfile.mempalace`. Specifically, the `mempalace` entry's `command` is `docker`, its `args` are `["exec", "-i", "<container>", "python", "-m", "mempalace.mcp_server", "--palace", "<path>"]` (the `python -m …` form this body says is NOT used in production IS in fact the production form under 2b — Session 2026-04-23 Q1 supersedes Session 2026-04-22 Q1), and its `env` carries `DOCKER_HOST` instead of `MEMPALACE_PATH` (the latter is passed via the `--palace` flag). The *behavioural* requirement is unchanged; only the implementation mechanism shifts. See Clarifications → Session 2026-04-23.
- **FR-205**: The `mcpServers` JSON object in the per-invocation config MUST remain additive: M2.2 adds one key (`mempalace`) without replacing or reshaping the existing `postgres` key, per the M2.1 retro's observation that the shape was already multi-server-ready.
- **FR-206**: `internal/mcpconfig.Write` MUST accept a parameter for the palace path so that the emitted JSON reflects the configured `GARRISON_PALACE_PATH`. The supervisor MUST NOT hard-code any palace path into the binary.

#### Wake-up context injection

- **FR-207**: Before `cmd.Start()`, the supervisor MUST execute `mempalace --palace <palace_path> wake-up --wing <palace_wing>` (palace_wing read from the spawned agent's `agents.palace_wing` column; corrected flag shape per clarification 2026-04-23 finding F2 — 3.3.2's `wake-up` subcommand accepts only `--wing`, `--palace` is a top-level flag, and `--max-tokens` does NOT exist) with a 2-second timeout. The supervisor MUST capture stdout verbatim; wake-up output size is not operator-controlled (bounded internally by MemPalace's L0+L1 renderer; ~79–170 tokens observed empirically). Stderr MAY be captured at debug level. **Note (deployment shape per Session 2026-04-23):** under the sidecar + socket-proxy topology, the command is invoked through `docker exec garrison-mempalace mempalace --palace /palace wake-up --wing <palace_wing>`; the 2-second timeout is enforced by the supervisor's `context.WithTimeout` wrapping the `docker exec` call. The *behavioural* requirement is unchanged.
- **FR-207a**: On success (exit 0 within timeout), the supervisor MUST compose the `--system-prompt` argument as:

  ```
  <agent.md content>

  ---

  ## Wake-up context

  <wake-up stdout>

  ---

  ## This turn

  You have been spawned as agent_instance <instance_id> to work ticket <ticket_id>. Read it, then execute your completion protocol.
  ```

  per "Wake-up injection contract" in `m2.2-context.md`. On the resulting `agent_instances` row, `wake_up_status` MUST be `'ok'`.

- **FR-207b**: On failure (non-zero exit, timeout, exec error), the supervisor MUST compose `--system-prompt` WITHOUT the wake-up block (agent.md content directly followed by the "This turn" block). It MUST log at `warn` level with structured fields `palace_wing`, `elapsed_ms`, `error`. It MUST proceed with the spawn. On the resulting `agent_instances` row, `wake_up_status` MUST be `'failed'`.
- **FR-207c**: Stale wake-up output (command completes after the timeout) MUST be discarded. The supervisor MUST NOT wait for or incorporate late-arriving stdout.

#### Init-event parsing extension

- **FR-208**: M2.1's FR-108 (init-event `mcp_servers[]` health check) is extended: the supervisor MUST verify BOTH `postgres` and `mempalace` entries have `status == "connected"`. Any other value on either entry — including `failed`, `needs-auth`, or any unknown string — MUST cause immediate process-group SIGTERM per M2.1's termination contract. The `exit_reason` MUST be of the form `mcp_<name>_<status>` where `<name>` identifies which server was unhealthy (e.g. `mcp_mempalace_failed`). If both are unhealthy, the supervisor records the first observed failure in the `mcp_servers[]` order; subsequent failures may be logged but the `exit_reason` is the first.

#### Agent-instance state extension

- **FR-209**: The M2.2 migration MUST add a nullable `wake_up_status TEXT` column to `agent_instances`. Permitted values: `'ok'`, `'failed'`, `'skipped'` (the latter for future use, e.g. roles configured to skip wake-up; M2.2 never writes `'skipped'`). The M1/M2.1 test suite MUST continue to pass; existing tests that insert `agent_instances` rows without specifying `wake_up_status` are still valid (NULL default).
- **FR-210**: The `exit_reason` vocabulary from M2.1 is extended with:
  - `mcp_mempalace_<status>` — MemPalace MCP unhealthy at init (FR-208);
  - `budget_exceeded` — terminal result reports per-invocation budget exceeded (M2.2 adopts this canonical value; M2.1 had no equivalent path).

  All M2.1 exit reasons remain valid unchanged.

#### Hygiene checker

- **FR-211**: The supervisor MUST run a hygiene checker as a single goroutine inside its process (not as a subprocess). The checker MUST LISTEN on `work.ticket.transitioned.*.*` via its own dedicated `pgx.Conn` (not from the pool; same discipline as the M1 dispatcher).
- **FR-212**: For each received notification, the checker MUST wait `GARRISON_HYGIENE_DELAY` (default 5 seconds, tunable) before evaluating, to give in-flight palace writes time to land.
- **FR-213**: After the delay, the checker MUST query MemPalace against the agent_instance's run window (from `agent_instances.started_at` to `agent_instances.finished_at`) for:
  - (a) drawers in the agent's `palace_wing` whose body mentions the ticket id;
  - (b) KG triples returned by `mempalace_kg_query(entity='ticket_<id>')` where either the subject or object is `ticket_<id>` (per clarification 2026-04-22 Q3). The query is direction-agnostic; presence of at least one returned triple whose `valid_from` falls within the agent_instance's run window satisfies the KG-presence check.
- **FR-214**: The checker MUST evaluate `hygiene_status` per "Hygiene checker (binding shape)" in `m2.2-context.md`:
  - Diary present, body ≥ 100 chars, at least one matching KG triple → `'clean'`;
  - Diary present, body < 100 chars → `'thin'` (overrides `'missing_kg'` evaluation — thin diary is flagged regardless of KG state);
  - No matching diary → `'missing_diary'`;
  - Diary present (≥100 chars) but no matching KG triple → `'missing_kg'`;
  - Palace query error or timeout → `'pending'`.
- **FR-215**: The checker MUST `UPDATE ticket_transitions SET hygiene_status = $1 WHERE id = $2` and MUST NOT overwrite a row whose `hygiene_status` is already set to a non-`'pending'` terminal value. At-most-once-to-terminal per row.
- **FR-216**: The supervisor MUST run a periodic sweep goroutine at cadence `GARRISON_HYGIENE_SWEEP_INTERVAL` (default 60 seconds, tunable) that SELECTs `ticket_transitions` rows with `hygiene_status IN ('pending', NULL)` older than `GARRISON_HYGIENE_DELAY` and re-evaluates them via the same code path as the LISTEN handler. The sweep is the recovery mechanism for missed notifications (inherited from M1's processed_at pattern).
- **FR-217**: The checker goroutine MUST respect supervisor shutdown: on root context cancellation, it finishes its in-flight evaluation via `context.WithoutCancel(ctx)` + `TerminalWriteGrace` and exits. No row is left in a partial state.

#### Structured logging of MemPalace tool calls

- **FR-218**: For every `tool_use` event in the stream-json stream whose `name` starts with `mempalace_`, the supervisor MUST emit an `info`-level slog line with structured fields: `ticket_id`, `agent_instance_id`, `tool_name`, `tool_use_id`, and `outcome='pending'`. For the corresponding `tool_result` event, the supervisor MUST emit a follow-up `info`-level line with `tool_use_id` and `outcome` resolved to one of `'ok'`, `'error'` (with the error detail). These log lines are the data stream M3's hygiene dashboard will consume.
- **FR-218a**: MemPalace tool-call logs are observational; they MUST NOT change dispatch decisions (no bailing, no cost reconciliation). The supervisor's terminal adjudication remains driven by the `result` event and the M2.1 adjudication table, extended only by FR-208 and FR-210.

#### Budget enforcement

- **FR-219**: The supervisor MUST set `--max-budget-usd 0.10` on every M2.2 Claude Code invocation per "Binding question 8" of `m2.2-context.md`. This is an increase from M2.1's $0.05 default (`--max-budget-usd` was `0.05` per M2.1 NFR-103). The value is a single supervisor-wide default for M2.2; per-agent overrides remain deferred (M6).
- **FR-220**: A terminal `result` event indicating the budget was exceeded (per Claude Code's terminal reason vocabulary; exact string depends on Claude Code's characterization) MUST produce `exit_reason='budget_exceeded'` and `status='failed'` on the `agent_instances` row. No `ticket_transitions` row is written. `total_cost_usd` is captured from the result event.

The QA seed `agent.md` MUST NOT contain `UPDATE tickets` or `INSERT INTO ticket_transitions` statements (clarification 2026-04-23 Q1).

### Non-functional requirements

- **NFR-201** (per-invocation budget cap): $0.10 USD per Claude Code invocation, applied via `--max-budget-usd 0.10`. This supersedes M2.1's NFR-103 ($0.05) for the M2.2 release onwards. Justification in "Real-world cost baseline (from M2.1)" of `m2.2-context.md`.
- **NFR-202** (wake-up timeout): 2 seconds hard ceiling for the `mempalace wake-up` subprocess. The supervisor uses `exec.CommandContext` with a 2-second-derived context; on timeout, the context cancellation kills the wake-up subprocess (SIGTERM → SIGKILL) and the supervisor proceeds without wake-up context.
- **NFR-203** (hygiene checker delay default): 5 seconds between transition notification and palace query. Tunable via `GARRISON_HYGIENE_DELAY`. Values ≤ 0 are rejected at supervisor startup.
- **NFR-204** (hygiene sweep cadence default): 60 seconds between periodic sweep runs. Tunable via `GARRISON_HYGIENE_SWEEP_INTERVAL`. Values ≤ 0 are rejected at supervisor startup.
- **NFR-205** (hygiene checker concurrency): single goroutine, sequential processing of notifications (per "Binding question 6" of `m2.2-context.md`). Concurrent evaluation is rejected.
- **NFR-206** (MemPalace MCP health bail latency): extends M2.1's NFR-106 — the supervisor MUST send SIGTERM to the Claude process group within 2 seconds of receiving the `system`/`init` event when that event reports a non-`connected` status for EITHER the `postgres` or `mempalace` server. The combined check MUST complete within the same 2-second budget.
- **NFR-207** (MemPalace version pin): MemPalace 3.3.2 is the spec-committed version, validated by spike §3 (per clarification 2026-04-22 Q2). No "or newer" hedge: the plan and implementation MUST install exactly 3.3.2. Install MUST be verified via SHA256 on the published wheel (same discipline as M2.1's Claude Code install). Upgrading to a later version is a future-milestone decision that re-runs the spike's behaviour checks.
- **NFR-208** (palace path default): `~/.garrison/palace/`, resolved at startup via `os.UserHomeDir()`. The path MUST be outside any git-tracked directory (operator-enforced; not code-enforced per spike §3.2). **Note (superseded by Session 2026-04-23 deployment-shape clarification):** under the 2b + socket-proxy topology the palace lives inside the `garrison-mempalace` sidecar container at default path `/palace` (a Docker-named volume), not on the supervisor host's filesystem. `GARRISON_PALACE_PATH` remains the override env var; its value is a container-internal path passed via `docker exec ... --palace <path>`. By topology (volume mount), the path is outside any git-tracked directory — the `outside any git checkout` invariant is preserved by construction rather than by operator discipline. The *behavioural* requirement is unchanged; only the implementation mechanism shifts. See Clarifications → Session 2026-04-23.
- ~~**NFR-209**~~ **deleted by clarification 2026-04-23 finding F2.** `mempalace wake-up` in 3.3.2 does not accept `--max-tokens`; wake-up output size is implicitly bounded by MemPalace's L0+L1 renderer. The supervisor accepts stdout verbatim (~79–170 tokens observed empirically). No operator-tunable token budget exists for this command.
- **NFR-210** (structured log fidelity for MemPalace tool calls): every `mempalace_*` `tool_use` and `tool_result` event MUST be logged with the fields enumerated in FR-218. No sampling; no redaction (there is no secret content in MemPalace payloads at M2.2).
- **NFR-211** (single supervisor instance per database): inherited from M1 FR-018 advisory lock, unchanged by M2.2. The hygiene checker does NOT require a separate advisory lock — the `at-most-once-to-terminal` discipline (FR-215) provides the correctness property.
- **NFR-212** (cost data fidelity): inherited from M2.1 NFR-108 unchanged. `total_cost_usd` is the exact value Claude reports; the supervisor does not round, accumulate, or synthesize.
- **NFR-213** (Dockerfile install discipline): MemPalace 3.3.2 is installed via a dedicated `Dockerfile.mempalace` (base `python:3.11-slim` per finding F3 — alpine fails because chromadb's Rust-compiled transitives have no musl wheels; debian-glibc installs from prebuilt wheels directly) that produces the `garrison-mempalace` sidecar image. The supervisor image itself gains only `apk add --no-cache docker-cli` in the runtime stage. Hash-pinning discipline is implemented via a committed `supervisor/requirements-lock.txt` generated by `pip-compile` (per finding F4 — pip's `--require-hashes` requires all transitive deps pinned; the lockfile approach achieves this). Installation: `pip install --no-cache-dir --require-hashes -r requirements-lock.txt`. `procps` is additionally installed in the sidecar image per finding F6 so `docker exec garrison-mempalace ps aux` works for human operators. Final sidecar image size observed in T001: 478 MB; the M2.1 supervisor baseline of 264 MB is the separate reference for the supervisor image, unchanged. Both numbers recorded in the retro.
- **NFR-214** (log-stream labelling for hygiene checker): the hygiene checker's slog lines MUST carry structured fields `ticket_id`, `agent_instance_id`, `ticket_transition_id`, and `hygiene_status` (for terminal evaluations) or `error` (for query failures).

### Key entities

Deltas from M2.1. Full schema lives in the migration referenced by FR-221a; do not duplicate it here.

## Assumptions

- MemPalace 3.3.2 is installed on the supervisor host such that the `mempalace` binary is discoverable on `$PATH` by the supervisor process. The Docker image installs it explicitly (FR-213 / NFR-213); for local development, the operator owns the install.
- The palace path (default `~/.garrison/palace/`) is writable by the supervisor's user and is outside any git-tracked directory. If the operator overrides `GARRISON_PALACE_PATH` to a git-tracked path, spike §3.2 warns of `.gitignore` mutation — the supervisor does not prevent this programmatically.
- Palace bootstrap runs `mempalace init --yes <palace_path>` unconditionally at supervisor startup per clarification 2026-04-23 F1; MemPalace 3.3.2's own idempotency (observed in T001) is the correctness guarantee.
- The `mempalace --palace <path> wake-up --wing <wing>` command is available on MemPalace 3.3.2 (T001 verified flags; `--max-tokens` does not exist — clarification 2026-04-23 F2). T001 measured wake-up p95 = 1031 ms through docker exec, comfortably inside the 2-second NFR-202 ceiling.
- Wings are created lazily by `mempalace_add_drawer` per spike §3.7. M2.2 does not bootstrap `wing_frontend_engineer` or `wing_qa_engineer` explicitly — the first agent write to each wing materializes it. Clarify item F tracks confirming this invariant.
- The MemPalace MCP server does not require authentication beyond the `MEMPALACE_PATH` environment variable pointing at the palace directory (flagged for clarify as item C). If an API key or token is required, it lands in the MCP config file's `env` block, parallel to M2.1's `GARRISON_AGENT_RO_DSN` handling.
- The MemPalace MCP server's stdio invocation is `mempalace <subcommand>` or `python -m mempalace.mcp_server --palace <path>` per spike §3.6; the exact pinned command is flagged for clarify as item A and MUST be decided before implementation.
- Concurrent palace writes from the two agents (engineer + qa-engineer) do not overlap in practice — the qa-engineer only spawns after the engineer's transition writes, and the default engineering concurrency cap is 1. Spike §3.8 validated 2-writer concurrency as clean even if they did overlap.
- Claude Code's `--max-budget-usd 0.10` enforces per-invocation, not per-session (the spec assumes this based on M2.1's operational experience; definitive characterization remains deferred to M6 per M2.1 open questions).
- The engineer's `agent.md` (from `migrations/seed/engineer.md`) is large enough to meaningfully exercise the palace write contract but small enough that the $0.10 cap holds for the representative flow. Empirical validation comes from SC-202 / SC-212.
- The QA engineer, given the engineer's diary + KG triples in MemPalace plus the ticket in Postgres, has enough context to verify the engineer's output (for M2.2's trivial "read changes/hello-<ticket_id>.md and confirm it exists and reads sensibly" acceptance). More stringent verification is out of scope; the QA role's sophistication is extension-only in future milestones.
- The supervisor runs on Linux (inherited from M1 / M2.1; `SysProcAttr.Setpgid` + `syscall.Kill(-pgid, ...)` are Linux-specific and covered by M2.1's `internal/spawn/pgroup` tests).
- Operators who run the M2.2 migration follow the ops checklist to `ALTER ROLE garrison_agent_mempalace PASSWORD '...'` post-migrate and set `GARRISON_AGENT_MEMPALACE_PASSWORD` in the supervisor's env. Without this, the hygiene checker's connection fails; the supervisor logs loudly on startup and declines to start. The ops checklist update is part of FR-222.
- The hygiene checker's dedicated `pgx.Conn` does not count against the supervisor's pool budget. M1's dispatcher connection is also dedicated; M2.2 adds a second dedicated connection. Resource impact is negligible (two long-lived connections vs. an N-sized pool).
- Agents writing to MemPalace trust that MCP tool calls succeed (or fail explicitly via `tool_result` with `is_error=true`). The supervisor does not second-guess tool-result errors at the dispatch layer; it logs them (FR-218) and lets the agent's own logic react. If an agent's write fails and it transitions the ticket anyway, hygiene status correctly flags `'missing_diary'` / `'missing_kg'` / `'thin'` at review time.
- The existing `internal/claudeproto` stream-json router from M2.1 requires no changes to recognize `mempalace_*` `tool_use` events — `tool_use` is already a routed type, and the `name` field is already exposed for the logger. If M2.2 discovers a routing gap, it is a plan-phase finding, not a spec ambiguity.
- The M2.2 migration does NOT rewrite existing M2.1 schema and does NOT introduce any new tables. It only ADDs: `wake_up_status` column on `agent_instances`, `garrison_agent_mempalace` role + SELECT grants on existing M2.1 tables, qa-engineer seed row in `agents`, and updates to existing seed rows' `agent_md` + `palace_wing` + the engineering department's `workflow` JSONB (per clarification 2026-04-22 Q5). If the plan phase discovers a schema gap requiring a new table or a rewrite of an M2.1 column, the spec is re-examined before the migration is written.
