## Ordering principle

Strict dependency order, same discipline as M1 and M2.1. M2.2 extends the M2.1 supervisor and introduces one new deployment container (`garrison-mempalace`) plus one filtered-socket proxy (`garrison-docker-proxy`); it does not rebuild the supervisor. After every task the repository is in a working state — the supervisor binary compiles, all existing M1 and M2.1 tests still pass, and the newly-added behaviour is independently verifiable by the completion condition of that task.

Acceptance criteria (SC-201–SC-214) refer to [spec.md](./spec.md) §"Success criteria (mandatory)" and equivalently to the 13 acceptance criteria in [../_context/m2.2-context.md](../_context/m2.2-context.md) §"Acceptance criteria for M2.2". Spec functional-requirement numbers (FR-2xx) and non-functional numbers (NFR-2xx) refer to [spec.md](./spec.md) §"Requirements". Every clarification entry under [spec.md](./spec.md) §Clarifications supersedes equivalently-scoped wording in the FR bodies.

Each task carries:

- **Depends on** — prior task IDs (or "M2.1 shipped" for tasks building directly on M2.1 code with no M2.2 predecessor)
- **Files** — exact paths the task creates or modifies
- **Completion condition** — the concrete passing check after the task
- **Out of scope for this task** — what to resist reaching ahead for
