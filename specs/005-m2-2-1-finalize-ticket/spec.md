# Feature specification: M2.2.1 — Structured completion via `finalize_ticket` tool

**Feature branch**: `005-m2-2-1-finalize-ticket`
**Created**: 2026-04-23
**Status**: Draft
**Input**: `/garrison-specify m2-2-1` — "Fix the M2.2 live-run compliance failure (agent skipped mandatory MemPalace writes) by making structured completion the only exit path."

This spec is bound by `specs/_context/m2-2-1-context.md`. Every decision settled there — the package name (`internal/finalize`), the subcommand shape (`supervisor mcp finalize`), the MCP config order (`postgres, mempalace, finalize`), the schema version tag (`garrison.finalize_ticket.v1`), the 3-attempt retry cap, the strict schema (no tiering), the supervisor-side atomic write boundary, the redefined `hygiene_status` vocabulary, the seed agent.md rewrite posture, the non-retroactive legacy-row policy, and the scope boundary (supervisor-side only; no MemPalace or MCP-tool changes) — is input, not question. Where this spec cites a constraint, the authoritative text lives in that file; consult it rather than the paraphrase here.

Tool behaviour of MemPalace (tool inventory, response shapes, write semantics) is characterized in `docs/research/m2-spike.md` Part 2 and M2.2's pre-implementation spike at `specs/004-m2-2-mempalace/research.md`. Where this spec references specific behaviour, the citation is by section; the spec does not re-derive behaviour.

M2.2.1 builds on shipped M2.2 (see `specs/004-m2-2-mempalace/spec.md` and `docs/retros/m2-2.md`). The supervisor, event bus, stream-json parser, MCP-config writer, MemPalace sidecar topology, socket-proxy filter, hygiene checker goroutine, and all M2.1-era subprocess-lifecycle discipline remain as shipped. M2.2.1 is a narrow patch: it adds one in-tree MCP server, rewrites two seed agent.md files, moves the palace-diary/KG/transition writes from agent-driven (via `mempalace_*` tool calls) to supervisor-driven (triggered by a single agent-issued `finalize_ticket` tool call), redefines `hygiene_status` semantics, and nothing else.

The justification is empirical: M2.2's live-run append recorded that real Claude haiku-4-5-20251001 executed the ticket work but skipped `mempalace_add_drawer` and `mempalace_kg_add` despite the seed agent.md marking them MANDATORY. The memory thesis (RATIONALE §3) cannot rely on prompt compliance alone. M2.2.1 revises §3 to: "agents emit structured reflection via `finalize_ticket`; the supervisor writes to MemPalace from the payload; mandatory completion is supervisor-enforced." The revision is documented in `specs/_context/m2-2-1-context.md` §"Why this milestone exists" and will be folded into `RATIONALE.md` §3 at ship time.

## Clarifications

### Session 2026-04-23

- Q: Finalize-to-exit timing — does the supervisor act on a successful `finalize_ticket` tool_use event immediately (before subprocess exit), or wait for the `result` event / subprocess exit before committing palace + transition writes? → A: **Immediately on the first successful `finalize_ticket` tool_use event.** The supervisor's stream-json event parser, on observing a tool_use with `name == "finalize_ticket"` whose paired tool_result carries a schema-valid payload, triggers the atomic palace+transition write straight away; the agent's subsequent tokens / result event / exit are observed but do not gate the write. Rationale: the agent's job is to produce the structured payload; anything it does after a valid `finalize_ticket` call is redundant. Acting immediately minimizes the window in which a subsequent subprocess death could lose the completion.

- Q: Idempotency of repeated `finalize_ticket` calls within one agent_instance — if the first call succeeds, what happens on a second call? → A: **Reject the second call with an error.** Successful finalize is write-once per agent_instance. The finalize server returns `{error_type: "schema", field: null, message: "finalize_ticket already succeeded for this agent_instance"}` on any call after a previous success. The retry counter (decision 3 in context) applies only until the first schema-valid payload commits; thereafter the agent cannot re-finalize, cannot overwrite the committed diary/triples, and cannot retry the transition. Rationale: atomic commit semantics forbid overwriting; the supervisor already owns the transition and has committed it; allowing a second call would invite race conditions and ambiguity about which payload is canonical.

- Q: String-length and array-size ceilings for the `finalize_ticket` input schema — context gives minima (outcome ≥ 10 chars, rationale ≥ 50 chars, kg_triples ≥ 1 element) but no maxima. Do we cap? → A: **Yes, cap at `outcome ≤ 500` chars and `diary_entry.rationale ≤ 4000` chars.** `artifacts`, `blockers`, `discoveries` arrays each cap at 50 elements with each string element ≤ 500 chars. `kg_triples` array caps at 100 elements; each triple's `subject`, `predicate`, `object` cap at 500 chars each. Rationale: bound payload size to prevent runaway generations from consuming the per-invocation budget or ballooning the palace row; 500/4000 comfortably covers thoughtful reflection without inviting abuse.

- Q: How does the finalize MCP server bind to the agent_instance it's serving? → A: **Env var injection at MCP-config-write time.** The MCP-config writer sets `GARRISON_AGENT_INSTANCE_ID=<uuid>` in the finalize entry's `env` block alongside any other env the server needs (e.g. `GARRISON_DATABASE_URL` for the `finalize_committed` read per FR-260). The server reads it once from `os.Getenv` at startup and scopes all tool-call handling to that id for its lifetime. This matches the `internal/pgmcp` pattern (which reads `GARRISON_DATABASE_URL` from env), keeps the server stateless, survives claude's per-turn exec model (claude execs the MCP server with the env baked in by its MCP-launcher), and avoids a second Postgres round-trip just to resolve identity. Rationale: the supervisor knows the agent_instance_id at the moment it writes the MCP config (the id was INSERTed in the same `spawn.Spawn` dedupe transaction that triggered the config write) — injecting at that moment is zero-cost and makes the server purely I/O-bound on the call itself.

- Q: Maximum wall-clock duration for the supervisor's atomic write before it's considered failed? → A: **30 seconds.** The full atomic-write path (MemPalace `add_drawer` + up to 100× `kg_add` + Postgres inserts/updates + commit) is bracketed by a context with a 30s timeout. If MemPalace hangs (GC pause, ChromaDB rebuild, disk contention) or the Postgres commit stalls, the timeout fires, the Postgres tx rolls back, and the supervisor writes a terminal `agent_instances` row with `exit_reason='finalize_write_timeout'` (new canonical value alongside the palace_write_failed / commit_failed vocabulary in FR-266) and `hygiene_status='finalize_partial'`. Rationale: 30s is `TerminalWriteGrace × 3` (M2.1's terminal grace is 10s), generous relative to the ~1–3s MemPalace calls observed in M2.2's live run, and bounded enough that a hung sidecar surfaces within the same minute rather than holding an `agent_instances` row lock indefinitely. Tighter (10s) would false-positive under MemPalace GC pauses; looser (60s+) extends the hang-detection window without operational benefit.

- Q: How does the qa-engineer actually find the engineer's diary in MemPalace, given the M2.2 live-run finding that `mempalace_search` with a raw UUID returns zero matches (search is vector-similarity over embeddings, not substring)? → A: **Supervisor prepends the ticket's objective text (plain prose from `tickets.objective`) to the drawer body ahead of the YAML frontmatter.** The atomic-write diary serialization produces a body shaped as `"<ticket_objective>\n\n<yaml_frontmatter>\n\n<rationale>"` — the prose `<ticket_objective>` gives semantic search meaningful content to match on, the `<yaml_frontmatter>` retains the ticket_id for precise post-retrieval filtering, the `<rationale>` carries the agent's reflection. Qa-engineer's protocol is unchanged at the agent-protocol level: it issues `mempalace_search(wing='wing_frontend_engineer', query=<ticket_objective>)` as part of its wake-up / pre-work scan. The ticket_objective is available to the qa-engineer via its postgres MCP `SELECT objective FROM tickets WHERE id=<ticket_id>` — the same mechanism it uses to read the ticket row at turn start. Rationale: keeps the agent-facing protocol simple, validates the memory thesis (agent B genuinely reads agent A's diary), and sidesteps M2.2's known search-over-UUID gap without requiring a MemPalace change (which is scope-excluded per context §"What M2.2.1 does not produce"). The M3 dashboard work can still reshape the query path later; M2.2.1 lands a working shape today.

## User scenarios and testing (mandatory)

### User story 1 — engineer completes a ticket via `finalize_ticket` and the supervisor writes palace + transition atomically (priority: P1)

The operator inserts an engineering ticket at column `in_dev`. The supervisor spawns the engineer Claude Code subprocess with three MCP servers (`postgres`, `mempalace`, `finalize`) wired into the per-invocation config. The engineer executes the ticket work (reads context, may use `mempalace_search` / `mempalace_add_drawer` / `mempalace_kg_add` for mid-turn notes in `hall_discoveries`, writes the deliverable to its workspace, etc.), then issues exactly one `finalize_ticket` tool call whose payload is schema-valid. The finalize MCP server validates the payload synchronously and returns success. The supervisor's stream-json parser observes the successful tool_use, triggers the atomic write (diary to `wing_frontend_engineer` via the existing MemPalace sidecar, KG triples via the same path, `ticket_transitions` insert + `tickets.column_slug` update + `agent_instances` terminal + `event_outbox.processed_at` all in one Postgres transaction), records `total_cost_usd`, and sets `hygiene_status='clean'` on the transition row. The agent's post-finalize tokens (if any) are observed but not acted on.

**Why this priority**: this is the smallest slice that demonstrates the whole compliance revision end-to-end — agent produces structured reflection, supervisor commits it atomically, hygiene_status reflects the outcome without any MemPalace queries. If this story ships, the architectural pivot is validated. It is the spec's load-bearing user story; everything else is variation on this core flow.

**Independent test**: against a freshly-migrated Postgres with the M2.2.1 seed agent.md applied, a running MemPalace sidecar, the `claude` binary on `$PATH`, and the supervisor running, insert one ticket at column `in_dev` for `engineering`. Observe: one `agent_instances` row with `status='succeeded'`, `exit_reason=NULL` or `'completed'`, populated `total_cost_usd ≤ $0.10`, `wake_up_status='ok'`; one `ticket_transitions` row (`in_dev → qa_review`) with `hygiene_status='clean'`; one diary drawer in `wing_frontend_engineer` whose body mentions the ticket id; at least one KG triple in `wing_frontend_engineer` mentioning the ticket id; structured slog line at `info` level for the `finalize_ticket` tool_use event.

**Acceptance scenarios**:

1. **Given** a fresh M2.2.1-migrated database, running MemPalace sidecar, and the supervisor configured per `m2-2-1-context.md` §"What M2.2.1 produces", **When** the supervisor starts, **Then** its MCP config writer emits a three-entry `mcpServers` block (order: `postgres`, `mempalace`, `finalize`) for every subsequent spawn; `finalize` entry's `cmd` is the supervisor binary path and `args` are `["mcp", "finalize"]`, modelled on M2.2's `pgmcp` invocation shape.
2. **Given** a ticket at column `in_dev` for `engineering`, **When** the supervisor spawns the engineer invocation, **Then** the Claude subprocess's init event reports `mcp_servers[].status == "connected"` for all three entries, including `finalize`. Any one not connected causes the supervisor to signal the process group per M2.1's termination contract and record a failed `agent_instances` row with `exit_reason` of the form `mcp_<server>_<status>` (e.g. `mcp_finalize_failed`).
3. **Given** MCP health passed, the agent executes the completion protocol from the rewritten `migrations/seed/engineer.md` — read the ticket, perform the work, optionally call `mempalace_*` for mid-turn hall_discoveries, then call `finalize_ticket` with the full structured payload, **When** the `finalize_ticket` tool_use / tool_result pair lands on stream-json with a schema-valid payload, **Then** the finalize server returns a success tool_result (`{ok: true, attempt: 1}`) and the supervisor's stream parser immediately (before the agent's `result` event is seen) triggers the atomic write.
4. **Given** the supervisor has observed a successful finalize call, **When** the atomic write executes, **Then** in a single Postgres transaction: (a) the supervisor invokes the MemPalace sidecar's `mempalace_add_drawer` with the engineer's wing, the diary content serialized from the `diary_entry` sub-object (the existing YAML-frontmatter-in-body shape from ARCHITECTURE.md "Diary entry schema"), (b) invokes `mempalace_kg_add` once per triple in `kg_triples`, (c) inserts the `ticket_transitions` row for `in_dev → qa_review` with `hygiene_status='clean'`, (d) updates `tickets.column_slug='qa_review'`, (e) writes the terminal `agent_instances` row with `status='succeeded'`, `total_cost_usd` populated, `wake_up_status='ok'`, and (f) sets `event_outbox.processed_at` on the inbound event. The transaction commits atomically; any mid-transaction failure rolls back per User Story 4's failure paths.
5. **Given** the atomic write has committed, **When** the agent emits any further tokens, MCP calls, or a final `result` event, **Then** the supervisor observes them but takes no further database action for this ticket; the terminal row is already committed. A second `finalize_ticket` call from the same subprocess returns `{error_type: "schema", field: null, message: "finalize_ticket already succeeded for this agent_instance"}` per Clarification 2026-04-23 Q2.
6. **Given** the recorded `total_cost_usd` for the invocation, **When** compared to M2.2's NFR-201 per-invocation cap, **Then** the value is strictly less than $0.10 for the representative "engineer does one `finalize_ticket`-enabled ticket" workflow. Retry attempts consume budget per decision 3 in context; exhausting budget during retries terminates via M2.2's existing `budget_exceeded` path (Adjudicate precedence unchanged).

### User story 2 — QA engineer reads the engineer's palace writes and completes the handoff via a second `finalize_ticket` call (priority: P1)

The engineer's successful transition fires `pg_notify('work.ticket.transitioned.engineering.in_dev.qa_review', ...)`. The supervisor spawns the qa-engineer invocation with the same three MCP servers. The QA agent receives a wake-up from `wing_qa_engineer` (via the existing M2.2 wake-up injection path; M2.2.1 does not change wake-up semantics), reads the ticket (including `tickets.objective`) via the postgres MCP, searches MemPalace for the engineer's diary entry with `mempalace_search(wing='wing_frontend_engineer', query=<ticket.objective>)` — the drawer body's prose-prepended-objective (FR-263) is what makes semantic search land — verifies the engineer's deliverable, then issues its own `finalize_ticket` call with a schema-valid payload for the `qa_review → done` transition. The supervisor commits the second atomic write. Both agent_instances end with `hygiene_status='clean'`; combined cost is under $0.20.

**Why this priority**: the memory thesis (RATIONALE §3, as revised) is that agent B reads what agent A wrote. M2.2 surfaced this as untested because agent A didn't reliably write. M2.2.1's compliance architecture makes the write near-certain (US1) — this story is the proof that the downstream read still works given guaranteed writes. Equal in priority to US1; both are ship-blockers.

**Independent test**: pre-populate `wing_frontend_engineer` with a plausible diary entry whose body carries the ticket's objective as leading prose (mirroring what FR-263's atomic-write serializer produces in production) plus a handful of KG triples, seed a ticket at `qa_review` carrying that same objective. Observe: qa-engineer spawns, issues a postgres MCP read to fetch the ticket objective, issues `mempalace_search(wing='wing_frontend_engineer', query=<objective>)`, the tool_result returns the pre-populated drawer (the objective-prose matching is load-bearing here), the QA agent issues `finalize_ticket`, the supervisor commits the second atomic write with wing `wing_qa_engineer`, and the ticket lands at `done`. Combined with US1, one ticket routed `in_dev → qa_review → done` produces two `agent_instances` with populated diaries in two wings, two `ticket_transitions` rows both with `hygiene_status='clean'`, and `SUM(total_cost_usd) < $0.20`.

**Acceptance scenarios**:

1. **Given** a `ticket_transitions` row committed by US1's atomic write, **When** the supervisor's dispatcher receives `work.ticket.transitioned.engineering.in_dev.qa_review`, **Then** it spawns the qa-engineer invocation with the same three-MCP-server config as US1 (the `finalize` entry is identical for both roles; the palace wing differs per agent row).
2. **Given** the qa-engineer has issued `mempalace_search(wing='wing_frontend_engineer', query=<ticket.objective>)` after reading the ticket objective from Postgres, **When** the tool_result lands, **Then** it contains at least one drawer whose body opens with the ticket's objective prose (per FR-263) and whose YAML frontmatter carries the matching ticket_id. Semantic search hits on the prose, not on the UUID — the objective-prose prepend in FR-263 is what makes this work. This is the first point at which the memory thesis is behaviourally validated.
3. **Given** the qa-engineer verifies the deliverable and composes its own `finalize_ticket` payload for the `qa_review → done` transition, **When** the payload is schema-valid, **Then** the supervisor's second atomic write commits: diary to `wing_qa_engineer`, triples to `wing_qa_engineer`, `ticket_transitions` row for `qa_review → done` with `hygiene_status='clean'`, `tickets.column_slug='done'`, `agent_instances` terminal row.
4. **Given** both atomic writes have committed, **When** the operator queries `SUM(total_cost_usd)` over the two rows, **Then** the result is strictly less than $0.20. Either row above $0.10 is a retro finding, not a ship failure per-se.
5. **Given** the scenario is run twice — once with `claude-opus-4-7` and once with `claude-haiku-4-5-20251001` as both agents' model, **When** results are compared, **Then** both runs produce the same outcomes (both `hygiene_status='clean'`, both wings populated, combined cost each under $0.20). This is the headline compliance criterion (criterion 6 in context § "Acceptance criteria for M2.2.1").

### User story 3 — the retry-then-success path surfaces schema errors and lets the agent correct within the 3-attempt cap (priority: P2)

An agent issues a `finalize_ticket` call with a malformed payload (e.g. `kg_triples` empty, `diary_entry.rationale` too short). The finalize MCP server returns a structured error pinpointing the field and reason. The agent issues a corrected call. If the correction is still invalid on attempt 2, the server returns a second error. On attempt 3 (whether valid or invalid), the retry counter reaches the cap: a valid payload at attempt 3 commits normally; an invalid payload at attempt 3 causes the supervisor to terminate the subprocess with `exit_reason='finalize_invalid'` and set `hygiene_status='finalize_failed'`. No `ticket_transitions` row is written on the failure path.

**Why this priority**: correctness of the retry semantics is what makes strict-schema practical on weaker models. Without it, strict would be unacceptable risk; with it, strict is compliance-enforcing. Priority P2 because US1+US2 cover the happy path which is the common case; the retry path is less frequent but load-bearing for robustness.

**Independent test**: a mockclaude fixture (`finalize_retry_then_success.ndjson`) emits a `finalize_ticket` tool_use with an invalid payload, observes the tool_result's structured error, emits a second invalid payload (different field), observes the error, emits a valid payload on attempt 3. Observe: the `ticket_transitions` row commits with `hygiene_status='clean'`; the two failed attempts are logged at `info` level with the `error_type` and `field` fields; the `agent_instances` row ends `status='succeeded'`.

Separately, a second mockclaude fixture (`finalize_retry_exhausted.ndjson`) emits three invalid payloads in a row. Observe: no `ticket_transitions` row is written; `agent_instances.status='failed'`, `exit_reason='finalize_invalid'`; `ticket_transitions` row is not present (because no transition happened), so `hygiene_status` cannot be set there — instead the evaluator sets `hygiene_status='finalize_failed'` on the `agent_instances` row via the M2.2.1 schema (see FR-260).

**Acceptance scenarios**:

1. **Given** an agent has issued a `finalize_ticket` call with `kg_triples: []` (empty, violating decision 2's `min_length: 1`), **When** the finalize server validates the payload, **Then** it returns `{error_type: "schema", field: "kg_triples", message: "<human-readable reason>"}` and increments the supervisor-side retry counter for this agent_instance from 0 to 1.
2. **Given** the agent sees the error response and issues a corrected call (attempt 2) that is still invalid (e.g. `diary_entry.rationale` 30 chars, below the 50-char minimum), **When** the server validates, **Then** it returns `{error_type: "schema", field: "diary_entry.rationale", message: "<reason>"}` and increments the counter to 2.
3. **Given** the agent issues a third call (attempt 3) with a schema-valid payload, **When** the supervisor's stream parser observes the successful tool_use, **Then** the atomic write commits exactly as in US1 scenario 4 and the `ticket_transitions.hygiene_status='clean'`. The two failed attempts are preserved in structured logs only; no database row records them.
4. **Given** the agent exhausts all three attempts with invalid payloads, **When** the third error is returned, **Then** the supervisor signals the process group per M2.1's termination contract, writes the terminal `agent_instances` row with `status='failed'`, `exit_reason='finalize_invalid'`, `hygiene_status='finalize_failed'`, and does NOT insert a `ticket_transitions` row. The ticket remains at its entry column (`in_dev` for engineer, `qa_review` for qa-engineer).
5. **Given** the supervisor's pipeline Adjudicate precedence, **When** `exit_reason='finalize_invalid'` competes with `budget_exceeded` (e.g. retries consumed the $0.10 cap before the third attempt returned), **Then** `budget_exceeded` wins per M2.2's precedence table (unchanged by M2.2.1) and `hygiene_status='finalize_failed'` is still set (the payload-failure class encompasses all non-committing finalize outcomes from the hygiene-observer's point of view; `exit_reason` provides the root cause).

### User story 4 — atomic-write failure modes surface cleanly in `hygiene_status` without orphan state (priority: P2)

The finalize payload validated successfully but the supervisor's atomic write fails mid-transaction — either MemPalace's `mempalace_add_drawer` errors, MemPalace's `mempalace_kg_add` errors, or the Postgres commit fails after palace writes succeeded. Each has a distinct disposition:

- **Palace write fails before Postgres commit**: Postgres tx rolls back; no `ticket_transitions` row; no `agent_instances` success row; `hygiene_status='finalize_partial'` recorded on the `agent_instances` row; agent_instance ends `status='failed'` with `exit_reason='finalize_palace_write_failed'`. No palace orphan (MemPalace write didn't complete).
- **Palace write succeeds, Postgres commit fails**: palace has an orphan drawer/triples; supervisor logs a warning `palace_write_orphaned`; `agent_instances.hygiene_status='finalize_partial'`; agent_instance ends `status='failed'` with `exit_reason='finalize_commit_failed'`. MemPalace has no transactional rollback (spike §Part 2), so the orphan is known behaviour; the operator's dashboard surfaces it in M3 for manual cleanup.

**Why this priority**: P2 rather than P1 because these failures are rare (MemPalace sidecar crashes mid-transaction, Postgres bounces during commit). Observability matters more than frequency: when this does happen, the hygiene_status must surface it cleanly so the operator can triage without reading logs.

**Independent test**: chaos test `TestM221AtomicWriteChaosPalaceKillMidTransaction`. Run a mockclaude fixture that emits a valid `finalize_ticket` payload. During the supervisor's atomic write (specifically, after the first `mempalace_add_drawer` returns but before the Postgres commit), the test kills the MemPalace sidecar (`docker stop garrison-mempalace`). Observe: no `ticket_transitions` row; `agent_instances.hygiene_status='finalize_partial'`; `agent_instances.exit_reason` is either `finalize_palace_write_failed` (if the kill hit between triple-writes) or `finalize_commit_failed` (if the kill hit after all palace writes succeeded but before Postgres commit); no panic; supervisor continues running and handles subsequent events.

**Acceptance scenarios**:

1. **Given** the supervisor has begun the atomic write and the first `mempalace_add_drawer` call errors, **When** the error propagates, **Then** the Postgres transaction rolls back; no `ticket_transitions` row is written; the supervisor writes the terminal `agent_instances` row in a separate transaction with `status='failed'`, `exit_reason='finalize_palace_write_failed'`, `hygiene_status='finalize_partial'`, `total_cost_usd` populated from the stream-json `result` event.
2. **Given** all palace writes in the atomic transaction succeeded but the Postgres commit itself fails (e.g. connection dropped), **When** the `tx.Commit` call returns an error, **Then** the supervisor logs `palace_write_orphaned` at `warn` level with the ticket id, agent_instance id, and palace wing; writes the terminal row with `status='failed'`, `exit_reason='finalize_commit_failed'`, `hygiene_status='finalize_partial'`.
3. **Given** the supervisor has written either `finalize_partial` disposition, **When** the operator queries `ticket_transitions` for the affected ticket, **Then** no row exists (the transition was either rolled back or never began). The operator reconstructs the palace-vs-Postgres divergence from the `agent_instances` row plus structured logs. M3 dashboard work will surface this; M2.2.1 ships with the data available on query but no UI.

### User story 5 — the agent runs to turn-limit or is killed without ever calling `finalize_ticket` (priority: P3)

An agent performs the ticket work but never calls `finalize_ticket` — it emits a `result` event without having invoked the tool, or it hits the subprocess timeout, or it's killed by the supervisor for budget exhaustion mid-turn. In every such case, no transition occurs; the supervisor sets `hygiene_status='stuck'` on the terminal `agent_instances` row. The ticket stays at its entry column; no palace writes happen.

**Why this priority**: P3 because a well-formed seed agent.md makes this near-impossible — the completion protocol's single directive is "call `finalize_ticket`." But it can happen (model loop, budget cutoff, operator SIGTERM). The disposition needs to be distinct from `finalize_failed` (which implies attempts were made) so the operator can distinguish "agent tried and failed to produce valid JSON" from "agent didn't try at all."

**Independent test**: mockclaude fixture `finalize_never_called.ndjson` emits a series of plausible tool_use events (reads, mid-turn `mempalace_add_drawer` to `hall_discoveries`) and then a terminal `result` event without any `finalize_ticket` call. Observe: `agent_instances.status='succeeded'` by stream vocabulary (the subprocess exited cleanly), `hygiene_status='stuck'`, no `ticket_transitions` row, no palace diary (mid-turn hall_discoveries entry may still exist from the pre-finalize tool calls and is preserved per decision 5).

**Acceptance scenarios**:

1. **Given** an agent's subprocess emits `result` with `is_error == false` but the supervisor never observed a `finalize_ticket` tool_use, **When** the supervisor writes the terminal row, **Then** it sets `status='succeeded'` by stream vocabulary but overrides the logical disposition with `hygiene_status='stuck'`, `exit_reason='finalize_never_called'` (new canonical value). No `ticket_transitions` row is inserted. `total_cost_usd` is populated.
2. **Given** an agent's subprocess is killed by subprocess timeout (M2.1 path) before finalize was called, **When** the supervisor writes the terminal row, **Then** `exit_reason='timeout'` wins per Adjudicate precedence (M2.1 unchanged); `hygiene_status='stuck'` is set alongside because the finalize-disposition surface is observationally distinct from the kill-reason surface.
3. **Given** a mid-turn `mempalace_add_drawer` call succeeded before the subprocess died or finalized, **When** the operator later searches the palace for the ticket, **Then** the mid-turn drawer is visible (decision 5 preserves mid-turn writes); only the finalize-written diary + triples are absent.

### Edge cases

- **Agent calls `finalize_ticket` for a ticket it was not spawned against** (wrong ticket_id in payload). → Server rejects with `{error_type: "schema", field: "ticket_id", message: "ticket_id does not match agent_instance's ticket"}`; counts against the 3-attempt cap.
- **Two agents running concurrently against the same ticket both finalize successfully**. → Forbidden by M2.1's concurrency cap per department (`concurrency_cap=1` for `engineering`). The scenario cannot arise in M2.2.1 unless the operator manually raises the cap. If it does arise, the second `finalize_ticket` call sees the `ticket_transitions` row already exists from the first and the server returns `{error_type: "transition_write", field: null, message: "ticket already transitioned"}` — the second agent_instance ends `status='failed'`, `exit_reason='finalize_transition_conflict'`, `hygiene_status='finalize_partial'`. Spec notes this but does not enforce it architecturally beyond M2.1's cap; the combined behaviour is correct.
- **`finalize_ticket` called with `valid_from: "now"` literal**. → The finalize server substitutes the current wall clock (UTC) at validation time. The handler passes a real timestamp to `mempalace_kg_add`; MemPalace does not see the string `"now"`.
- **Network partition between supervisor and MemPalace sidecar during finalize**. → Treated as palace-write failure (US4 scenario 1). `exit_reason='finalize_palace_write_failed'`; `hygiene_status='finalize_partial'`; no orphan.
- **`finalize_ticket` payload's `kg_triples` contains 150 triples** (exceeds the 100-element cap from Clarification 2026-04-23 Q3). → Schema rejection with `{error_type: "schema", field: "kg_triples", message: "array exceeds max length 100"}`; retry attempt counted.
- **Legacy `hygiene_status` values on M2.2-era rows** (`missing_diary`, `missing_kg`, `thin`) encountered at read time by M2.2.1 code. → Passed through unchanged. Per decision 9 in context, no retroactive UPDATE. New rows from M2.2.1 onwards use only the new vocabulary (`clean`, `finalize_failed`, `finalize_partial`, `stuck`, `pending`).

## Requirements (mandatory)

### Functional requirements

**MCP server**

- **FR-251**: the supervisor MUST expose `supervisor mcp finalize` as an in-tree MCP subcommand, implemented in `internal/finalize`, following the `internal/pgmcp` pattern — single-tool MCP server, stateless, stdio-based JSON-RPC.
- **FR-252**: the `finalize` MCP server MUST expose exactly one tool named `finalize_ticket` whose `description` includes the schema version tag `garrison.finalize_ticket.v1`. The server refuses to handle non-v1 calls (future revisions bump the version per decision 4's answer; M2.2.1 implements v1 only).
- **FR-253**: the `finalize_ticket` tool's `input_schema` (per MCP protocol) MUST enforce the required fields and constraints from `m2-2-1-context.md` §"Architectural decisions (binding)" Decision 2, extended by the Clarification 2026-04-23 Q3 maxima: `ticket_id: string(uuid)`; `outcome: string, 10 ≤ length ≤ 500`; `diary_entry.rationale: string, 50 ≤ length ≤ 4000`; `diary_entry.artifacts|blockers|discoveries: array, 0 ≤ length ≤ 50, each element string ≤ 500`; `kg_triples: array, 1 ≤ length ≤ 100, each element {subject, predicate, object: string, 3 ≤ length ≤ 500; valid_from: iso timestamp or "now"}`.
- **FR-254**: the `finalize_ticket` tool's response, on successful validation, MUST be `{ok: true, attempt: <int>}` where `attempt` is the 1-based attempt number (1 on first success; 2 or 3 if earlier attempts failed). The response is an MCP `tool_result` payload.
- **FR-255**: the `finalize_ticket` tool's response on failure MUST be `{ok: false, error_type: <enum>, field: <string|null>, message: <string>, attempt: <int>}` where `error_type ∈ {"schema", "palace_write", "transition_write", "budget_exhausted"}` per decision 5. The `field` is the JSON path of the failing field (e.g. `"diary_entry.rationale"`) or `null` for non-field-scoped errors. `message` is a human-readable reason suitable for the agent to correct.

**Integration with the supervisor**

- **FR-256**: the MCP config writer MUST include the `finalize` server as a third entry in `mcpServers`, order `postgres`, `mempalace`, `finalize` (decision 3's answer). The entry's `cmd` is the supervisor binary path (resolved at startup from `os.Executable`, same mechanism as the existing `postgres` entry) and `args` are `["mcp", "finalize"]`. The entry's `env` block MUST carry `GARRISON_AGENT_INSTANCE_ID=<uuid>` (the uuid inserted in the `spawn.Spawn` dedupe transaction) per Clarification 2026-04-23 Q4, plus whatever DSN-shaped env the server needs to query `agent_instances` for the `finalize_committed` check described in FR-260 (mirrors the `pgmcp` entry's `GARRISON_DATABASE_URL` pattern).
- **FR-257**: the supervisor's per-agent-instance state (in `internal/spawn/pipeline.go`) MUST track a per-agent-instance `finalize_attempt` counter starting at 0. The counter increments on every `finalize_ticket` tool_use event seen on stream-json, regardless of validity.
- **FR-258**: the finalize MCP server MUST be stateless across process restarts — retry counts and the committed flag are NOT stored in the server process memory across process restarts. The server reads its scope (`GARRISON_AGENT_INSTANCE_ID` per FR-256) from env on startup and, on every tool call, queries `agent_instances` by that id to check whether a prior finalize has already committed (the `finalize_committed` state surfaces via the terminal row's `status` being already `'succeeded'` and a `ticket_transitions` row existing for the agent_instance). The retry counter itself is tracked supervisor-side in `internal/spawn/pipeline.go` per FR-257; the supervisor is the authority. The server's response carries the attempt number observed supervisor-side via the existing stream-json round-trip (the supervisor sees the tool_use, knows the real attempt number, and reports it through the Adjudicate/log path) — the server itself reports `attempt: 1` if it cannot determine otherwise and the value is not trusted for cap enforcement.
- **FR-259**: the supervisor's stream-json parser MUST observe `tool_use` events with `name == "finalize_ticket"` paired with their `tool_result`. On a successful validation (tool_result's `ok == true`), the supervisor triggers the atomic write path (FR-261) immediately, without waiting for the subprocess's `result` event or exit, per Clarification 2026-04-23 Q1.
- **FR-260**: after the atomic write has committed for a given agent_instance, subsequent `finalize_ticket` calls from the same subprocess MUST be rejected by the server with `{ok: false, error_type: "schema", field: null, message: "finalize_ticket already succeeded for this agent_instance", attempt: <n>}` per Clarification 2026-04-23 Q2. The server detects the already-committed state by reading `agent_instances.status = 'succeeded'` AND the presence of a `ticket_transitions` row for its `GARRISON_AGENT_INSTANCE_ID` — Postgres is the source of truth. This query happens on every tool call and is in the hot path; spec defers indexing/caching choices to `/garrison-plan m2-2-1`.

**Atomic write**

- **FR-261**: on a successful finalize payload, the supervisor MUST perform the following as a single Postgres transaction, bracketed by a context with a 30-second wall-clock timeout per Clarification 2026-04-23 Q5: (a) call `mempalace_add_drawer(wing=<agent's palace_wing>, room="hall_events", content=<serialized diary>)` via the existing MemPalace sidecar, (b) call `mempalace_kg_add(subject, predicate, object, valid_from)` once per triple in `kg_triples`, (c) `INSERT INTO ticket_transitions (...) VALUES (..., hygiene_status='clean')`, (d) `UPDATE tickets SET column_slug=<to_column>`, (e) `UPDATE agent_instances SET status='succeeded', finished_at=NOW(), total_cost_usd=<captured>, wake_up_status=<captured>, hygiene_status='clean'`, (f) `UPDATE event_outbox SET processed_at=NOW()` for the inbound event. If the 30s timeout fires before commit, the tx rolls back and FR-265a's timeout disposition applies.
- **FR-262**: the palace writes in FR-261 steps (a) and (b) MUST reuse the M2.2 `Client` type (currently defined at `supervisor/internal/hygiene/palace.go` because the M2.2 hygiene checker was its only caller) — extended with write methods (`AddDrawer`, `AddTriples`). Because M2.2.1 introduces a second caller (the atomic writer in `internal/spawn`), the `Client` type MUST be relocated to `supervisor/internal/mempalace/client.go` as part of T003; `internal/hygiene/palace.go` then imports the type rather than owning it. The binding constraint is: one `Client` type, shared by the hygiene read path and the atomic write path, living in `internal/mempalace` from M2.2.1 onwards. The transport is the same `docker exec → python -m mempalace.mcp_server → stdio JSON-RPC` pattern M2.2 already ships.
- **FR-263**: the diary content serialization in FR-261 step (a) MUST produce a body shaped as `"<ticket_objective>\n\n---\n<yaml_frontmatter>\n---\n\n<rationale>"` per Clarification 2026-04-23 Q6, where: (i) `<ticket_objective>` is the plain-prose `tickets.objective` value the supervisor reads in the same atomic-write tx (selecting from `tickets` before the `UPDATE`); (ii) `<yaml_frontmatter>` is the YAML block documented in ARCHITECTURE.md "Diary entry schema" (`ticket_id`, `outcome`, `artifacts`, `blockers`, `discoveries`, `completed_at`); (iii) `<rationale>` is the `diary_entry.rationale` string from the finalize payload, unescaped. The `completed_at` field is supervisor-generated as `NOW()` at atomic-write time. The leading objective prose is what makes `mempalace_search` by objective text return this drawer (vector-similarity works on prose; semantic-search-by-UUID does not — see Clarification 2026-04-23 Q6).
- **FR-264**: if any palace write in FR-261 step (a) or (b) fails, the supervisor MUST roll back the Postgres transaction, write a separate terminal `agent_instances` row (outside the rolled-back tx) with `status='failed'`, `exit_reason='finalize_palace_write_failed'`, `hygiene_status='finalize_partial'`, and emit a structured log at `error` level with the ticket id, agent_instance id, failing step (drawer vs. triple), and the MemPalace error text.
- **FR-265**: if the Postgres `Commit` itself fails after palace writes succeeded, the supervisor MUST log `palace_write_orphaned` at `warn` level with ticket id, agent_instance id, wing; write a separate terminal row with `status='failed'`, `exit_reason='finalize_commit_failed'`, `hygiene_status='finalize_partial'`. No attempt is made to compensate (MemPalace has no rollback; orphans are surfaced, not corrected).
- **FR-265a**: if the 30s atomic-write timeout fires per FR-261 (any step hangs — MemPalace call, Postgres write, commit), the supervisor MUST cancel the transaction context (Postgres auto-rolls-back), log `finalize_write_timeout` at `error` level with ticket id, agent_instance id, and the last step observed (diary/triple/transition/commit); write a separate terminal row with `status='failed'`, `exit_reason='finalize_write_timeout'`, `hygiene_status='finalize_partial'`. If the last step was a palace write that may have completed before the cancel took effect, emit `palace_write_orphaned` alongside per FR-265's pattern — MemPalace completions observed after the timeout are tolerated as orphans, not rolled back.

**Exit reasons and hygiene status vocabulary**

- **FR-266**: the `agent_instances.exit_reason` vocabulary MUST be extended with the canonical values: `finalize_invalid` (3-retry cap exhausted), `finalize_palace_write_failed` (atomic-write palace step errored), `finalize_commit_failed` (Postgres commit errored after palace writes), `finalize_write_timeout` (30s atomic-write ceiling fired per FR-261/265a), `finalize_never_called` (subprocess exited without issuing `finalize_ticket`), `finalize_transition_conflict` (transition row already exists; edge-case only). Existing M2.1/M2.2 exit reasons (`completed`, `acceptance_failed`, `timeout`, `supervisor_shutdown`, `spawn_failed`, `mcp_<server>_<status>`, `rate_limited`, `budget_exceeded`, etc.) remain unchanged.
- **FR-267**: the `ticket_transitions.hygiene_status` and `agent_instances.hygiene_status` values for rows written from M2.2.1 onwards MUST be one of: `clean`, `finalize_failed`, `finalize_partial`, `stuck`, `pending`. Legacy values (`missing_diary`, `missing_kg`, `thin`) remain valid for M2.2-era rows per decision 9; readers MUST accept both vocabularies.
- **FR-268**: the hygiene checker goroutine (`internal/hygiene`) MUST be simplified for M2.2.1: it reads `agent_instances.exit_reason` and the presence of `ticket_transitions` rows, and writes the matching `hygiene_status`. It no longer issues MemPalace queries (`mempalace_search`, `mempalace_kg_query`). The M2.2 palace-query code path is preserved only for legacy-value evaluation; no new calls are made against it.
- **FR-269**: the hygiene evaluator MUST produce `clean` when `exit_reason='completed'` and a `ticket_transitions` row exists; `finalize_failed` when `exit_reason='finalize_invalid'`; `finalize_partial` when `exit_reason ∈ {finalize_palace_write_failed, finalize_commit_failed, finalize_write_timeout}`; `stuck` when no `ticket_transitions` row exists and `exit_reason='finalize_never_called'` OR `exit_reason='timeout'`; `pending` during transient in-flight evaluation (spans the gap between agent_instance write and hygiene evaluation; FR from M2.2 unchanged).

**Seed agent.md rewrite**

- **FR-270**: `migrations/seed/engineer.md` MUST be rewritten for M2.2.1. The completion protocol section MUST be a single-paragraph directive instructing the agent to call `finalize_ticket` with the full structured payload (ticket id, outcome, diary_entry sub-object, kg_triples array) as its final tool call. The mid-turn MemPalace MCP tool usage (for `hall_discoveries`) MUST be preserved and explained as optional reflective-work-in-progress. The file MUST NOT instruct the agent to call `mempalace_add_drawer` or `mempalace_kg_add` for the completion diary/triples; those writes are supervisor-issued per FR-261. Target length: 3000–4000 characters per context §"Implementation notes" (non-binding guidance; brevity aids compliance).
- **FR-271**: `migrations/seed/qa-engineer.md` MUST be rewritten in parallel, with role-specific differences (wing name, verification-focused mid-turn instructions) but the same completion-protocol shape. The QA engineer's completion protocol also terminates in `finalize_ticket`. Mid-turn usage of `mempalace_search` is explained as the mechanism for reading the engineer's prior diary, with the documented query-by-objective-text pattern per Clarification 2026-04-23 Q6 (read the ticket's `objective` via postgres MCP; pass it as the `mempalace_search` query string; the returned drawer body carries the objective as leading prose followed by YAML frontmatter with `ticket_id`).
- **FR-272**: the M2.2.1 migration MUST embed the rewritten seed files via the existing `+embed-agent-md:<role>:begin|end` tooling; no schema changes to the `agents` table.

**Migration and dependencies**

It MUST NOT introduce new tables, columns, or indexes. It updates the `agents.agent_md` via the embed tooling per FR-270/FR-271, and optionally adds a CHECK constraint on `hygiene_status` values — the CHECK constraint decision is deferred to `/garrison-plan m2-2-1` (context §"Implementation notes" explicitly defers it).

**Observability**

The existing FR-218 (M2.2) structured-logging infrastructure is extended to recognize `finalize_ticket` as a first-class tool name alongside the `mempalace_*` family.
- **FR-277**: atomic-write outcomes MUST be logged at `info` level on success (fields: `ticket_id`, `agent_instance_id`, `wing`, `triple_count`, `duration_ms`) and at `warn`/`error` level on failure (fields: `ticket_id`, `agent_instance_id`, `failure_class` ∈ {palace_write, commit}, the underlying error text).

### Key entities

- **`finalize_ticket` tool input (the structured completion payload)**: the canonical reflection object an agent produces at the end of a turn. Carries `ticket_id` (uuid matching the agent_instance's assigned ticket), `outcome` (short summary string), `diary_entry` (structured rationale + artifacts + blockers + discoveries), `kg_triples` (array of knowledge-graph triples). Schema version `garrison.finalize_ticket.v1`. The payload's shape is what the operator-visible diary + triples look like after the atomic write commits.
- **Atomic completion transaction**: the single Postgres transaction FR-261 defines, bracketed around two MemPalace sidecar calls. Not a database entity but a conceptual unit whose success produces `hygiene_status='clean'` and whose partial failure produces `hygiene_status='finalize_partial'`. Understanding this unit is the key to understanding what M2.2.1 does differently from M2.2.
- **Finalize attempt counter (supervisor-side)**: an integer in `internal/spawn/pipeline.go`'s per-agent-instance state, incremented on every `finalize_ticket` tool_use event observed on stream-json. Not persisted to the database (the 3-attempt cap is enforced in memory; on subprocess death mid-retry the whole agent_instance fails and the counter is lost with it). The supervisor is the sole authority for cap enforcement; the finalize server is not given the counter and does not enforce the cap itself — it just validates each call and returns pass/fail. The cap-exhaustion outcome (`exit_reason='finalize_invalid'`) is written by the supervisor when its own counter reaches 3.
- **Hygiene status (redefined vocabulary)**: the `ticket_transitions.hygiene_status` and `agent_instances.hygiene_status` TEXT columns take a strictly enumerated set of values from M2.2.1 onwards (`clean`, `finalize_failed`, `finalize_partial`, `stuck`, `pending`), with legacy M2.2 values (`missing_diary`, `missing_kg`, `thin`) preserved on historical rows (decision 9).

## Assumptions

- The existing M2.2 MemPalace sidecar topology (socket-proxy + Python container + `docker exec` transport) continues to function unchanged. M2.2.1 reuses `internal/mempalace.Client` for its supervisor-side palace writes; the docker-exec transport, the JSON-RPC framing, the palace path default, and the sidecar container name are all M2.2-inherited.
- Claude Code supports a third MCP server entry in the per-invocation config without regressing M2.2's two-server init-parse contract. The MCP spec supports arbitrary server counts; M2.1's init-event parser handles `mcp_servers[]` as a list (per spike §1.7). No spike is required for the third-entry case because the parser is list-shape-agnostic; a unit test will confirm this in implementation.
- The per-invocation cost cap of $0.10 (M2.2's NFR-201) remains appropriate for the `finalize_ticket`-enabled flow. The one-retry case costs marginally more than M2.2's direct-write flow (one extra tool call); the 3-retry exhaustion case could plausibly approach the cap. The cap is unchanged for M2.2.1; if live runs show three-retry scenarios routinely exhausting it, the cap revision is an M3 task, not M2.2.1.
- The MemPalace 3.3.2 `mempalace_add_drawer` and `mempalace_kg_add` tools accept the payload shapes M2.2.1's supervisor-side writer produces (wing/room/content; subject/predicate/object/valid_from). These are validated behaviorally in M2.2's spike and shipped code; M2.2.1 reuses the identical transport path.
- The operator's dashboard work for reconciling `hygiene_status='finalize_partial'` orphans is deferred to M3. M2.2.1 produces observable orphan data (logs + agent_instances row) but no UI.
- The MCP protocol allows servers to read env vars from their own process environment (standard stdio subprocess semantics); `GARRISON_AGENT_INSTANCE_ID` per FR-256 uses the same mechanism as M2.2's `pgmcp` entry and has no spec-compliance risk.

## Dependencies on prior milestones

- M2.2's MemPalace wiring (`internal/mempalace`, `internal/hygiene`, `internal/spawn`, the MCP config writer, the MemPalace sidecar topology) ships in production before M2.2.1 work begins. This is already true as of 2026-04-23.
- M2.1's subprocess-lifecycle discipline (process groups, pipeline-drain-before-Wait, init-event health check, stream-json parser) is unchanged and load-bearing. Any regression in M2.1 paths breaks M2.2.1 identically.
- M1's event bus, `event_outbox.processed_at` guarantee, and `pg_notify` channel semantics are unchanged. The atomic write's `UPDATE event_outbox` step reuses the M1 shape.
