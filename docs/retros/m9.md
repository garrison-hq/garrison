**Shipped**: 2026-06-11 (implementation phase — commits T001…T021 on
branch `021-m9-scheduled-wakeups`; T001 `2afb153` → T020 `3b2dc80`).
**Plan**: [`specs/021-m9-scheduled-wakeups/plan.md`](../../specs/021-m9-scheduled-wakeups/plan.md)

**Spike**: none — deliberate (see "No spike" in
[`specs/_context/m9-context.md`](../../specs/_context/m9-context.md)).
**M8 retro** (substrate): [`docs/retros/m8.md`](./m8.md)
**M7.1 retro** (transport): [`docs/retros/m7-1.md`](./m7-1.md)

M9 adds the proactive axis to the event loop: a supervisor-internal
tick fires concrete named jobs on a cadence, alongside — never
replacing — the pg_notify-reactive loop. Four threads ship together:
(1) the `scheduled_tasks` / `scheduled_task_runs` schema + bounded
expression grammar + tick loop with SKIP LOCKED claim and
fire-on-recovery-with-collapse; (2) two firing modes — ticket
(rendered ticket into the normal Kanban flow) and oneshot (containered
spawn exiting through the new `finalize_oneshot` sibling tool); (3)
the dashboard `/admin/recurring-jobs` CRUD surface with audit-writing
Server Actions; (4) chat-driven `create_scheduled_task` as the 11th
sealed verb with a per-turn ceiling. Zero idle cost is preserved: the
tick is one SQL claim query; no agent wakes to look for work.

## What the spec/plan got wrong

### Migration version collision — second milestone running

The plan named the migration `20260610000000_*`; that goose version
was taken (M7.1-era `agents_changed_trigger`), and `20260610000001`
by M7.1's own migration. Shipped as `20260610000002`. M7.1's retro
flagged exactly this ("check same-day versions at plan time") and the
M9 plan still carried a colliding name. The check belongs in the
plan template, not the retro.

### The "existing outbox insert query" didn't exist

Plan §1 assumed the oneshot firing could reuse an existing sqlc
outbox-insert query. The only outbox inserts in the codebase live
inside SQL trigger functions — there was no Go-callable query.
Shipped as a tx-level raw query (mcpserverwork precedent) with the
channel literal baked in.

### `isFinalizeToolName` was a hidden ticket-mode assumption

`spawn/pipeline.go`'s matcher only recognized `finalize_ticket`, so
the oneshot pipeline could never observe the `finalize_oneshot`
tool_use/tool_result pair and T008's onCommit routing could never
fire. The plan treated the pipeline as mode-agnostic; the matcher
wasn't. Caught by T016's golden path before any live run — in
production this would have produced oneshot sessions that did their
work and then terminated as finalize-never-observed. Matcher extended;
per-mode tool exclusivity at the MCP server keeps ticket mode
byte-for-byte unchanged. Any future finalize sibling tool must touch
this matcher — see gotchas.

### Import cycle forced the integration suite out-of-package

The golden-path suite dispatches into `internal/spawn`, and spawn
imports `internal/schedule` — an in-package `schedule` test importing
spawn is a cycle the toolchain rejects. The suite lives in package
`schedule_test` with a small exported-for-test `TickOnce` bridge. The
plan placed the tests in-package; the fix is shape-preserving.

### Coverage probe vs orchestration-only files

First probe run read 45.4% new-code coverage — the gap sat entirely
in `spawn/oneshot.go`'s runner orchestration (reachable only
end-to-end) and the `mcp_finalize.go` main-package entry, the exact
file class `sonar-project.properties` already excludes. M8's remedy
applied: exclusions amended with documented rationale, real tests
topped up on every unit-reachable surface (finalize guards, validate
wrappers, RunLoop error paths, arg parsing, handler branches) →
88.1%.

### Local-host accommodations in the acceptance run

go1.25.0's trimmed module-cache toolchain breaks on-demand covdata
for no-test main packages (the M7.1 `make test` gap, now worked
around in-script by enumerating per-tag tested packages), and
container-backed suites run `-p 1` (M7.1 address-pool lesson).
Environment friction, not product bugs; both documented in
`scripts/m9-acceptance.sh`.

## The context file's named retro questions

**Did the zero-idle-cost invariant hold?** Yes, by construction and
by test. The tick is one SKIP LOCKED claim query; nothing spawns,
nothing spends tokens, when no task is due. `TestZeroIdleCost` pins
it observationally (N idle ticks → zero runs, zero instances, zero
tickets, zero outbox rows). The full SC-008 reading (≥1 hour live
supervisor, zero scheduled spawns/spend) rides the operator-attended
acceptance pass alongside the SC-006 dashboard walk.

**Did any schedule misfire or double-fire?** No. The chaos suite
drove concurrent claims at one due task and got exactly one firing,
one run row, one advance; the SKIP LOCKED probe confirmed contended
rows are skipped, not waited on. No misfire or double-fire was
observed in any suite or during development. The one near-miss —
the `isFinalizeToolName` gap — was not a scheduling fault: the tick
and spawn were correct; the session exit path was the hole, and it
was caught at T016 before any live firing existed.

**Did the collapse rule behave on a real supervisor restart?** At the
integration level, yes: `TestRecoveryCollapseFiresOnce` seeds a task
whose `next_fire_at` lies three slots in the past — exactly the row
state a supervisor finds after being down across those slots — and
the next tick produces one firing and one future-dated slot, no
backfill. Because `Expr.Next` is strictly future and there is no
missed-slot bookkeeping anywhere in the schema, collapse is a
property of the date arithmetic, not a recovery code path; there is
no separate "restart mode" to misbehave. A literal stop-the-binary /
restart walk against the seeded stack is part of the
operator-attended acceptance posture.

**Did chat authoring stay inside the amended threat model?** Yes.
The amendment (T010) is committed in history before the verb code
(T011) — asserted by `git log` in the acceptance script, not by
convention. The shipped surface matches the amended model exactly:
`create_scheduled_task` is the only chat-reachable scheduled-task
verb (Tier 3, full-args audit row, chat-session anchor);
edit/pause/resume/delete exist in `ServerActionVerbs` as
registry-only entries whose Go handlers are typed refusals; agent
callers are rejected at the handler; the per-turn ceiling (default 3)
bails the turn with a typed reason at call N+1. The live
chat-session walk is operator-attended.

## Gotchas for the next milestone

1. **`gate_deferred` asymmetry is deliberate** (documented in
   `schedule/deps.go`): for oneshot runs it is non-terminal — the
   outbox event stays unprocessed and a successful poll re-dispatch
   clears the run back to `fired`; for ticket-mode dept-weekly
   defers it is terminal for that slot. Don't "fix" one to match the
   other.
2. **Scheduled ticket firings count against department weekly
   budgets** — the M8 dept-weekly gate counts ALL tickets in the
   window, now including schedule-fired ones. Budget tuning on
   departments with daily scheduled tickets must account for ~7
   firings/week of standing load.
3. **`agent_instances` origin is exactly-one-of** `ticket_id` /
   `scheduled_task_run_id` (CHECK-enforced). NULL `ticket_id` now
   means "oneshot origin", just as NULL `pid` means "container path"
   since M7.1.
4. **Any future finalize sibling tool must extend
   `isFinalizeToolName`** in `spawn/pipeline.go`, or the pipeline
   will never observe its commit. Third sibling should probably turn
   the matcher into a registry.
5. **Check same-day goose versions at plan time** — two consecutive
   milestones shipped with a plan-named migration version that was
   already taken.
6. **`{{fire_at}}` renders the firing timestamp**, i.e. the value
   persisted to `last_fired_at` — under recovery collapse that is
   the actual (late) firing time, not the missed slot's nominal
   time. Template consumers should not treat it as the slot time.

## Post-ship adversarial review (2026-06-11)

A spec-fidelity review ran against the merged-candidate branch the day
after the implementation phase closed; nine findings, seven fixed
(`fix(m9): … (review #N)` commits below), two explicitly accepted.

1. **Oneshot double-fire on the 2nd consecutive overlapping slot
   (CRITICAL → fixed, `d6a67d2`).** `HasRunningOneshotForTask`
   examined only the latest run by `fired_at`; the slot-2
   `skipped_overlap` row became the latest run and blinded the
   predicate, so slot 3 fired a second oneshot while slot 1's
   instance was still running. Reshaped to the ticket-mode pattern:
   EXISTS over ANY run of the task with
   `outcome IN ('fired','gate_deferred')` and a NULL-or-still-running
   joined instance. `TestTickOnceConsecutiveOverlapStillSkips`
   reproduces the exact fire → skip → must-skip sequence.

2. **Oneshot spend invisible to the M6 budget (HIGH → fixed,
   `27a0119`).** Two halves: (a) `oneshotOnCommit` discarded the
   pipeline `*Result`, so the committed terminal instance row carried
   NULL cost — now mirrors the ticket-mode ResultGrace contract,
   capturing `TotalCostUSD` + wake-up status at commit time via the
   new `OneshotTerminal` argument (NULL cost only when the grace
   window elapses without a result). (b) `GetCompanyThrottleState`
   JOINed through `tickets`, silently dropping every `ticket_id`-NULL
   oneshot instance from the rolling-24h sum — now resolves company
   through `agent_instances.department_id → departments`. The
   pre-existing M6 throttle suite passes unchanged.

3. **Chat verb couldn't see `GARRISON_SCHED_MIN_INTERVAL` (HIGH →
   fixed, `48ed13a`).** The verb read the env var in the
   garrison-mutate MCP subprocess, but `BuildChatConfig` never put it
   there — every chat-authored task was bounded by the 15m default
   regardless of operator tuning. The bound now rides the
   garrison-mutate entry's env (the DATABASE_URL pattern), threaded
   `cfg → chat.Deps → writeChatMCPConfig`. The verb keeps its
   fallback-to-default-with-warning (agent-mode entries deliberately
   never carry the var; the verb rejects agent callers anyway). A
   parity test pins the verb's rejection to the exact
   `/schedule/validate` detail string for the same config value.

4. **Dashboard validation missed role existence + name uniqueness
   (MEDIUM → fixed, `748a2f4`).** The validate endpoint was
   expression-only (handler constructed without queries), so the
   dashboard caught duplicate names only as an index collision and
   unknown roles not at all. `POST /schedule/validate` now accepts
   optional task-identity fields and runs the full
   `schedule.ValidateTask` (the chat verb's path) with typed field
   errors; `scheduledTasks.ts` create/edit send the full body, the
   ad-hoc department SELECT is gone, and unique-name violations map
   to a friendly message (closes analyze-finding U2). Expression-only
   bodies are untouched (resume).

5. **Hardcoded oneshot thin-diary threshold (MEDIUM → fixed,
   `3f900b3`).** `oneshotThinDiaryThreshold = 200` ignored
   `GARRISON_HYGIENE_THIN_DIARY_THRESHOLD`. `spawn.Deps` gains
   `ThinDiaryThreshold`, wired from config in main.go; the zero value
   falls back to the documented 200 default.

6. **Self-referential descriptor pin test (LOW → fixed, `f74aea1`).**
   `TestToolsListTicketModeUnchanged` compared the served descriptor
   against `ToolDescriptor()` — both current code, so drift would
   self-confirm. The marshaled pre-M9 descriptor is now frozen as a
   golden string constant; both surfaces compare against it.

7. **Finalize matcher widened into ticket pipelines (LOW → fixed,
   `3afbae4`).** The T016-era `isFinalizeToolName` union matcher
   meant a ticket agent's hallucinated `finalize_oneshot` call ticked
   the 3-attempt counter (pre-M9 it didn't). The expected finalize
   tool is now explicit per pipeline (`FinalizeDeps.ToolName`, ticket
   name as the empty-value default; oneshot sessions pass
   `finalize_oneshot`). This supersedes gotcha 4 above: a future
   sibling extends the per-pipeline `ToolName`, not a shared matcher.

8. **`last_fired_at` stays bumped on pre-pipeline spawn failure (LOW
   → ACCEPTED, documented, not fixed).** The tick tx advances
   `last_fired_at` when the slot's outcome is `fired`; if the
   dispatcher's spawn later fails pre-pipeline, the run flips to
   `outcome='failed'` but the task row keeps the bumped timestamp.
   Brief rendering is protected regardless: `{{last_fired_at}}` for
   oneshot briefs resolves through `selectPreviousFiringSQL`, which
   filters `outcome='fired'` — a failed firing never feeds a later
   brief. The task-row column is operator-display state; accepting
   the drift beats widening the tick tx with a compensating write.

9. **Four raw-SQL queries outside sqlc (LOW → ACCEPTED, documented,
   not fixed).** `insertOneshotOutboxSQL` (schedule/fire.go),
   `insertRunningOneshotInstanceSQL`, `selectPreviousFiringSQL`, and
   `markOneshotEventProcessedSQL` (spawn/oneshot.go) live as tx-level
   raw constants, each self-justified in a code comment (trigger-era
   outbox shape, sealed-query-set precedent, FR-107 previous-firing
   semantics, sealed WriteFinalizeOneshot signature). The plan's own
   §sqlc table disagreed with its walkthrough about what the query
   set contained, which invited the drift — the acceptance here is
   that the comments carry the justification; the lesson for the
   plan template is to keep the query-set table authoritative.
