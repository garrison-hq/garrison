## What shipped

### Cross-boundary schema (T001 + T011 follow-up)

Two new Postgres roles created via goose migrations:

- `garrison_dashboard_app` — operational reads on M2-arc tables
  (tickets, ticket_transitions, agents, agent_instances, event_outbox,
  departments, companies). Also the role better-auth uses against
  its own dashboard-side tables (Drizzle grants the writes).
- `garrison_dashboard_ro` — vault sub-views ONLY; SELECT on
  agent_role_secrets, vault_access_log, secret_metadata, plus the
  joinable read tables required by FR-021.

Both ship as `NOINHERIT NOLOGIN`; operators set passwords + flip
LOGIN at deploy time per docs/ops-checklist.md M3 section.

`migrations/20260426000010_m3_dashboard_roles.sql` (T001)
`migrations/20260426000011_m3_dashboard_dept_grants.sql` (T011 follow-up:
the org overview + Kanban surfaces also need departments + companies)

### Dashboard subsystem scaffold (T002–T004)

Next.js 16 + React 19 + Tailwind v4 + Vitest + Playwright bootstrap
under `dashboard/`. Drizzle ORM with a partitioned schema:

- `drizzle/schema.supervisor.ts` — introspected from goose-managed
  M2-arc migrations via `bun run drizzle:pull`. Never hand-edited.
- `drizzle/schema.dashboard.ts` — hand-written; the source-of-truth
  for the five dashboard-owned tables (better-auth core +
  operator_invites). drizzle-kit generate diffs against this file
  only.

Three Drizzle clients, one role each — `appDb` / `vaultRoDb` /
`makeListenClient`. The vault-isolation pin is enforced at the
source level by `lib/queries/vault.test.ts` and at the runtime
level by Postgres GRANTs.

### Auth + invites (T005–T007)

- `lib/auth/index.ts` initialises better-auth with the drizzle
  adapter, additional `themePreference` user field surfaced on
  every session read, email/password as the only provider.
- First-run wizard at `/setup` returns 404 once any user exists.
- `/api/invites/redeem` is the only path to create accounts
  post-first-run (FR-002e).
- Atomic invite redemption uses a single `UPDATE … WHERE
  redeemed_at IS NULL AND revoked_at IS NULL AND expires_at > now()
  RETURNING id`; concurrent redemptions serialise via Postgres MVCC
  (proven by the 9th unit test in `lib/auth/invites.test.ts`).

### Shell + i18n + theme (T008–T009)

- next-intl with `localePrefix: 'as-needed'` so English routes
  stay unprefixed; the test build adds a stub `zz` locale to
  exercise the locale-swap path without baking a second real
  catalog.
- 10 UI primitives (Chip, StatusDot, Kbd, Tbl, KpiCard, ExpandRow,
  ThemeSwitcher, LocaleSwitcher, EmptyState, CostCaveatIcon) +
  Sidebar + Topbar + LogOutButton.
- Theme persistence via `users.theme_preference`; PUT /api/theme
  writes back. Resolved at the root layout via `resolveTheme()`
  (system fallback → 'dark') and emitted as
  `<html data-theme=…>`.

### Seven read surfaces (T010–T014)

- `/` — org overview (KPIs + per-department row strip, 60s soft-poll)
- `/departments/[slug]` — Kanban (4 columns, no drag handle)
- `/tickets/[id]` — metadata + history (with sandbox-escape icon
  + ExpandRow detail) + agent-instances (with cost-blind-spot
  caveat icon) + palace-links placeholder
- `/hygiene` — filterable hygiene table (3 failure-mode buckets,
  30s soft-poll, suspected_secret_emitted shows pattern category
  only — never the matched value)
- `/vault`, `/vault/audit`, `/vault/matrix` — vault sub-views via
  `garrison_dashboard_ro` only
- `/agents` — agents registry with per-agent stats from
  agent_instances (last spawned + spawns_this_week)
- `/admin/invites` — pending-invites list + generate form

### SSE activity feed (T015–T016)

- Singleton `lib/sse/listener.ts` with a 5-state machine
  (dormant/connecting/live/backoff/idle-grace), 100ms→30s
  exponential backoff, 60s idle-grace before closing.
- `KNOWN_CHANNELS = ['work.ticket.created']` literal channels;
  `KNOWN_CHANNEL_PATTERNS` for the parameterised
  `work.ticket.transitioned.<dept>.<from>.<to>` family.
- `lib/queries/activityCatchup.ts` replays event_outbox rows
  after a Last-Event-ID cursor in chronological ASC order.
- `/api/sse/activity` route handler, auth-gated, with 25s
  heartbeat to keep proxies alive.
- Activity feed surface buckets events by agent_instance_id into
  RunGroups; @tanstack/react-virtual keeps render cost bounded.
  10-event acceptance proven in tests/integration/activity-feed.spec.ts.

### Cross-cutting + deployment (T017–T020)

- Theme parity / responsive (768/1024/1280) / concurrent-operators

- `dashboard/Dockerfile` two-stage build (oven/bun:1-alpine →

## What the spec got wrong

### FR-003 vs FR-020 inconsistency (caught at /speckit.analyze)

The spec said better-auth tables were goose-managed (FR-003) and
also Drizzle-managed (FR-020). The clarification round resolved the
ownership question (Drizzle) but FR-003 wasn't reconciled until the
analyze pass. Already fixed before T001 implementation; recorded as
a learning that the clarification log should be the single
source-of-truth for amended FRs and non-amended FRs should be
re-read end-to-end before /speckit.implement.

### Inviter-deletion edge case vs `ON DELETE RESTRICT`

The spec edge case says "if an inviter is deleted via direct SQL
while their invite is pending, the invite remains valid." Plan's
data-model originally had `created_by_user_id … ON DELETE RESTRICT`,
which would block the deletion entirely. Fixed during analyze
remediation; the schema now uses `ON DELETE SET NULL`.

### `garrison_dashboard_ro` joinable-table grants

T001 (per data-model) only granted SELECT on the three vault tables.
FR-021 also requires SELECT on `tickets, ticket_transitions, agents,
agent_instances` (so the audit-log filters and matrix render can
join). Caught at analyze; T001's migration body was extended before
implementation.

### FR-060 wording — pg_notify mechanism vs event coverage

Original FR-060 implied every event in the activity feed comes
from a pg_notify channel the supervisor subscribes to. Reality:
the supervisor only emits two channels (`work.ticket.created`,
`work.ticket.transitioned.…`). Finalize and agent-instance lifecycle
events live in `event_outbox` but aren't pg_notify'd. Reworded
during analyze to separate the *coverage requirement* from the
*subscription mechanism*; T015's listener implements both
(LISTEN + event_outbox poll fallback).

### Drizzle uuid + better-auth ID generation

Better-auth's drizzle adapter passes nanoid-shaped strings as the
user `id` by default; the schema's `id uuid` rejected them. Fixed
with `advanced.database.generateId: false` so the Postgres default
`gen_random_uuid()` takes over. Documented during T005.

### Drizzle timestamp mode + better-auth

Initial schema used `mode: 'string'` for timestamps; better-auth
passes Date objects. Postgres-js rejected the cast. Switched all
dashboard timestamps to `mode: 'date'`.

### sessions.ip_address: inet vs text

Better-auth passes an empty string for the IP when no request IP
is parseable (Playwright in-process navigation, etc). The Drizzle
`inet` type rejects `""` with `"invalid input syntax"`. Switched
to `text` to match better-auth's reference schema.

### Turbopack root drift on this monorepo layout

`bun run dev` (Turbopack) drifted up to the repo root looking for
tailwindcss because of a stray `garrison/package.json + bun.lock`
left behind by an earlier `bun add` from a moved cwd. Fixed by
deleting the stray files; integration tests use `next start`
against a prebuilt server (closer to production behaviour anyway).

### data-theme nesting

First attempt at the [locale]/layout.tsx put `<html data-theme=…>`
inside the layout; Next.js's hoisting flattened it and the
attribute didn't reach the rendered DOM. Moved data-theme to the
root layout (`app/layout.tsx`); the [locale] layer wraps children
in NextIntlClientProvider only.

### postgres-js JSONB round-trip in tests

`${jsonString}::jsonb` interpolation in postgres-js's tagged
template lands the value as a quoted scalar rather than a parsed
object. Test seeds use `jsonb_build_object` instead.

## Dependencies added outside the locked list

M3 introduces a substantial dashboard-side dep set. The supervisor's
locked-deps streak is supervisor-scoped (Go); dashboard deps follow
the same justification discipline on the dashboard side per
AGENTS.md soft-rule. SC-011 captures these in this retro:

1. **next 16.2.4** — the App Router framework. No alternative for
   the React-Server-Components + Server-Actions surface plan. M3
   chose Next 16 over a non-framework setup because of the
   dashboard's surface count (8 routes + a streamed feed) and the
   built-in i18n / API-route / standalone-build affordances.

2. **react 19.2.5 / react-dom 19.2.5** — paired with Next 16. No
   alternative.

3. **tailwindcss 4.2.4 + @tailwindcss/postcss** — token system in
   the mocks is built around CSS variables; Tailwind v4's
   variable-driven utility generator is the cleanest match.
   Alternatives (PostCSS + custom utility generator, vanilla CSS)
   would re-derive what Tailwind already provides.

4. **drizzle-orm 0.45.2 + drizzle-kit (devDep)** — type-safe query
   builder; partitioned schema (introspected supervisor side +
   hand-written dashboard side) supports the goose/drizzle
   migration split per FR-020. Alternatives (Prisma, Kysely) would
   couple worse with better-auth's drizzle adapter.

5. **postgres 3.4.9 (postgres-js)** — driver. Required for
   per-connection LISTEN semantics (T015's listener client). pg
   was considered; postgres-js's tagged-template interface
   composes more cleanly with drizzle.

6. **better-auth 1.6.9** — auth runtime. Hand-rolling sessions
   was rejected upfront; the load-bearing reasons are CSRF,
   cookie management, and the email/password provider with
   bcrypt-backed account.password column. Better-auth's drizzle
   adapter integrates with the dashboard's existing schema
   without a second persistence layer.

7. **next-intl 4.9.1** — i18n. Required by FR-012 + FR-013.
   Alternatives (react-i18next, FormatJS) work but next-intl is
   the only one with a first-class App Router middleware
   integration that handles `localePrefix: 'as-needed'` cleanly.

8. **@tanstack/react-query 5.100.5** — TanStack Query. Used by
   the soft-poll surfaces (org overview, hygiene, audit log).
   Alternatives (custom useEffect + setInterval, SWR) are
   simpler but lose retry-on-focus + staleTime semantics.

9. **@tanstack/react-virtual 3.13.24** — virtualization for the
   activity feed (FR-065). At 10 assistant events per run × N
   concurrent agents, the unbounded event list needs windowing;
   the alternative is a fixed visible window with manual pagination,
   which loses the "scroll back through history" affordance.

10. **testcontainers 11.14.0 + @testcontainers/postgresql** —
    devDep. Required by plan §"Test strategy" for testcontainer-
    backed integration tests on the dashboard side. Alternatives
    (manual docker-compose + setup hooks) would re-derive what
    testcontainers already provides.

11. **vitest 4.1.5 + @vitest/ui (devDep)** — unit-test runner.
    Pre-justified by plan §"Technical Context".

12. **playwright 1.59.1 + @playwright/test (devDep)** — e2e
    runner. Pre-justified by plan §"Technical Context".

13. **@types/node, @types/react, @types/react-dom, typescript,
    postcss (devDeps)** — required by the runtime deps above.

Total new dashboard-side deps: 13 direct + ~150 transitive. Each
is justified above; the bar for future additions on the dashboard
side mirrors the supervisor's locked-deps soft-rule.

## Open questions deferred to the next milestone

1. **Drizzle CLI in production image**: per CL-004, drizzle-kit is
   NOT in the runtime image. Operators run migrations from a
   separate ops shell. Reaffirm or revisit at M4 when mutations
   land.

2. **next start vs next standalone**: T021's golden-path uses
   `bun run start` (which warns about `output: 'standalone'`).
   The Dockerfile uses `node server.js` against the standalone
   bundle directly — the production runtime is correct. Only the
   test harness uses `next start`. Could migrate the harness to
   the standalone runtime for closer production parity in M4.

3. **Pattern-category extraction for `suspected_secret_emitted`**:
   M3 surfaces a generic "secret-shape" label (FR-074). The M2.3
   scanner's pattern label set lives in supervisor code; threading
   each label through to the hygiene table requires either a new
   migration column or a join against a label table. Deferred to
   M4 (which gains schema mutation rights).

4. **Activity-feed run grouping for cross-instance flows**: today
   the run is keyed by agent_instance_id. M5's CEO chat will
   produce events that span multiple agent instances; the
   grouping model needs revisiting then.

5. **Cost-blind-spot fix**: surfaced everywhere total_cost_usd
   shows; not fixed in M3 per spec §"What this milestone is NOT".
   Tracked at `docs/issues/cost-telemetry-blind-spot.md`.

6. **Workspace-sandbox-escape fix**: surfaced in hygiene + ticket
   detail; not fixed in M3 per the same scope discipline. Tracked
   at `docs/issues/agent-workspace-sandboxing.md`.

7. **next-intl + force-dynamic interaction**: pages that use
   `getTranslations()` work fine in static prerender as long as
   they don't pull in DSN-bound modules; pages that do pull such
   modules need `force-dynamic`. M3 set this where needed; future
   surfaces should keep the pattern.
