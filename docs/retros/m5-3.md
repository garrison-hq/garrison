## What the spec got wrong

### Verb arg shapes diverged from the actual ticket schema

The spec / plan ┬¦13 enumerated `create_ticket` args as `(title, description, priority, labels)` ŌĆö the M4 idealized shape. The real `tickets` schema has `objective` (single field, not title+description split), `acceptance_criteria` (nullable text), `column_slug` (with default `'todo'`), `metadata` (JSONB), `origin` (text, no CHECK enum on the M2.x set). The verb implementation adapted to the real schema (`CreateTicketArgs` carries `objective`, `department_slug`, `acceptance_criteria`, `column_slug`, `metadata`); the threat model + chip surface remained shape-agnostic. The spec's "(title, description, priority, labels)" remains as a forward-looking sketch ŌĆö `tickets` schema may extend toward M4-shape fields in a later milestone.

### Verb count "9" vs "8" inconsistency caught by /speckit.analyze

`/speckit.analyze` flagged a four-site inconsistency where the prose said "nine verbs" / "9-verb set" but the explicit list contained 8. Remediation commit (`efd0803`) corrected all four sites in spec/plan/tasks. The architecture amendment paragraph plan ┬¦10 also corrected. No production-code drift; corrected before T018 landed the doc.

### Stale `[NEEDS CLARIFICATION]` markers in spec post-clarify

`/speckit.clarify` resolved 5 of 8 deferred items but didn't clean up all spec markers; 3 markers remained referencing already-resolved items (FR-444 failure-chip copy, FR-494 sidebar placement, edit_agent_config edge case bullet). `/speckit.analyze` caught these; `efd0803` replaced the markers with the plan's resolved defaults.

### `internal/finalize` precedent had a `Raw` field on `ToolResultSummary` that the chat side doesn't have

T012's first cut wired `EmitToolResult` to pass `tr.Raw` from `claudeproto.ToolResultSummary`. The struct has `Detail string` instead ŌĆö `Raw` doesn't exist on the chat-side type. Adapted to emit a synthetic `{detail, is_error}` envelope; full structured Result lives in `chat_messages.raw_event_envelope` per M5.2 amended FR-261 reconnect mechanism. No spec impact; emit shape is internal.

### sqlc COALESCE-with-cast confused parameter-name inference

The first cut of `migrations/queries/tickets.sql` used `COALESCE(NULLIF($4::text, ''), 'todo')` for the column_slug default. sqlc 1.x generated `Column4 string` / `Column5 interface{}` positional names instead of inferring `ColumnSlug` / `Metadata`. Switched to `@named-param` syntax + Go-side default resolution. Same pattern applied to `UpdateTicketEditableFields`. No spec impact; SQL idiom-level.

### `init()`-replaced handler vars need closure-indirection in the registry slice

verbs.go's `Verbs` slice initially captured the package-level `handleCreateTicket` function values at var-decl time (which were the stub-handlers at that moment). Per-domain `init()` reassignments to `handleCreateTicket = realCreateTicketHandler` updated the var but not the slice's stored function value. Fix: forwarding closures in the slice that look up the var per call (`func(ctx, deps, args) { return handleCreateTicket(ctx, deps, args) }`). Caught by `TestRegisteredHandlersAreRealNotStubs`.
