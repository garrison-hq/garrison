## What shipped

One new in-tree MCP server (`internal/finalize`) exposing a single
`finalize_ticket` tool; extensions to five existing packages
(`config`, `exitreason`/`spawn`, `mempalace`, `mcpconfig`, `hygiene`);
two rewritten seed `agent.md` files (engineer + qa-engineer) at
3559 / 3593 bytes each (within the 3000–4000 target); one migration
(`20260424000005_m2_2_1_finalize_ticket.sql`; seed-UPDATE only, no
schema DDL, no CHECK constraint); seven mockclaude fixtures for the
new control-flow paths; five integration + one chaos + one
live-acceptance test file.

The empirical goal: turn agent-prompt-dependent MemPalace writes into
agent-tool-mandatory MemPalace writes so the memory thesis (RATIONALE
§3, as revised by this milestone) is no longer model-compliance-gated.

Architectural change: the supervisor's stream-json parser now observes
`finalize_ticket` tool_use/tool_result pairs in addition to the M2.2
`mempalace_*` pairs. On a successful finalize, the pipeline fires
`spawn.WriteFinalize` which runs an atomic Postgres transaction
bracketing two MemPalace sidecar calls (AddDrawer + N× AddTriples)
and the four DML writes (ticket_transitions insert, tickets update,
agent_instances terminal, event_outbox mark-processed). The whole
path is bounded by a 30-second wall-clock ceiling
(`GARRISON_FINALIZE_WRITE_TIMEOUT`). Retries cap at 3, enforced by a
supervisor-side counter in `internal/spawn/pipeline.go`.

`hygiene_status` vocabulary gains `finalize_failed`, `finalize_partial`,
and `stuck` alongside the M2.2 values (`clean`, `pending`,
`missing_diary`, `missing_kg`, `thin`). The hygiene listener now routes
on `agent_instances.exit_reason`: finalize-shaped rows go through a
pure-Go `EvaluateFinalizeOutcome`; legacy M2.2 rows continue through
the palace-query-based `Evaluate`. Two vocabularies coexist on the
column during the transition window (per plan §"Decisions baked into
this plan" item 12 — no retroactive UPDATE, no CHECK constraint).

   by `/speckit-analyze` as finding I2 (HIGH) before implementation;
   remediated by amending FR-262 + T003 to include a relocation step.
   The relocation turned into the largest T003 diff (691 insertions,
   290 deletions) but preserved all M2.2 hygiene tests unchanged via
   type aliases — behavioral continuity intact.

2. **T007 test scope vs. practical coverage.** Tasks.md named five
   DB-requiring unit tests (`TestWriteFinalizeHappyPath`,
   `TestWriteFinalizeRollsBackOnPalaceError`, etc.) but on
   implementation those duplicate what T013's golden-path and T016's
   chaos integration tests exercise against real infrastructure.
   Retro note: the plan's test-strategy section would benefit from a
   "unit vs integration coverage overlap" check during
   `/speckit-analyze` — two different tests validating the same code
   path is a tell that one should move.

3. **FR-261 internal cross-reference.** Early draft said
   "FR-264a's timeout disposition applies"; the timeout disposition
   lives in FR-265a. Caught by `/speckit-analyze` as finding I1 (HIGH)
   and remediated before T001. Lesson: sub-letter FRs (FR-264a,
   FR-265a) make cross-references fragile; future milestones should
   prefer consecutive top-level FR numbers and only introduce sub-
   letters when the addition is genuinely tightly-coupled.

4. **`mempalace_search` cross-wing for diary lookup (T009).** The
   `/speckit-clarify` session 2026-04-23 Q3 (objective prose prepend
   on diary body) was remedy for M2.2's live-run finding that search

## Open questions deferred to M2.3+

1. **Dashboard surfacing of `finalize_partial` orphans.** When
   `exit_reason=finalize_commit_failed` (palace has orphan drawer/
   triples but Postgres rolled back), operator reconciliation is
   manual in M2.2.1. M3's dashboard work should offer a "reconcile
   orphan palace writes" action. Noted in
   [plan.md §"Open questions"](../../specs/005-m2-2-1-finalize-ticket/plan.md)
   item 1.

2. **CI automation of the compliance matrix.** SC-261's haiku-vs-opus
   test is operator-invoked via the `live_acceptance` build tag.
   Automating it in CI requires parameterizing the Anthropic key +
   per-run budget allocation; deferred to operator choice.

3. **Mid-turn `hall_discoveries` searchability.** The objective-prose-
   prepend in FR-263 makes the supervisor-written `hall_events`
   diaries searchable by `mempalace_search(query=<ticket.objective>)`.
   Agent-written mid-turn `hall_discoveries` drawers do NOT get this
   prepend — the agent composes them freely. If mid-turn discoveries
   need cross-ticket search, the agent is responsible for including
   searchable prose. M3's query-reshape work should address this
   systematically.

4. **Retry state durability.** The supervisor-side retry counter
   lives in process memory (`internal/spawn/pipeline.go`). If the
   supervisor crashes mid-retry, the counter is lost and the agent
   rerun would reset to attempt 1. Acceptable for M2.2.1 (crashes
   rare; retry reset is fail-open, not fail-closed). M3+ may want to
   persist the counter.

## Compliance-thesis validation

**The headline outcome of the milestone.** M2.2 live-run recorded:

> "real claude haiku-4-5-20251001 executed the ticket READ but did NOT
> call `mempalace_add_drawer` or `mempalace_kg_add` despite the
> agent.md marking them 'MANDATORY'. The palace remains empty post-run
> for this ticket. This is the exact soft-gate failure mode RATIONALE
> §3 + §5 anticipates."

M2.2.1's thesis: replacing "MANDATORY in prompt" with "MANDATORY via
tool mechanism" removes the compliance gap because:

- The agent cannot exit with a successful transition any other way —
  `finalize_ticket` is the only tool that commits the transition.
- The supervisor commits the memory writes from the tool payload, not
  from subsequent agent tool calls — so once the tool is called
  successfully, the writes are guaranteed.
- The schema is strict enough that the agent can't satisfy it with
  empty / trivial content (outcome ≥ 10 chars, rationale ≥ 50,
  kg_triples ≥ 1 with each field ≥ 3 chars).

Empirical validation: [to be filled by T017's
`TestM221ComplianceModelIndependent` run at operator acceptance].
Expected observation: haiku-4-5-20251001 and claude-opus-4-7 produce
identical palace-populated + hygiene_status=clean outcomes. If they
don't — if haiku somehow fails to finalize despite the tool being the
only exit path — the thesis needs revision and M3 delays pending a
second-order fix.

Pre-live-run prediction: **thesis holds**. The tool-architectural
compliance is stricter than prompt-based compliance by construction.
The failure mode M2.2 observed (agent ignoring the prompt) cannot
produce a successful transition in M2.2.1 — the agent either calls
`finalize_ticket` (at which point writes are guaranteed) or it
doesn't (at which point no transition lands and the ticket is
observably stuck). No middle ground.

## Live-run append (2026-04-23)

`TestM221ComplianceModelIndependent` executed against real claude-
haiku-4-5-20251001 (opus run deferred). Cost: $0.0927 for the engineer
subprocess. Full log at `/tmp/m221-compliance-claude-haiku-4-5-20251001.log`;
mechanical observations in `specs/005-m2-2-1-finalize-ticket/acceptance-evidence.md`.

### Thesis: partially validated

The pre-run prediction said "tool-architectural compliance is stricter
than prompt-based by construction." The observation refines that to:

**Strictly better than M2.2, not fully compliant.** The good news:
haiku DID call `finalize_ticket` — which is more than happened in M2.2
where haiku skipped MANDATORY `mempalace_add_drawer` entirely. The
tool-as-attractor pattern worked: haiku treated finalize_ticket as the
natural completion mechanism.

The behaviour gap: haiku's first `finalize_ticket` call failed schema
validation (likely malformed JSON — the server returned
`error_type="schema", field=""` which is the decode-level rejection,
not a specific field violation). Haiku did NOT respond by correcting
and retrying. Instead, haiku spent the next ~25 seconds exploring the
palace (`mempalace_search`, `mempalace_status`, `mempalace_list_drawers`,
`mempalace_kg_query`) — behaviour consistent with "error from tool X
→ try tool Y" rather than "error from tool X → fix arguments and
retry tool X."

Adjudicate correctly classified this as `finalize_never_called`
(attempts reached 1, not 3; no SIGTERM-on-cap-exhaustion fired; the
subprocess emitted a natural `result` event and exited). No
`ticket_transitions` row committed; the ticket stayed at `in_dev`.

### What this tells us about tool-architectural compliance

- The tool IS the attractor. Haiku reached for finalize_ticket without
  being told to — the agent.md's "your last tool call MUST be
  finalize_ticket" combined with the tool's schema description was
  enough to draw the first attempt.
- Strict schemas punish probabilistic models. One malformed JSON from
  a smaller model is enough to derail the retry discipline because
  haiku doesn't interpret schema errors as "fix and try again" — it
  interprets them as "try something else."
- The 3-attempt cap is the wrong knob. What would help more: (a)
  relaxed-first-attempt / strict-final-attempt grading; (b) richer
  error feedback pinpointing the exact parse failure; (c) concrete
  example payload in the seed agent.md.

### Implications for M2.2.1 ship + M3 planning

**M2.2.1 ships** as a partial-compliance milestone. It's strictly
better than M2.2 (agent at least attempts the completion mechanism;
an operator watching the hygiene dashboard sees `finalize_never_called`
as a signal rather than M2.2's mysterious palace silence). The
architectural pivot is validated; the retry semantics need tuning.

**Open work for M3 (or a focused M2.2.2 patch)**:
1. Example-based seed agent.md — include a concrete `finalize_ticket`
   payload in the engineer.md so haiku's first attempt has a template.
2. Richer schema error responses — the handler currently returns
   `error_type="schema", field=""` for `json.Unmarshal` failures.
   Parsing the exact error position (line/col) would give haiku
   actionable feedback.
3. Prompt-level retry framing — the completion section says "you have
   3 attempts" but doesn't frame retry as expected. Rewording to
   "correct the error and call finalize_ticket again" might shift the
   model's behaviour.
4. Flexible-first-attempt mode — accept a looser shape on attempt 1,
   tighten on attempts 2-3. Trades architectural purity for practical
   compliance rate.

### Opus comparison — executed (2026-04-23 second pass)

First attempted run had a test-setup bug: the seed `agents.model`
column (hardcoded to haiku in SeedM22) overrides `GARRISON_CLAUDE_MODEL`
because `spawn.go` uses `agent.Model` first. Fixed with `UPDATE agents
SET model = <test-model>` in the test's setup. Confirmed via the
stream-json `model` field.

**Actual opus-4-7 observation**: opus emitted 2 `thinking` + 2 tool_use
events in 6 seconds, then Claude's internal budget check triggered
`terminal_reason="error_max_budget_usd"` at $0.2564 (on a $0.20 cap —
Claude's `--max-budget-usd` is advisory, not hard). Neither tool_use
was `finalize_ticket` — opus never reached the completion tool before
its budget fired.

Cost: $0.2564 (2.8× haiku's $0.0927 for the same time window).

**Cross-model finding (the honest SC-261 answer)**:

Both models fail to produce `hygiene_status='clean'` on the live-
compliance test, but for DIFFERENT reasons:
- **haiku** reaches `finalize_ticket` (attempt 1), fails schema,
  doesn't retry, explores palace, exits cleanly.
- **opus** doesn't reach `finalize_ticket` at all — burns budget on
  early thinking + non-finalize tool calls.

The SC-261 "haiku ≡ opus" assertion holds only in the trivial
negative sense (both don't finalize). It does NOT hold in the
intended validated-thesis sense (both produce clean outcomes).

### Bonus finding — M2.2 Adjudicate precedence bug

Opus's `is_error=true + terminal_reason="error_max_budget_usd"`
combination surfaced a pre-existing M2.2 bug: the `IsError` case
runs above the `isBudgetTerminalReason` case in Adjudicate's
precedence table, so Claude's own budget-cap signal gets classified
as `claude_error` rather than `budget_exceeded`. This is load-
bearing for operator observability: a budget-cap event currently
shows up as a generic claude_error, hiding the cost root cause.

**Fix is a one-line swap** (check budget-terminal-reason before
is_error when ResultSeen=true), but it's not in M2.2.1's scope —
flagging as a focused M2.2.2 / M3 item.

### Implications for M2.2.1 ship + M3 planning (refined)

didn't reach for mempalace_* at all). The hygiene_status dashboard

**M3 / M2.2.2 follow-up work (expanded)**:
1. Example-based seed agent.md (concrete `finalize_ticket` payload

   benefit: haiku.
2. Richer schema error responses (JSON parse line/col). Primary

3. Prompt-level retry framing ("correct and retry"). Primary

4. Flexible-first-attempt mode. Primary benefit: both models, but

5. Front-load the finalize directive — bring `finalize_ticket` to
   the top of the agent.md work loop so opus reaches it before
   burning budget on context exploration. Primary benefit: opus.
6. Fix Adjudicate precedence so Claude's own budget-cap doesn't
   get masked by `IsError`. Primary benefit: operator observability
   (both models).
