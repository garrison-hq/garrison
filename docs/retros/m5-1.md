## What shipped

### Schema + cross-cutting refactors (T001–T004)

Four foundational landings before the `internal/chat/` package proper:

- `migrations/20260428000015_m5_1_chat_backend.sql` — `chat_sessions`
  + `chat_messages` + the indexes that drive the listener's restart
  sweep and the dashboard's transcript read. `dashboard/drizzle/
  schema.supervisor.ts` regenerated; the lifted types for both new
  tables flow into the dashboard's `lib/queries/chat.ts` and
  `lib/actions/chat.ts` server actions without hand-edits.
- `internal/dockerexec/` — promoted from `internal/mempalace/
  dockerexec.go` (M2.2's helper) into a top-level package, gaining a
  `RunStream` method that takes a `writeStdin` callback and a
  `scanStdout` callback. Both mempalace and chat depend on it; the
  rename was confirmed during clarify Q-D and folded into the
  milestone scope.
- `internal/spawn/policy.go` — extracted the `spawn.Policy` interface
  out of the existing `pipelineRouter`; renamed `pipelineRouter` to
  `FinalizePolicy` (constructor `NewFinalizePolicy`); changed the
  spawn `Run()` to accept a policy. **Every existing finalize
  behaviour preserved byte-for-byte** — verified by re-running the
  M2.1 + M2.2 + M2.2.1 + M2.2.2 integration suite green.
- `internal/claudeproto/router.go` — added `Router.OnStreamEvent` +
  `StreamEvent` type + dispatch. `FinalizePolicy` ships a no-op
  `OnStreamEvent` so the M2.x finalize path is unchanged; the chat
  package wires `OnStreamEvent` to live delta forwarding.

### Chat package internals (T005–T012)

Eight commits that build bottom-up inside `supervisor/internal/chat/`:

- **`deps.go` + `errorkind.go`** — `Deps` struct (Pool, Queries,
  VaultClient, DockerExec, MCPConfigDir, container/network/timeout
  config) + the canonical `ErrorKind` vocabulary
  (`session_cost_cap_reached`, `session_ended`, `vault_token_not_found`,
  `vault_token_expired`, `vault_unavailable`, `claude_oauth_invalid`,
  `mcp_postgres_unhealthy`, `mcp_mempalace_unhealthy`,
  `container_crashed`, `subprocess_timeout`, `parser_error`).
- **`persistence.go`** + sqlc queries in `migrations/queries/chat.sql`
  — `CreateChatSession`, `InsertOperatorMessage`,
  `InsertAssistantPending`, `MarkAssistantStreaming`,
  `MarkAssistantCompleted`, `MarkAssistantError`, `RollUpSessionCost`,
  `MarkActiveSessionsIdle`, `GetSessionTranscript`,
  `GetActiveSessionsForRestartSweep`. Tested against testcontainer
  Postgres.
- **`transcript.go`** — multi-turn NDJSON assembler. Reads
  `chat_messages` for a session in turn-index order and emits the
  prompt-stream-json shape Claude expects on stdin replay (spike §5
  confirmed this is the intended replay shape; SC-002's
  `cache_read_input_tokens > 0` assertion proves the supervisor's
  output reaches the container's stdin correctly).
- **`transport.go`** — `pg_notify` wrappers for `chat.delta` (per-
  message) and `chat.audit` (session-level). Channel names follow
  M2.1's pg_notify convention (LISTEN-on-quoted-channel-name pattern).
- **`policy.go`** — `ChatPolicy` implements both `spawn.Policy` and
  `claudeproto.Router`. Live deltas forwarded via `OnStreamEvent`;
  terminal writes wrapped in `context.WithoutCancel(ctx)` +
  `TerminalWriteGrace` per AGENTS.md rule 6; `OnTerminate` finalises
  the assistant row with `error_kind` populated on failure paths.
- **`spawn.go`** + `internal/mcpconfig/buildchat.go` — `SpawnTurn`
  composes the docker-run argv (`docker run --rm -i --network
  garrison-net -e CLAUDE_CODE_OAUTH_TOKEN=… -v <mcp.json>:/mcp.json
  garrison-claude:m5 …`), wires the MCP config bundled with the
  chat-specific server set (`postgres` + `mempalace` only — no
  `finalize` server, enforced by `BuildChatConfig`). The
  `tokenEnvSpec` site is the only `vault.UnsafeBytes` call in the
  chat path, marked `//nolint:vaultlog` matching the M2.1 spawn.go
  pattern (single env-var injection point; never reaches the rawEvents
  accumulator).
- **`worker.go`** — per-message orchestration: cost-cap pre-check →
  `InsertAssistantPending` row → spawn → release per-session mutex
  on completion or error. `sessionMutexRegistry` keyed on session_id
  serialises turns within a session; concurrent sessions still run
  in parallel.
- **`listener.go`** + `restart.go` + `idle.go` — supervisor-side
  LISTEN dispatch on `chat.message.sent` (drives the worker per
  message) + boot-time restart sweep (re-emits `chat.message.sent`
  for messages stuck in `pending`/`streaming` after a supervisor
  crash, per FR-083) + idle-timeout ticker (60s production cadence,
  injectable via `Deps.IdleSweepTick` for tests; marks active
  sessions whose newest message is older than `SessionIdleTimeout`
  as `'ended'`, per FR-081).

## What the spec got wrong

### `internal/dockerexec/` rename was a Q-D answer, not a planned task

The plan named `internal/chat/` as the only new package; the spike
already showed that mempalace's `dockerexec.go` had the right
subprocess shape but lived in the wrong package, and chat would need
a `RunStream` method that mempalace didn't. Clarify Q-D folded the
rename + `RunStream` extension into M5.1 as T002. Cleanly resolvable
in-flight; the rename touched 1 import site (mempalace) and added 1
new package (chat). No external callers since both packages are
supervisor-internal.

### `Deps.IdleSweepTick` wasn't in the plan

The 60s production ticker for `RunIdleSweep` is the right operational
cadence but makes the SC-013 test impossible to run in CI without
either sleeping a minute or cheating the cutoff at the SQL layer.
The fix landed at retro-prep time (commit `301e742`) — added a
`Deps.IdleSweepTick` field that defaults to 60s but can be shrunk
to 1s in tests. Two-line change to `listener.go`. The lesson: any
production interval that gates a success criterion needs an
injectable override at the `Deps` boundary, decided at design-time
rather than discovered at test-write-time.

### SC-001 was originally framed as a Playwright-only assertion

The spec wrote SC-001 as "First SSE delta within 5s wall-clock from
operator INSERT" and the acceptance doc initially deferred the
wall-clock pin to T020 Playwright. Reality: the canned-NDJSON
integration path completes in <1s and a `time.Since(turnStart) ≤ 5s`
assertion in `TestM5_1_HappyPath_SingleTurn` is a meaningful
regression guard (catches any change that serialises the worker /
persistence / transport hops behind a blocking call). The Playwright
run still adds the dashboard-side wall-clock measurement but it's
no longer the only signal. Lesson similar to the M3 / M4 pattern:
"Playwright will catch it" defers verification to a slower harness
when an in-process assertion would have caught regressions sooner.

### `chat_messages.content` is `*string`, not `string`

sqlc generated the column as nullable by default because the M5.1
migration left `content` without a `NOT NULL` constraint. The pointer
type ripples through every callsite (`InsertOperatorMessageParams`,
`GetSessionTranscriptRow`). M5.1 lives with it via `ptrFn(s string)
*string` helpers in tests; a future schema tightening could add the
`NOT NULL` constraint (operator messages are always non-empty;
assistant messages move from NULL → text on completion).

### Mockclaude turn-aware template required real extension

The plan named "mockclaude chat-mode" as a tagged-along T013
extension. Reality: turn-aware response selection (favorite-color →
"Purple" on turn 2) + cache_read_input_tokens emission on turn ≥ 2
needed an actual stdin parser inside mockclaude that counts prior
user/assistant pairs in the prompt-stream-json input. Landed cleanly
in commit `7739d5c`; the extension is small (~80 lines) but spec
treated it as a one-line change. Captured the actual scope at
implementation time.

## Open questions deferred to the next milestone

1. **T020 Playwright e2e**. Boots the full M5.1 stack (testcontainer
   Postgres + Infisical + `garrison-mockclaude:m5` chat image +
   supervisor process + standalone dashboard bundle) via an
   extended `_chat-harness.ts`. Closes SC-010's runtime measurement
   and SC-011's automated token rotation. Deferred because the
   harness extension is a multi-hour build and the in-process
   integration suite already covers the structural assertions for
   SC-001 and SC-002. The chat-image build step is idempotent
   (cached after first run); first-run cost is a one-time CI infra
   step.

2. **`chaos_m5_1_test.go`** — real docker-kill chaos test wired
   against supervisor + docker-proxy + `garrison-mockclaude:m5`
   end-to-end. Closes SC-006's live SIGTERM-cascade verification
   and FR-101's external-kill scenario. Same harness as T020;
   conventionally lands in the same follow-up session.

3. **Explicit-close server action (FR-082 → M5.2)**. The idle-timeout
   sweep handles abandoned sessions; an explicit "close session"
   action that the operator drives intentionally is M5.2 territory
   when the chat UI lands.

4. **Warm-pool optimization for cold-start latency (RATIONALE 127
   follow-up)**. Each turn spawns a fresh container; cold-start adds
   measurable wall-clock to the first delta. The spike measured
   1.4s on warm cache (§8.1); operator-felt latency on a cold image
   pull is unmeasured. If the operator-week-of-use evaluation
   surfaces this as friction, M5.2 or M5.3 may layer in a warm-pool
   pattern.

5. **Threat-model amendment for chat-driven mutations (M5.3)**. M5.1
   ships read-only — `BuildChatConfig` enforces no `finalize`
   server in the chat MCP set. M5.3 (when chat can drive ticket
   decomposition or vault edits) needs a threat-model amendment
   covering the mutation paths: actor identity in
   `vault_access_log.metadata`, audit-trail completeness, the
   leak-scan parity for chat-initiated writes.

6. **Orphan operator row case (spec §Edge Cases)**. The spec accepts
   as a known property: if the supervisor crashes between operator
   INSERT and assistant `pending` row creation, the operator row
   sits at `status='completed'` with no assistant counterpart and
   the restart sweep doesn't catch it (filters on `status IN
   ('pending','streaming')`). The polish-round mitigation (sweep
   detects "session whose newest message is `role='operator'`
   without an assistant pair, > 60s old") is M5.2 polish-round
   work.

## Spike pay-off audit (retro-mandated)

From the M5 spike (`docs/research/m5-spike.md`):

**(a) Did the spike pay off?** Yes. The spike pinned six binding
decisions before the context doc named them:

- §3 confirmed **ephemeral containers** (no warm pool) match the
  operator-billing constraint cleanly. M5.1 ships exactly that:
  one container per turn, `--rm` on the docker run, no shared
  state across turns except the stdin replay.
- §5 confirmed **stdin stream-json multi-turn** is the working
  replay path. M5.1's `transcript.go` assembles the same shape
  the spike's manual reproduction used; SC-002 verifies the
  output reaches the container correctly.
- §8.1 captured the wire-shape **canned NDJSON** that the
  `fakeDockerExec` test fixture mirrors. Every spike-recorded
  field type matches what the supervisor's parser expects;
  zero parser surprises across all integration runs.
- §8.2 confirmed **tool-isolation** for chat: `BuildChatConfig`
  rejects mutation servers by name, the chat MCP set is exactly
  `postgres` + `mempalace`, no `finalize` server. The fail-closed
  test (`TestBuildChatConfig_RejectsMutationServerByName`)
  prevents an agentic-tool leak that would have been easy to
  introduce.
- §6 confirmed `CLAUDE_CODE_OAUTH_TOKEN` env-var injection works
  end-to-end against a real container; M5.1's `tokenEnvSpec`
  callsite in `spawn.go` uses the same shape.
- §7 confirmed the docker-proxy `CREATE: 1` flag is the minimal
  expansion of the M2.2-era proxy allow-list (POST + EXEC +
  CONTAINERS already present); compose change in T014 is one
  line.

**Issues prevented vs. discovered ratio**: 6 prevented (above) vs.
3 genuinely surprising discoveries (`Deps.IdleSweepTick` not in
plan; mockclaude turn-aware extension scope larger than expected;
`chat_messages.content` nullability ripples). Pay-off well above
the ≥ 50% validation-rate target the spike-first pattern targets.

**(b) Did stdin stream-json multi-turn replay hold up under the real
mockclaude binary?** Yes. SC-002's `cache_read_input_tokens > 0`
assertion passes against the real `garrison-mockclaude:m5` chat-mode
binary, proving the supervisor's transcript-assembly output reaches
the container's stdin in the shape Claude expects. The mockclaude
emits non-zero `cache_read_input_tokens` on turn ≥ 2 by counting
prior user/assistant pairs in stdin — so the assertion is a real
wire-shape check, not just a value check.

**(c) Did the docker-run invocation pattern hold up?** Yes. No
container-startup races, no orphaned containers across all
integration runs. The `--rm` flag handles cleanup on success; the
chaos test (T019) verifies cleanup on container crash; supervisor
SIGTERM cleanup is structural via `context.WithoutCancel(ctx)` +
`TerminalWriteGrace` (live verification deferred to the chaos
follow-up).

**(d) Was the `BuildChatConfig` exhaustive-allow-list approach
right?** Yes. The fail-closed test
(`TestBuildChatConfig_RejectsMutationServerByName`) is the kind of
guard that's easy to forget when the threat is "future commit adds
a server"; making it the *first* test in `mcpconfig_test.go` for
chat means any future addition to the MCP set has to confront the
test. Same shape as M2.1's pattern of writing the parser-rejection
tests before the parser-accept tests.

**(e) Were there real-runtime surprises the spike didn't catch?**

- **`Deps.IdleSweepTick`** — the 60s production interval is correct
  but un-injectable. Discovered at retro-prep (SC-013 test write);
  fix was 2 lines. The spike measured the production cadence but
  didn't think about CI-cadence.
- **Mockclaude turn-aware extension scope** — turn-counting in the
  stdin parser is ~80 lines, not the one-line tweak the plan
  implied. The mockclaude binary needed to actually parse the
  prompt-stream-json input format; it had been a static
  responder before chat-mode landed.
- **`chat_messages.content` nullability ripples** — sqlc generates
  pointer types for nullable columns; the design choice (assistant
  rows move from `content=NULL` to `content=text` on completion)
  is correct but ripples through every callsite. Living with the
  pointer types in M5.1; possible schema tightening in a future
  milestone.

Three discoveries against six prevented issues — the spike paid off
roughly 2:1 by issue count, well above the ≥ 50% threshold from the
M2.x spike pattern.

## What this milestone is NOT

Recap of held non-goals:

- **No CEO chat UI** (M5.2). M5.1 ships server actions and an SSE
  endpoint; the dashboard chat surface (left-rail session list,
  message thread, composer) is M5.2.
- **No chat-driven mutations** (M5.3). `BuildChatConfig` enforces
  no `finalize` server; the chat MCP set is `postgres` + `mempalace`
  only, both read-mostly. Chat that can decompose tickets or edit
  vault is M5.3 with a threat-model amendment.
- **No CEO ticket decomposition** (M6).
- **No hiring or skill-browser flows** (M7).
- **No agent-spawned tickets** (M8).
- **No per-day or per-month cost caps** (FR-062 explicitly out of
  scope; soft per-session cap only).
- **No multi-user / multi-tenant** — operator-only single pool
  continues from M3.
- **No mobile-first design** — responsive ≥768px continues.
- **No fix for the cost-telemetry blind spot or workspace-sandbox
  issue** — both still surfaced from M3.
- **No replacing or revising any sealed M2-arc surface.**

## Ready to start M5.2

M5.1 ships a complete server-side chat substrate with no unresolved
issues blocking the M5.2 UI build:

- **Server actions** (`startChatSession`, `sendChatMessage`) are
  the binding interface the UI calls; both are tested and stable.
- **SSE endpoint** at `/api/sse/chat?session_id=…` already emits
  `delta` and `terminal` events in the shape an EventSource client
  consumes directly.
- **Transcript reads** via `getSessionTranscript` are paginated-ready
  (single-session bounded result; per-message ordering by
  `turn_index, role`) and surface the rolled-up cost.
- **Idle sweep + restart sweep** keep the session lifecycle
  self-healing without UI involvement; M5.2 doesn't need to think
  about the "operator left tab open for 4 hours" case.

The two follow-up items (T020 Playwright + chaos test) are
test-harness gaps, not feature gaps. They land in a focused next
session before or in parallel with M5.2 UI work.
