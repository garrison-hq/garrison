### Spawn-path integration

Seven-step ordering in `spawn.go` (D4.5 / FR-416):

1. `ListGrantsForRole` ŌĆö query grants for this role+customer.
2. `mcpconfig.CheckExtraServers` (Rule 3 pre-check) ŌĆö runs BEFORE vault fetch.
3. `vaultOrchestrate` ŌĆö Infisical fetch, classify errors, abort with appropriate exit_reason.
4. `RuleOneLeakScan` (Rule 1) ŌĆö abort if any fetched value appears in agent.md.
5. `WriteAuditRow` inside the spawn tx ŌĆö abort (fail-closed) if INSERT fails.
6. Build subprocess env with `SecretValue.UnsafeBytes()`; `cmd.Start()`; `val.Zero()` in defer.
7. Existing M2.2.x subprocess pipeline ŌåÆ finalize path, unchanged.

Eight new exit_reasons, one new hygiene_status value (`suspected_secret_emitted`).

## What the spec got wrong

**Rule 3 ordering ambiguity**: The spec said Rule 3 runs inside `mcpconfig.Write`. The plan resolved this as "BEFORE vault fetch" but the initial implementation had Rule 3 only inside `WriteWithOps`, which runs AFTER the vault fetch. Fixed by adding `mcpconfig.CheckExtraServers` as a pre-fetch call in spawn.go. The plan was right; the first implementation missed it. Spec and plan were consistent ŌĆö execution deviated.

**Infisical proxy vs real 403/429**: The spec assumed testcontainers could be configured to return 403 (no-project-access ML) and 429 (rate limit) natively. In practice, Infisical's API behavior for an ML with no project access doesn't cleanly produce a 403 on secret fetch (the ML auth succeeds, but the project-level permission check happens differently). Used a local HTTP proxy (`newInfisicalProxy`) injecting status codes for matching URL paths. Same result, different mechanism. Spike did not cover this path; the research spike was for the general SDK shape, not error-injection testing.

**Auth-expired test**: The spec imagined "rotate the ML's client_secret before spawn." Infisical's SDK caches the access token eagerly at client construction and renews on 401. To force re-auth failure, used a short-lived ML (`accessTokenTTL=1`, `numUsesLimit=1`) so the token expires (1s TTL) and the re-auth also fails (secret exhausted). The spec's framing was correct as intent but not actionable as written.

**`setupVaultFailureMode` stub**: An early attempt to extract a shared helper for failure-mode tests was added but never wired up. Deleted before T015 commit. The individual tests were simple enough not to warrant a helper.

## Dependencies added outside the locked list

M2.3 breaks the five-consecutive-milestone zero-new-deps streak with two principled additions (FR-425):

1. **`github.com/infisical/go-sdk v0.7.1`** (T004): The official Infisical Go client. Provides Universal Auth, secret fetch, and machine identity management. Stdlib HTTP + `encoding/json` cannot reproduce the Universal Auth token refresh loop and the Infisical-specific secret-path API shape without reimplementing the SDK. Alternatives considered: raw `net/http` client (would need to implement/maintain the auth flow + error classification ourselves); Vault OSS client (wrong product). The SDK is MIT-licensed and has a small dependency footprint.

2. **`golang.org/x/tools v0.44.0`** (T010): Provides `go/analysis` ŌĆö the framework for the `vaultlog` custom vet analyzer. No stdlib alternative for building a `go vet`-compatible analyzer. Alternatives: a standalone linter script (less composable); a regexp-based grep (can't distinguish types). `golang.org/x/tools` is the canonical extension module for the Go toolchain and is maintained by the Go team.

Additionally, two existing indirect deps were promoted to direct (organic `go mod tidy` consequence of T011 directly importing them):
- `github.com/testcontainers/testcontainers-go v0.42.0`
- `github.com/testcontainers/testcontainers-go/modules/postgres v0.42.0`

These were already in the module graph as transitive deps; promotion is not a new security surface.

## Garrison's first Postgres trigger

The `rebuild_secret_metadata_role_slugs()` trigger (AFTER INSERT/DELETE on `agent_role_secrets`) is Garrison's first use of a PL/pgSQL trigger in the production schema.

**Observed behavior**: The trigger fires correctly under goose up-down-up. The `TestSecretMetadataPopulatedAtBootstrap` integration test (T015) validates it end-to-end: after an `INSERT INTO agent_role_secrets`, the trigger rebuilds `secret_metadata.allowed_role_slugs` to `['engineer']` within the same transaction ŌĆö verified by a SELECT after the INSERT in the test.

**Surprises**: None in development. The `DROP TRIGGER IF EXISTS` + `CREATE OR REPLACE FUNCTION` Down/Up pattern in the migration is idiomatic and composed correctly.

**Recommendation for future denorm columns**: Follow this trigger pattern when the denorm column is derived exclusively from another table's content and the rebuild is cheap (array aggregation). Avoid triggers for denorm columns that require cross-schema joins or expensive computation ŌĆö prefer an explicit recompute query in the application layer for those cases.

## Custom vet analyzer

The `tools/vaultlog` package implements a `golang.org/x/tools/go/analysis` pass that rejects slog/fmt/log-family calls with `vault.SecretValue` arguments.

**Behavior during implementation**: The analyzer was run post-T008 and post-T009. It reported zero diagnostics against the production codebase. The two `UnsafeBytes()` call sites (spawn env injection, Rule 1 leak scan) are not slog/fmt calls ŌĆö they reach the raw bytes for the OS process or substring comparison, not for logging. The finalize scanner (`scanAndRedactPayload`) touches raw bytes via `ScanAndRedact(p.DiaryEntry.Rationale)` on the `string` type (not `SecretValue`), so it is also not flagged.

**False positives**: Zero in the codebase. The analyzer is correctly scoped to the `vault` package type and the slog/fmt/log function set.

**Maintenance burden**: Low. The analyzer's pattern set matches the Go standard printing surfaces. If a new logging library is added, the analyzer's `logFuncs` list needs a one-line update. The test suite in `tools/vaultlog/vaultlog_test.go` runs as a unit test with `go test ./...` and catches regressions.
