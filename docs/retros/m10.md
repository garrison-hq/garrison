# M10 retro — Ingress connectors (the company becomes reactive to the outside world)

**Shipped**: 2026-06-12 (implementation phase — commits T001…T017 on
branch `022-m10-ingress-connectors`; T001 `a018b93` → T017 `0486ac0`).
**Plan**: [`specs/022-m10-ingress-connectors/plan.md`](../../specs/022-m10-ingress-connectors/plan.md)
**Tasks**: [`specs/022-m10-ingress-connectors/tasks.md`](../../specs/022-m10-ingress-connectors/tasks.md)

**Spike**: [`docs/research/m10-spike.md`](../research/m10-spike.md)
**M9 retro** (substrate): [`docs/retros/m9.md`](./m9.md)

M10 makes external events first-class ticket origins on the existing bus. The
connector framework ships alongside the GitHub connector as the first concrete
connector: HMAC-SHA256 signature verification over the raw body, fail-closed;
`ingress_deliveries` unique-constraint-on-insert idempotency surviving the M1
dedup race; per-connector token-bucket rate cap at the HTTP edge wired to M6
throttle evidence; provenance in `tickets.metadata`; a separate `ingress.Server`
on port 8082; and the dashboard `ingressOrigin` chip + `/admin/connectors` status
surface. Zero new Go or TypeScript dependencies.

## What the spec/plan got wrong

### `issue.id` as `json.Number` not `int64`

`github.go`'s `Map` method assigns `issue.id` (a JSON integer field) to
`TicketDraft.ExternalID string`. If the payload struct declares `ID int64` and
`ExternalID` is a `string`, the assignment `draft.ExternalID = issue.ID` is a
compile error. The task description explicitly flagged this ("do not assign an
`int64` directly to `ExternalID string`") and the fix — `strconv.FormatInt` or
parsing via `json.Number` — was applied correctly. The plan's code snippet did
not show a full struct type; the implementation correctly resolved the coercion.
No prod bug, but the plan's snippet was slightly misleading.

### `GetConnectorStatus` query name vs tasks.md prose

The tasks.md completion condition for T002 listed `GetConnectorStatus :one` as
the status query name. The actual implementation named it `GetConnectorStatus` (as
one query), but the dashboard API required multiple distinct reads (last delivery,
accepted count, rate-cap breach count from throttle_events separately) rather than
a single monolithic JOIN. The query was adapted to match the handler's actual data
fetch pattern. Behaviorally correct; the plan's query description was simplified.

### Drizzle empty-default mangling — same gotcha, third consecutive milestone

Despite the gotcha being noted in the M7, M8, and M9 retros and the T002
completion condition explicitly including the inline-fix step, Drizzle pull
still requires a manual fixup after regeneration when new columns with default
values appear. Applied and confirmed clean. This is now a repeatable known step,
not a surprise.

## The context file's named retro questions

### (a) Did idempotency hold under real redelivery and under the M1 concurrent-delivery race?

Yes, on both axes.

**Serial redelivery**: `TestIngress_SerialRedelivery_NoSecondTicket` POSTs the same
`X-GitHub-Delivery` GUID twice against a live `ingress.Server`; the second POST
returns 200 with no second ticket, no second `ingress_deliveries` row, and no
second `work.ticket.created` notify. The `23505` unique-constraint violation on the
second `InsertIngressDelivery` call is the dedup signal — no pre-check SELECT is in
the codebase.

**Concurrent-delivery race (the M1 shape)**: `TestIngress_ConcurrentRedelivery_RaceYieldsOneTicket`
drives two goroutines from a start barrier with the same GUID. Exactly one
`tickets` row and one `ingress_deliveries` row are committed; both requests return
2xx. The unique-constraint-on-INSERT locks at the index level in Postgres, so the
second concurrent writer blocks on the index lock until the first tx commits or
rolls back, then sees `23505` — surviving the M1 race without any in-memory dedup
layer.

### (b) Did signature verification fail closed against a forged delivery?

Yes. `TestIngress_BadSignature_NoTicket_401` posts a delivery whose
`X-Hub-Signature-256` header carries a forged HMAC against the live stack. The
handler returns 401, zero `tickets` rows are created, zero `ingress_deliveries`
rows are written, and the rejection counter increments. `TestVerifyGitHubSignature_EmptyHeader`
confirms that an absent header also returns `ErrBadSignature` (fail-closed per
FR-300). The constant-time comparison path via `crypto/subtle.ConstantTimeCompare`
was exercised by `TestVerifyGitHubSignature_Mismatch` and `TestVerifyGitHubSignature_Valid`.

### (c) Did the rate cap actually bound a burst and did the breach write correct M6 evidence?

Yes. `TestIngress_BurstExceedsCap_BoundedTickets` drives 2× burst deliveries as
fast as possible at a low-cap connector. Results:

- Ticket count ≤ configured burst (fan-out bounded at the HTTP edge, FR-600/FR-602).
- At least one `throttle_events` row with `kind='ingress_rate_cap_exceeded'` is
  written with the correct JSON payload (`connector_id`, `rate_per_minute`, `burst`)
  and the `work.throttle.event` notify fires (FR-601).
- Over-cap responses are 429 with no `ingress_deliveries` row, so a subsequent
  redelivery of a 429'd GUID (new row never inserted) processes correctly to one
  ticket (FR-602, idempotency composing with rate-cap correctly).

`TestRateCap_PerConnectorIsolation` confirms two connectors have independent
buckets.

### (d) Did the inbound-only boundary hold — did anything accidentally reach back out to GitHub or any external service?

Yes, and asserted by script. SC-007 in `scripts/m10-acceptance.sh` runs:

```
git grep -r "github.com/google/go-github\|PostComment\|SendMail\|external_action" supervisor/internal/ingress/
```

and asserts empty output. No outbound call, no GitHub API read-back, no mail send,
no external mutation exists anywhere in `internal/ingress/`. The GitHub connector
ingests the POST payload and creates a ticket; nothing reaches back out. M11's
Action Broker is the correct boundary for outbound action.

### (e) Did the threat model (`ingress-threat-model.md`) precede all connector code in git history?

Yes, asserted by script. SC-008 in `scripts/m10-acceptance.sh` runs:

```
git log --oneline -- docs/security/ingress-threat-model.md supervisor/internal/ingress/
```

The output shows `T001: threat model — docs/security/ingress-threat-model.md`

### Did the M10 spike pay off?

Yes. `docs/research/m10-spike.md`'s SR1–SR10 findings were accurate:

- **SR1 (HMAC-SHA256, raw body, stdlib)**: exactly as implemented. `crypto/hmac`
  + `crypto/sha256` + `crypto/subtle.ConstantTimeCompare` + `encoding/hex` — zero
  dependency, raw body captured before any JSON parse.
- **SR2 (X-GitHub-Delivery idempotency key, manual redelivery only)**: confirmed.
  The GUID is stable across redeliveries and the manual-redelivery model means
  the retention window can be generous. The `UNIQUE (connector_id, external_delivery_id)`
  constraint-on-insert is the dedup signal.
- **SR3 (repo/org webhook + pre-shared secret, no GitHub App)**: confirmed. Alpha
  ships exactly this path. No JWT, no installation token flow.
- **SR4 (zero new Go dependency)**: confirmed. Locked-deps streak continues
  through M10 (M3 → M10, no new deps).
- **SR5 (issues + pull_request + ping subscription set)**: confirmed. No
  `issue_comment` for mention detection in M10-core.
- **SR6 (noise-filter sequence binding implementation order)**: confirmed and
  faithfully implemented. The handler pipeline follows SR6 exactly, with the
  plan's documented R1 reconciliation (rate cap before delivery-row insert to
  satisfy FR-602).
- **SR7 (separate port for trust isolation)**: port 8082, separate `ingress.Server`
  in the errgroup, no auth-middleware bypass on the dashboard mux.
- **SR8 (mention detection out of scope for M10-core)**: confirmed, not shipped.
- **SR9 (ping → 200, no ticket, no idempotency row)**: confirmed and tested.
- **SR10 (tickets.origin as unconstrained TEXT, no migration change)**: confirmed.
  `'ingress'` added as a new application-code value with no SQL CHECK-constraint
  change.

One spike finding that benefited most: F1's emphasis on verifying over the raw
body before any JSON parse. `TestVerifyGitHubSignature_RawBodyExact` pins this
property — a re-encoded body fails verification, as the spike predicted. Without
the spike this edge case could easily have been missed and surfaced as a production
bug with legitimate deliveries being rejected.

## Gotchas for the next milestone

1. **`ingress_deliveries` is append-only** (FR-203) — no pruning job ships in
   M10-core. If a later milestone adds pruning, it must respect GitHub's 3-day
   manual-redelivery window and never prune while that window is open.

2. **Ingress tickets count against M8 dept-weekly budgets** — the M8 gate
   counts ALL tickets in the window, now including ingress-created ones. Department
   budget tuning must account for the external webhook load. This is the same
   accounting note as M9 gotcha 2 (scheduled firings), now also applicable to
   ingress.

3. **`tickets.origin = 'ingress'` is unconstrained TEXT** (SR10) — no SQL CHECK
   constraint was added or changed. The dashboard Drizzle schema and `TicketCard`
   component recognize the value by string comparison; any new origin value added
   in a future milestone must update the origin-chip switch in `TicketCard.tsx`.

4. **Rate-cap bucket resets on supervisor restart** — the token bucket is
   process-local (plan decision 8). A burst that partially exhausted the bucket
   before a restart will see a full bucket after restart. Acceptable given the
   manual-redelivery model; noted for any future persistent rate-cap implementation.

5. **Vault secret path convention for future connectors** — the GitHub connector's
   webhook secret lives at `ingress/github/webhook_secret`. Future connectors
   should follow the pattern `ingress/<connector-id>/<credential-name>` for vault
   path consistency.

6. **The `Connector` interface is the extension seam** — a second connector
   (Stripe, email-in) adds one `.go` file implementing `Connector`, one config
   section in `internal/ingress/config.go`, and one route registration in
   `server.go`. The framework, handler pipeline, idempotency, rate cap, and
   ticket-insert machinery require no changes. This is the design payoff of the
   one-package, interface-first approach.
