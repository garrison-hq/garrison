## What shipped

### Schema + sqlc + drizzle (T001)

- `migrations/20260429000016_m5_2_chat_archive_and_cascade.sql` — adds `chat_sessions.is_archived BOOLEAN NOT NULL DEFAULT false` + the partial index `idx_chat_sessions_user_active_unarchived`, recreates the `chat_messages.session_id` FK with `ON DELETE CASCADE`, grants `DELETE ON chat_sessions` to `garrison_dashboard_app`. `vault_access_log` intentionally not touched (FR-236 — JSONB-deep references survive deletion as forensic trail).
- Three new sqlc queries: `FindOrphanedOperatorSessions :many` (subquery-based, not `JOIN LATERAL` — sqlc 1.29 doesn't analyse the LATERAL alias correctly), `InsertSyntheticAssistantAborted :exec`, `MarkSessionAborted :exec`. The existing M5.1 `AbortSessionsWithAbortedMessages` was left in place; the new `MarkSessionAborted` is the single-row-by-id companion.
- `dashboard/drizzle/schema.supervisor.ts` mirrors the migration shape (hand-edited; `bun run drizzle:pull` would have regenerated against a live DB). Both forms are byte-identical to what introspection produces.

### Supervisor extension (T002)

- `internal/chat/listener.go` `RunRestartSweep` extends with a second pass after the M5.1 pending-message pass. Detects orphan operator rows (newest message `role='operator'`, > 60s, no assistant pair), synthesises an aborted assistant row at `turn_index+1` with `error_kind='supervisor_restart'`, rolls the session to `aborted`. Per AGENTS.md concurrency rule 6 the synthetic terminal write runs under `context.WithoutCancel(ctx)` + `TerminalWriteGrace` so a SIGTERM mid-sweep still commits the invariant.
- Per FR-292 (resolved Q4) the orphan pass does NOT emit `pg_notify`. Dashboard sees the aborted state on next page load or SSE reconnect via the existing `/api/sse/chat` route's row-state-read mechanism. If post-week-of-use surfaces stuck loading states, M5.3 can layer in a notify emission — tracked here.
- Six listener-side tests + five store-side tests; both sets are integration-tagged (`//go:build integration`) and gated on a real testcontainer Postgres.

### Dashboard data layer (T003)

- Five new server actions in `lib/actions/chat.ts`: `createEmptyChatSession`, `endChatSession`, `archiveChatSession`, `unarchiveChatSession`, `deleteChatSession`. All four mutating actions share a `requireSessionOwner` ownership gate that collapses non-owner to `ChatError(SessionNotFound)` per plan §1.3 (no separate `NotOwner` kind exposed — enumeration-resistance, slate item 11).
- `getRecentThreadsForCurrentUser(limit?)` — auth-gated query wrapper used by the sidebar subnav. Returns rows with the per-operator `thread_number` computed via `ROW_NUMBER() OVER (PARTITION BY started_by_user_id ORDER BY started_at ASC)` per plan §1.11.
- `lib/queries/chat.ts` — `listSessionsForUser` gains a backwards-compatible `{archived?, limit?}` shape; `getSessionWithMessages` paginates with `{limit, beforeTurnIndex, hasMore}` per plan §1.8; `getMostRecentMempalaceCallAge` powers the composer's PalaceLiveChip per plan §1.10.

### Dashboard SSE consumer hook (T005)

- `lib/sse/chatStream.ts` exports `useChatStream(sessionId)` returning `{state, partialDeltas, terminals, sessionEnded, lastError}` per plan §1.5. State machine matches the M3 listener's vocabulary (`dormant/connecting/live/backoff/idle-grace`) but the implementation is independent — per plan §R1, two ~150-line files using the same state names is acceptable; refactoring to a shared core would force the M3 listener to absorb chat-specific concerns it doesn't have.
- Backoff: 100ms → 200 → 400 → … → 30000ms cap. Dedupe on `(messageId, seq)` keys. Partial buffer survives mid-stream disconnect; on reconnect the consumer relies on the SSE route's row-state-read mechanism per amended FR-261. Idle-grace (60s) on unmount keeps state cached for tab-switch / back-nav remounts.
- The state-machine logic is factored into a pure `ChatStreamMachine` object so unit tests don't need jsdom — the M3/M4 dashboard test stance explicitly excludes browser-DOM unit tests in favor of Playwright integration. Ten Vitest tests cover plan §1.5's eight named scenarios.

## What the spec got wrong

### `/speckit.analyze` round amended FR-261, FR-322

The pre-implementation `/speckit.analyze` round (operator-side) caught two FR drift cases:

- **FR-261 amendment** — original spec implied partial deltas are replayed on reconnect. Reality: the M5.1 SSE route doesn't replay deltas; it reads the row state on subscriber connect and emits any committed terminal it missed. The amended FR-261 names this mechanism explicitly so consumers don't accidentally clear-and-redraw on reconnect.
- **FR-322 scoping amendment** — original spec said "render every chat channel in EventRow"; M5.2 ships rendering only for `work.chat.session_deleted`. The other M5.1 chat channels (session_started, message_sent, session_ended) are deferred operator-tooling polish — landed if operator-week-of-use surfaces utility.

### T015's eleven sub-scenarios came out of the analyze round

The original tasks listed 7 sub-scenarios (a–g). The analyze round identified four more required scenarios for SC-208 / SC-221 / FR-244 race coverage: `i` (idle-pill flip after sweep), `j` (end-thread mid-stream race), `k` (long-thread TTI), and `h` (vault-leak grep moved out of `g`'s axe-core block). Task list was updated mid-analysis.

### sqlc 1.29 LATERAL alias bug

`FindOrphanedOperatorSessions` originally used `JOIN LATERAL (SELECT ... FROM chat_messages WHERE session_id = cs.id ORDER BY turn_index DESC LIMIT 1)` per plan §1.14. sqlc 1.29 reported "table alias does not exist" against the LATERAL join's outer alias. Worked around with a subquery in WHERE: `cm.turn_index = (SELECT MAX(turn_index) FROM chat_messages cmx WHERE cmx.session_id = cm.session_id)`. Functionally equivalent; the rewrite cost ~5 minutes and is self-explaining in the SQL comment.

### Vitest test environment is `node`, not `jsdom`

The plan §2.4 listed Vitest unit tests for Composer / MessageStream / ThreadHeader using `@testing-library/react` + simulated user-events. The dashboard's vitest config uses `environment: 'node'` and the project doesn't ship jsdom — the M3/M4 testing posture exclusively uses Playwright for component-level interactive tests. The tests landed instead as `renderToString`-based static rendering pins covering the SSR markup; the interactive scenarios shifted to Playwright sub-scenarios in T013–T015. Net assertion coverage is preserved; the partition between "static pin" and "interactive pin" just moved.

## Open questions deferred to the next milestone

### M5.3 scope confirmations

- Chat-driven mutations (assistant calls a `garrison-mutate` MCP server with confirmation gating). Threat-model amendment timing: M5.3 ships its own context doc with the amendment as a binding input, not the M5.2 retro.
- Whether the chat container's MCP config gains a third entry (`garrison-mutate`) is the M5.3 BuildChatConfig change. M5.2's defensive test `BuildChatConfig` is unchanged here.

### Operator-week-of-use observations

- Whether the orphan-sweep "no notify" decision (resolved Q4) holds. If operators report stuck loading states after a supervisor crash, M5.3 layers in a `work.chat.session_ended` emission with `status='aborted'`.
- Whether the FR-322 deferred operator-tooling polish (EventRow rendering for the M5.1 chat channels) becomes useful in practice. If yes, lands as a focused patch in M5.3 polish round.
- Whether the per-operator `thread_number` recomputation (no stored column per plan §1.11) becomes confusing when the operator deletes thread #5 of 10 — current behavior post-delete renumbers to 1..9. If post-week-of-use says "I expected thread #5 to be the slot that emptied," M5.4 polish round adds a stored `thread_number` column.

### Warm-pool optimization (post-M5)

Per the M5.1 retro §Open Q 4, each turn spawns a fresh container; cold-start adds measurable wall-clock to the first delta. The spike measured ~1.4s on warm cache (§8.1). M5.2 does not address this; the M5.2 retro tracks it explicitly so it doesn't drop off the radar.

### Post-M5 surfaced (not fixed)

- Cost-telemetry blind spot — still in `docs/issues/cost-telemetry-blind-spot.md`. M5.2 reads from `chat_messages.cost_usd` (per-turn) and `chat_sessions.total_cost_usd` (rolled), both of which are correct on chat (the M5.1 plan §"Cost rollup" verified this against the supervisor's chat path); the M3 caveat is about agent runs that signal-kill before result events land — chat doesn't hit that path.
- Workspace sandbox escape — still tracked at `docs/issues/agent-workspace-sandboxing.md`. Chat doesn't write to the workspace path; the issue is unaffected.
