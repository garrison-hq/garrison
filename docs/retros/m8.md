**M7 retro** (substrate): [`docs/retros/m7.md`](./m7.md)

M8 closes the event-driven zero-human loop. Four threads ship
together (mirrors M7's three-thread merge): (1) agent-callable
`create_ticket` with the agent-anchored audit row, auto-inherit
parent, and cycle/depth-cap dependency walker; (2) per-department
weekly ticket-creation budget as a runaway gate on the same write
path; (3) MCPJungle MCP-server registry sidecar + reactive worker
that picks up `mcp_servers` INSERT events and calls the admin API;
(4) per-agent bearer-token resolver wired through the M2.3 vault
fetcher.

## What the spec got wrong

### `register_mcp_server` audit-row anchoring (FR-306 evolution)

The original spec carried `register_mcp_server` as a chat-driven verb
with an audit row written at Server Action time. Implementation
revealed the two-rows-with-different-outcomes ambiguity: the row
written at SA-time would have to be `success` (the row insert
succeeded), but if MCPJungle rejected the registration the operator
sees a `success` audit + a `failed` `mcp_servers` row. The amendment:
move the canonical audit row write to the reactive worker
(`mcpserverwork.writeAudit`) anchored on the final outcome
(`success` | `failed`), and assert at the Server Action that no audit
row lands there (`TestRegisterDoesNotWriteAuditAtServerActionTime`).
The FR-306 single-row invariant is now enforceable.

### Worker-level dedupe

The plan implicitly assumed the M1 dispatcher's `inFlight` sync.Map
guarded against concurrent dispatch. That's true within a process,
but the LISTEN+poll path can still double-fire if the poller picks
up a row while a LISTEN handler is mid-flight on the same id. The
M8 chaos test caught it: two goroutines both saw `status='pending'`,
both called MCPJungle, both wrote success audit rows. Fix: add
optional `Pool` field to `mcpserverwork.Deps` and use
`SELECT FOR UPDATE NOWAIT` to hold the row lock across the MCPJungle
call. Documented in the worker file's `Handle` doc comment.

### Auto-inherit parent vs cross-dept create

`applyAutoInheritParent` initially resolved the agent's current
ticket as parent unconditionally. Cross-dept tests then failed
because `resolveParentTicketID` validates `parent.DepartmentID ==
target.DepartmentID` — a cross-dept agent caller would get
`validation_failed` (parent in different dept) before the gate even
ran. Fix: skip auto-inherit when the inherited parent's dept ≠
target dept; the `cross_dept_create=true` tag still lands via
`tagCrossDeptCreate`.

### Dept-weekly gate counts ALL tickets in the dept

The `dept_weekly_state` query counts every ticket created in the
last 7 days, including the agent-instance parent ticket. Tests that
seed `budget=N + fillN=N` get rejected because the parent ticket
adds one more. Tests had to add `+1` to the budget to account for
the parent. Worth noting in operator docs if budget tuning becomes a
support topic.

### `chat_mutation_audit.affected_resource_type` CHECK extension

The M5.3 CHECK only allowed `('ticket', 'agent_role',
'hiring_proposal')`. The M8 worker writes audit rows with
`affected_resource_type='mcp_server'`, which the CHECK rejected on
first integration-test run. Migration was extended to add
`'mcp_server'` to the allowlist (with the corresponding rollback in
the `+goose Down` block).

### `'failed'` outcome on `chat_mutation_audit`

The M5.3 outcome CHECK didn't include a generic `'failed'` outcome.
The worker writes audit rows with `outcome='failed'` when MCPJungle's
API rejects. The migration was extended to add `'failed'` (alongside
`'dept_weekly_ticket_budget_exceeded'` for the runaway gate).

## What worked

- **Reactive worker pattern**: the M1 dispatcher's envelope-only
  payload + the M8 trigger's `event_id := NEW.id` duplication keeps
  the dispatcher contract single-shape and the worker stateless.
- **Per-customer naming convention**: every McpClient + mcp_servers
  name carries the `<customer_slug>.` prefix from the schema level
  up through the dashboard form. Multi-tenant beta becomes a
  configuration change, not a refactor.
- **Sticking to the locked deps list**: zero dependency adds. M8's
  custom HTTP client to MCPJungle is ~200 lines of stdlib; the spike
  found MCPJungle's API simple enough that an SDK wasn't worth the
  weight.

---

## Was the spike worth it?

Yes. `docs/research/m8-mcpjungle-spike.md` answered three questions
the spec couldn't have: (1) per-McpClient AllowList semantics
(MCPJungle enforces server-name-level access, not method-level — so
the customer-prefix invariant is the right hierarchy); (2) admin-API
auth model (single bearer token at boot, no per-call OAuth); (3)
multi-tenant story (single-instance vs per-customer was a real
trade-off; Option A — per-customer instance — won for hard isolation,
with the `URLForCustomer` seam preserving the alpha shape).
