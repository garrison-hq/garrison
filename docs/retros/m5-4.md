## What the spec got wrong

### 1. Plan's "HTTP sidecar" wording for MemPalace

Plan §3 + §"Test strategy" referred to MemPalace as an HTTP sidecar (`httpClient`, `baseURL`, `httptest.Server` fakes). The m2-spike §3.6 + `Dockerfile.mempalace` + the existing `internal/mempalace.Client` all confirm MemPalace is stdio-only — there is no HTTP surface. The plan's open question §"Exact MemPalace sidecar API" anticipated this; T007's first sub-step (verify the sidecar API surface) caught it before any code was written.

Net result: data contract unchanged, transport differs from the plan's literal wording, no rework after the operator approved Path A. Cost: ~one round-trip with the operator at the start of T007 to surface the inconsistency.

### 2. T001 / T008 reality-adjustment around `garrison_supervisor` Postgres role

The plan + spec assumed a constrained `garrison_supervisor` role would gate the supervisor's reads of the better-auth `sessions` table; T001 found no such role in the codebase. The supervisor connects as the schema owner with full privileges. T001 was reality-adjusted to a no-op marker migration and T008's auth middleware uses direct pgx (mirroring `internal/health/server.go`) instead of sqlc-generated code (sessions is Drizzle-managed, not in the supervisor's sqlc schema list).

Net result: minus one migration, minus one sqlc query, minus one GRANT discipline gate. Auth boundary is still tight (read-only access to sessions; SELECT user_id, expires_at WHERE token = $1 only), just realised through schema-owner privileges rather than a constrained role.

### 3. T005's vault_access_log audit-row write

T005's task draft mentioned writing `vault_access_log` rows on the MinIO credential fetch path; the actual T005 commit + T011 wiring do not write audit rows for boot-time secret access. `vault_access_log` writes remain a spawn-site concern in `internal/spawn`. M5.4 boot creds are NOT recorded in the audit log.

Operator-known gap; flagged in `acceptance-evidence.md` § "Deferred / known gaps". Follow-up if audit needs a "startup-time secret access" trail — likely a one-line addition next to the Fetch call.

### 4. FR-666's "no compose-level healthcheck"

FR-666 said "the MinIO container does NOT have a Docker-level healthcheck baked into the compose service". T012 implemented one anyway (`mc ready local` every 10s with 10s start_period) for operator-side `docker compose ps` ergonomics. The supervisor's startup probe remains the actual readiness gate (no `depends_on: condition: service_healthy` from the supervisor — only `service_started`). Net-positive FR tighten; flagged here so future readers don't trust the FR's literal wording.

### 5. T009 task "9 named tests"

Plan §"Test strategy" enumerated 8 named objstore handler tests; T009's task said 9. T009 ships 9 by adding `TestObjstoreHandler_GetSurfacesAuthFailed` to cover the MinIOAuthFailed path on GET. Minor variance, no spec impact.

## Dependencies added outside the locked list

Per AGENTS.md soft-rule, two new dependency clusters joined the locked list at M5.4:

### Go: `github.com/minio/minio-go/v7`

- **What it does**: Official MinIO/S3 client SDK. Used by `internal/objstore` for `BucketExists`, `MakeBucket`, `GetObject`, `StatObject`, `PutObject` against the in-network `garrison-minio:9000` endpoint.
- **Why stdlib isn't sufficient**: implementing S3 signature v4 by hand via net/http would effectively reimplement the SDK without the ergonomics; signature v4 is non-trivial and a real auth-correctness surface.
- **Alternatives considered**: `aws-sdk-go-v2/s3` (rejected — heavier with more transitive deps; supports more S3-specific features M5.4 doesn't need; pointed at MinIO via endpoint override which adds config surface). `minio-go` is upstream-maintained by the same team that ships the server, follows MinIO server semantics exactly, and ships smaller.

### TypeScript: `@uiw/react-codemirror` + `@codemirror/lang-markdown`

- **What it does**: React wrapper around CodeMirror 6 plus the markdown language extension. Used by `CompanyMDEditor.tsx` to render the syntax-highlighted Company.md surface in both read and edit modes.
- **Why simpler isn't sufficient**: a textarea + Marked render forces the operator to context-swap between "edit" and "view"; same-component view+edit (toggled via `readOnly`) keeps the editor surface stable across mode flips and preserves cursor + scroll position.
- **Alternatives considered**: react-markdown for the read path + textarea for the edit path (rejected — two components to keep visually identical; any CSS/font drift is a regression surface). monaco-editor (rejected — significantly larger, designed for full IDE-grade editing; overkill for a single-document Markdown surface).

Locked-deps streak: M2.3 → M5.3 shipped with zero new dependencies (3 milestones). M5.4 breaks the streak with two principled additions, both load-bearing for the milestone. Future milestones should treat the post-M5.4 list as the new bar.

## Spike-vs-implement (this is a spike-first milestone)

Per RATIONALE §13, M5.4 ran a pre-spec MinIO spike (`docs/research/m5-4-spike-minio.md`). The validation question: did the spike pay off?

**Issues the spike prevented** (would have surfaced during /garrison-implement otherwise):

1. **Image digest pinning** (spike §F1) — the spike captured the live digest; T012 + integration_test.go reference it as the single source of truth.
2. **testcontainers-go has no built-in MinIO module** (spike §F2) — the integration test uses the generic-container API instead of the typical module-scoped helper. Saves one round-trip mid-implement.
3. **Named-volume vs bind-mount choice** (spike §F3) — spike confirmed named-volume preserves bucket state across `down + up` and bind-mount is unnecessary; operator constraint matched the technical fit.
4. **Scoped service-account credential split** (spike §F6) — the `mc admin user svcacct add` invocation form was characterised in the spike and copied verbatim into ops-checklist.md.
5. **ETag-based optimistic concurrency** (spike §F7) — `If-Match` semantics + StatObject pre-check pattern characterised; T005's PutCompanyMD implementation followed the spike pattern directly.
6. **BucketExists+MakeBucket bootstrap shape** (spike §F5) — idempotent, single-call recovery; matches the M2.2 mempalace-bootstrap pattern T011 mirrors.

**Surprises that escaped to /garrison-implement**:

1. The plan's "HTTP sidecar" framing for MemPalace (caught at T007's verify-first step; not a spike issue but a plan/spec drift the spike couldn't have prevented since the MinIO spike scoped to MinIO).
2. T001's missing `garrison_supervisor` role discovered via codebase grep, not via the spike (out of spike scope).

**Net**: 6 issues prevented, 0 spike-scoped surprises that escaped. Prevention rate above the informal 50% RATIONALE §13 target. Spike-first discipline empirically validated for the third time (M2.1 spike was first; M2.3 vault threat-model-first was the second related precedent).
