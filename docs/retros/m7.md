## What the spec got wrong

1. **`chat_mutation_audit.chat_session_id` + `chat_message_id` NOT
   NULL.** Spec implicitly assumed Server-Action audit rows could land
   in `chat_mutation_audit` without a chat anchor, but the M5.3 schema
   declared both columns NOT NULL. Caught during T010 implementation.
   Fix: amended the M7 migration's up section to drop NOT NULL on both
   columns + extended the down section to delete M7-era audit rows
   before restoring NOT NULL (the verb CHECK rollback would otherwise
   block on non-M5.3 verb values too). Lesson: when introducing a
   non-chat audit-row source, schema check the existing audit table's
   NULL-ability up-front.
2. **Spec FR-214a places install-step rows in `chat_mutation_audit`.**
   The chat audit's `verb` column wasn't designed for non-verb step
   identifiers; bending it would either re-purpose the column or
   require a side-channel column. Resolved during T001 with
   `agent_install_journal` as a dedicated table; spec intent
   preserved.
3. **`registry.go` referenced in T009's Files list doesn't exist.**
   The garrisonmutate package's verb registry lives in `verbs.go`
   (the `Verbs` slice). T009 modified `verbs.go` per the actual
   surface; the Files list in tasks.md was a typo carried over from
   an earlier plan draft.
4. **T015 vitest tests omitted per operator memory rule.** The
   operator's "Tests for Go only, never frontend" memory rule
   (recorded pre-M7) supersedes the tasks.md vitest target. The
   Go-side T010 + T017 integration tests cover the data shape; the
   dashboard surface relies on shape correctness rather than per-
   component vitest.

## Gotchas worth remembering for M8

1. **chat_mutation_audit chat_session_id/chat_message_id NULL after
   M7.** Server-Action verbs (approve_hire, reject_hire, etc.) commit
   audit rows with both columns NULL. Future audit-row consumers
   should treat these columns as nullable from M7 onwards.
2. **agent_install_journal exists; chat_mutation_audit doesn't carry
   install-step rows.** When a future milestone reads install
   forensics, the journal is the canonical surface — not a verb=
   'install_step:*' query against chat_mutation_audit.
3. **Per-agent container name convention `garrison-agent-<role-slug>`.**
   Both spawn (T011) and migrate7 (T014) address containers by this
   name; future code that interacts with the agent runtime should
   use the same convention rather than maintaining a parallel
   agent-id → container-id lookup table.
4. **`UseDirectExec=true` is the default until the operator flips.**
   Any test or integration that depends on the container path running
   must set UseDirectExec=false explicitly; otherwise spawn flows
   through the legacy exec.CommandContext branch.
5. **The preamble is unconditional.** ComposeSystemPrompt prepends
   it whether or not the wake-up succeeded. Future milestones that
   bypass the wake-up RPC (e.g. for performance or for chat-runtime
   parity) must still carry the preamble; bypassing it widens the
   operator-trust posture.
6. **Rules hold for both threat models.** M2.3 vault Rule 1 (secrets
   never in prompts) carries unchanged; vault env-injection happens
   at container-start time in M7, not at direct-exec time. The
   secret-leak scan in `internal/leakscan` still gates agent.md edits
   the same way.
7. **Diary-vs-reality verifier hits live in the hygiene listener.**
   The full path-stat walk that produces hygiene_status='missing_
   artefact' (FR-210) is the M2.2 hygiene listener's job; M7 records
   the artefact paths on ticket_transitions but the verifier loop
   itself is M2.2 territory.
