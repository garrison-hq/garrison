**Shipped**: 2026-06-12 (implementation phase — commits T001…T015 on

**Plan**: [`specs/023-m11-action-broker/plan.md`](../../specs/023-m11-action-broker/plan.md)

**M10 retro** (substrate): [`docs/retros/m10.md`](./m10.md)
**M9 retro** (substrate): [`docs/retros/m9.md`](./m9.md)

M11 closes the outbound half of the company's external interface. With
M10 making the company reactive to the outside world (inbound), M11
gives agents a requesting path — never a direct-acting one — through a
single audited, policy-gated broker. Agents can now express an intent
to act on the world (reply to a GitHub issue, for example); the broker
turns it into a governed, immutable, operator-gated request. Four
threads shipped together: (1) the sealed `request_external_action` verb
as the 12th verb in the registry; (2) the tier policy table with the
permanent-Approve floor; (3) the supervisor-side dispatcher worker
executing approved/auto/notify actions with vault-scoped credentials;
(4) the `/admin/outbox` dashboard approval surface. GitHub
comment-back is the first concrete action type riding all four. Zero
new Go or TypeScript dependencies.

## What the spec/plan got wrong

### `ClaimFn` test seam — unforeseen production-file addition

The spec and plan described `TestHandleNeverExecutesHumanOnly` as a
defence-in-depth test asserting the in-Handle `human_only` guard fires
even when a `human_only` row reaches `Handle` directly. The plan
implicitly assumed this could be tested by inserting a `human_only` row
into a real Postgres instance and calling `Handle` against it. In
practice, the production `ClaimDispatchablePendingAction` query already
filters `tier <> 'human_only'` at the SQL level, so such a row is never
claimed through the normal LISTEN path — `Handle` would simply find
nothing to claim and return a no-op, leaving the `skipped_human_only`
outcome branch untested. The only mechanism to exercise the in-Handle
guard was to inject the row directly into `Handle` via an exported
`ClaimFn` field on `Deps`, bypassing the claim query entirely. This
field was added to `dispatcher.go` under the T009 seam note as an
authorised production-file addition, documented "Never set in production
code". The spec's implicit testing model was wrong; the implementation
surface (the defence-in-depth guard) was correct.

### `RegisterTestPolicyEntry` test seam — unforeseen production-file addition

The spec and plan described `TestRequestExternalActionAutoTierEmitsDispatchNotify`
as testing that the verb emits a `work.action.dispatch_requested` notify
for an `auto`-tier action type. The plan assumed this could be achieved
via `export_test.go` in the `actionbroker` package, which is compiled
only into that package's own test binary. Because the test lives in
`garrisonmutate` (an external package that imports `actionbroker`), the
`export_test.go` mechanism does not reach it. The only mechanism that
lets `garrisonmutate`'s tests inject a `policy` entry is an exported
function in a production file. `RegisterTestPolicyEntry` was therefore
added to `policy.go` under the T008 seam note as an authorised
production-file addition, documented "Never set in production code". The
spec's implicit cross-package testing model was wrong; the production
`Classify` logic was correct.

### Nothing else material

Aside from the two test-seam additions above, the spec and plan held. The
Phase 0 GitHub contract assumptions matched reality. The three-layer floor
enforcement (Classify floor-first, struct-level silence, DB CHECK) was
correctly anticipated. The `FOR UPDATE SKIP LOCKED` exactly-once contract
transferred cleanly from M9/M10 with no surprises. No features were
dropped, no scope was cut, and no production behaviour diverged from the
spec's intent.

## The context file's named retro questions

### (a) Did the broker hold as the only door — did anything reach the outside world not through the dispatcher?

Yes, asserted by script. SC-004 in `scripts/m11-acceptance.sh` runs:

```
grep -rn 'GITHUB_PAT' supervisor/internal/agents/ \
    supervisor/internal/claudeproto/ supervisor/internal/finalize/
```

and asserts empty output. No outbound call, no GitHub credential, no
external mutation exists anywhere in agent-container code paths. The
dispatcher is supervisor-side; agent containers remain on the
`internal: true` `garrison-agents` network with `api.anthropic.com` as
the only egress. A secondary check asserts no `squid.conf` reference in
`internal/actionbroker/` or `main.go` (the dispatcher does not route
through the agent network).

### (b) Did the permanent-Approve floor resist a lower-the-tier attempt?

Yes, on all three enforcement layers.

**Layer (a) — `Classify` floor-first**: `TestFloorCannotBeLowered`
constructs `policy["github_issue_comment"] = TierAuto` and asserts
`Classify` still returns `(TierApprove, "permanent-Approve floor
(public-facing)")` — the `floor` map is consulted before the `policy`
map.

**Layer (b) — struct-level silence**: `TestRequestExternalActionIgnoresAgentSuppliedTier`
posts args JSON containing `"tier":"auto"`; the `RequestExternalActionArgs`
struct has no `tier` field so the key is silently dropped on unmarshal.
The stored tier is `approve` from the policy table.

**Layer (c) — DB CHECK**: `TestFloorEnforcedAtDB` attempts to INSERT a
`pending_actions` row directly with `action_type='github_issue_comment'`
and `tier='auto'`; the `pending_actions_floor_is_approve` CHECK fires and
the insert is rejected.

`TestFloorCheckMatchesPolicy` asserts the migration's CHECK action-type
list equals `floor`'s keys exactly — no drift between the two enforcement
surfaces.

### (c) Did dispatch stay exactly-once under restart/race?

Yes. `TestConcurrentClaimDispatchesExactlyOnce` drove N goroutines from a
start barrier against the same `approved` `approve`-tier row. The `FOR
UPDATE SKIP LOCKED` claim yielded exactly one `PostComment` call and one
`executed` outcome row; all subsequent claimants found the row in a terminal
state and were no-ops.

`TestRestartMidDispatchNoDoublePost` simulated a crash after claim+POST but
before commit (the tx rolls back, the row stays `approved`). A second
`Handle` call then ran. The test documents the at-most-once-extra window
only when the first attempt's commit was lost — the standard exactly-once-
on-success contract matching the M9/M10 ships. A committed `executed` row
makes the re-claim a no-op; the final state is exactly one `executed`
outcome.

### (d) Did the dispatcher's credentials stay out of every agent context?

Yes, on two axes.

**Vault fail-closed**: `TestHandleVaultUnavailableFailsClosed` confirmed
that when `Vault.Fetch` returns an error, the dispatcher records
`status='failed'` + `outcome='failed'` with `detail="vault unavailable"`
and does NOT call `PostComment`. No unscoped fallback is attempted.

**PAT never logged**: `TestPostCommentNeverLogsPAT` captured the slog
output during a `PostComment` call and confirmed the PAT string never
appeared. The PAT is passed as a `string` parameter only to the HTTP
`Authorization` header and is never stored on the `GitHubPoster` struct.
SC-004 grep confirmed no `GITHUB_PAT` reference in any agent-container
code path (`internal/agents/`, `internal/claudeproto/`, `internal/finalize/`).

### (e) Did the threat model (`action-broker-threat-model.md`) precede the first dispatcher code in git history?

Yes, asserted by script. SC-008 in `scripts/m11-acceptance.sh` ran:

```
git log --format='%H' -n1 -- docs/security/action-broker-threat-model.md
git log --format='%H' -- supervisor/internal/actionbroker/ | tail -1
```

`git merge-base --is-ancestor` confirmed the threat-model commit (`5e6061d`,
T001) is an ancestor of the earliest `internal/actionbroker/` commit
(`73cad10`, T004 — the policy module). T002 (migration) and T003 (config)
landed between the threat model and the first dispatcher code, which is
correct — they are schema and config, not dispatcher code.

## Did the GitHub comment-create contract match Phase 0's assumptions?

Yes. The plan's Phase 0 record described `POST /repos/{owner}/{repo}/issues/{issue_number}/comments`,
`Authorization: token <PAT>`, `Accept: application/vnd.github+json`, body
`{"body": "<comment text>"}`, `201` → `html_url`, `403`/`429`/`5xx` →
recoverable, `404`/`422` → terminal. The `httptest.Server` in
`TestPostCommentBuildsCorrectRequest` verified all of these properties.
No spike was needed; the M10 GitHub spike's PAT auth findings (F5) were
accurate for comment-create.

The one nuance: GitHub comment-create is not natively idempotent (no
idempotency key on their side). The spec flagged this; the exact-once
guarantee is entirely Garrison-side via the claim + terminal-status
transition. This is why auto-retry (FR-022) is forbidden — a retry of
an in-doubt POST could double-post. The test harness documented the
at-most-once-extra window in `TestRestartMidDispatchNoDoublePost`.

## Were there surprises in tier enforcement or the exactly-once claim?

One structural observation worth recording: `TestHandleNeverExecutesHumanOnly`
(T009) required the `ClaimFn` test seam to be added to `dispatcher.go`.
The production `ClaimDispatchablePendingAction` query already filters
`tier <> 'human_only'`, so a `human_only` row can never be returned through
the normal LISTEN path — meaning a test that simply calls `Handle` against a
real DB row with `tier='human_only'` returns a no-op without asserting the
`skipped_human_only` outcome. The defence-in-depth path (the in-Handle
`human_only` guard) is dead code under normal operation. The `ClaimFn`
injection was the only mechanism to exercise it. This mirrors the
`RegisterTestPolicyEntry` seam in `policy.go` for the auto-tier dispatch-
notify test in T008 — both are exported production-file functions with
"Never set in production code" documentation, added under explicit task
authority.

The concurrent-claim test (T010) exercised the `FOR UPDATE SKIP LOCKED`
mechanics cleanly with no unexpected behaviour. The M1/M9/M10 discipline
translated directly.

## Gotchas for the next milestone

1. **`pending_actions` is the only door** — any future action provider
   that wants to reach the outside world must go through `InsertPendingAction`
   + the dispatcher. A new action type adds a `policy` map entry + the
   `pending_actions_floor_is_approve` CHECK extension (if it is public-
   facing) + a `github.go`-parallel poster; it does not add a separate
   outbound channel.

2. **Three-layer floor enforcement must stay consistent** — the `floor`
   map in `policy.go`, the `pending_actions_floor_is_approve` CHECK in
   the migration, and `TestFloorCheckMatchesPolicy` are a trio. Adding a
   new public-facing action type requires updating all three together; the
   `TestFloorCheckMatchesPolicy` test will fail if the CHECK and the map
   diverge.

3. **`ClaimFn` and `RegisterTestPolicyEntry` are test seams** — both are
   exported production-file symbols with "Never set in production code"
   documentation. They exist to exercise defence-in-depth paths that normal
   operation cannot reach. Future milestones adding test seams should follow
   the same "exported production-file function, documented, task-authorised"
   discipline rather than unexported `export_test.go` hacks (which don't
   work across package boundaries).

4. **`approved_by` is NULL for non-`approve`-tier actions** — the
   `approved_by` column is NULL for `auto`/`notify`/`human_only` rows (only
   set by the `approveAction` Server Action). Audit reconstruction for non-
   approve tiers should not expect a non-NULL `approved_by`.

5. **Drizzle empty-default mangling — fourth consecutive milestone** —
   despite the gotcha being noted in M7, M8, M9, and M10 retros and the
   T002 completion condition explicitly including the inline-fix step,
   Drizzle pull still requires a manual fixup after regeneration when new
   columns with default values appear. Applied and confirmed clean. This is
   now the standing known step.

6. **`actions/GITHUB_PAT` must be provisioned in Infisical before the
   dispatcher is useful** — a missing PAT causes per-row dispatch failure,
   not a startup abort (the fail-closed posture). Operators enabling the
   dispatcher without the secret will see `status='failed'` rows in the
   Outbox. The ops-checklist M11 section names this step.
