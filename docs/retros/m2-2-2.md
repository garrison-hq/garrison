## What the spec got wrong

1. **FR-317 named the wrong test file.** Spec said new unit tests
   land in `internal/finalize/tool_test.go`; plan + tasks correctly
   placed them in a new `richer_error_test.go`. Caught by
   `/speckit-analyze` as finding F1 (MEDIUM); remediated by amending
   FR-317 + the sibling Adjudicate-test file path before T001. Lesson:
   when the plan corrects an obvious spec slip, fold the correction
   into the spec rather than letting the artifacts disagree — the
   analyze pass caught this on the first cycle.

2. **T007's "transient failing test" carve-out was wrong scope-
   discipline.** Tasks T007 and T008 were specified as separate
   commits with the explicit note that T007 would leave the seed-test
   in a transient failure for the qa-engineer.md path. In practice,
   the test helper naturally covers both files in one code path, so
   splitting created a dishonest commit boundary. Combined into one
   commit during execution per the skill's "unless explicitly small
   enough to combine" exception. Lesson: a task that explicitly
   accepts a transient-failing repository state is asking the wrong
   question; the right question is whether the two tasks should be
   one task.

3. **The compliance thesis assumed prompt-only calibration would
   move models toward autonomous compliance on trivial tickets.**
   This was the central thesis behind FR-308 through FR-312. Live
   data shows it doesn't, on either model class. The spec wasn't
   "wrong" — it correctly named the thesis as "under test" and gave
   the honest-ship-on-partial-pass clause — but the optimism baked
   into the priority-1 status of US4 / US5 (haiku produces well-
   formed first attempt; opus reaches finalize within budget) was
   overconfident. Future spec phrasing should hedge calibration-
   thesis user stories as "we hope this happens; the live matrix
   tells us if it did" rather than asserting the desired outcome.

4. **Hint copy was implementer-iterable but never iterated.** Plan
   §"Open questions" called out hint phrasing as iteration-prone;
   the live matrix produced no successful retry, so the hint copy
   shipped at v1. We don't know if better hint wording would have
   helped haiku because we never observed haiku reading a hint and
   acting on it. M3-era spike candidate: instrument the hint
   delivery path to capture what the agent does after receiving
   each hint variant.

## Open questions deferred to the next milestone

The live-matrix evidence (acceptance-evidence.md §"What didn't work")
implies a small set of next-milestone candidates. None are pre-
committed; the operator decides scope.

1. **Flexible-first-attempt schema mode.** M2.2.2 explicitly dropped
   this (context §"What M2.2.2 does not produce" item 1). The live
   evidence — haiku making 1-2 attempts then giving up rather than
   reading hints — strongly suggests the schema's strictness is the
   compliance ceiling, not the error format. Worth reconsidering for
   M2.2.3 / M3.

2. **Supervisor-enforced palace-search budget.** Decision 3 deferred
   per-call enforcement to M3 specifically because "the agent itself
   is the right place to make this judgment." The opus pattern (3/3
   runs hit budget cap before reaching finalize) suggests this needs
   supervisor enforcement, not prompt guidance. Concrete next-spike
   candidate: "implement a turn-counter that hard-caps palace tool
   calls at N=3 per spawn."

3. **Exemplar-driven prompt** vs angle-bracket-placeholder prompt.
   The Q3 placeholder convention (`<ticket_id>`, `<your one-line
   outcome>`) may be too abstract for haiku. An alternative:
   render the prompt with concrete-but-disposable values (literal
   UUIDs, literal prose) and a sentence telling the agent "these
   are EXAMPLES, replace them." Live matrix would tell us which
   convention produces fewer malformed first attempts.

4. **Tooling-side verification before subprocess exit.** Currently
   when claude emits a `result` event without a prior successful
   `finalize_ticket`, the supervisor classifies as
   `finalize_never_called` and exits. An alternative: detect this
   condition and emit a synthetic "you exited without finalizing
   — call finalize_ticket now or accept failure" tool_result back
   to claude before letting the result event terminate the run.
   Speculative; would require claude protocol verification.

5. **Hint-feedback instrumentation.** The current architecture
   emits hints into tool_result envelopes but doesn't track which
   hints the agent reads or acts on. Adding telemetry would give
   future calibration milestones empirical signal instead of "did
   the hygiene status come out clean."

6. **`hall_discoveries` searchability fix.** Deferred from M2.2.1;
   still deferred. M2.2.2's QA-specific calibration intentionally
   doesn't depend on hall_discoveries because the searchability
   issue is unresolved.

## Compliance-thesis validation (the load-bearing finding)

**Did the M2.2.2 calibration produce reliable compliance on
haiku-4-5-20251001 AND opus-4-7?**

**No.** Both models scored 0/3 clean runs against the trivial
hello-world ticket. SC-311's "≥ 2/3 clean per model" bar was missed
by both. M2.2.1's ship retro warned this was a possibility; the
matrix confirmed it.

The data, in detail:

- haiku × 3 runs: each made 1-2 `finalize_ticket` attempts with
  `field=""` (decode-shape rejections), then emitted a `result`
  event with `subtype="success"` claiming the run was complete.
  Adjudicate (correctly) classified as `finalize_never_called`. Mean
  cost ~$0.047/run, ~6 minutes wall-clock per run.
- opus × 3 runs: each exhausted the $0.20 per-spawn budget cap
  before reaching `finalize_ticket`. Spent the budget on palace
  exploration despite the new "Skip if trivial" calibration bullet.
  Adjudicate's M2.2.2 fix correctly classified as
  `budget_exceeded` (M2.2.1 would have classified as
  `claude_error`). Mean cost ~$0.215/run.

This is not a system failure — the supervisor + finalize MCP server
+ atomic-write pipeline behaved exactly as designed. It is a
prompt-only-calibration ceiling: the M2.2.2 thesis (that prompt
changes alone could move autonomous compliance) does not hold at
the trivial-ticket level for either model class.

What survives M2.2.2:

- The Adjudicate precedence fix permanently improves operator
  observability around budget-cap events.
- The richer-error infrastructure (failure/line/column/excerpt/
  constraint/expected/actual/hint) is correct, fully tested, and
  ready for future milestones to actually iterate on hint copy
  given empirical signal.
- The mockclaude-based regression coverage (TestM222Retry... +
  TestM222ThreeRetries...) keeps the retry path covered through
  the new error shape.
- The seed-rewrite infrastructure (front-loading, example, role-
  specific calibration) ships in production prompts; even if it
  doesn't move the autonomous-compliance needle today, it doesn't
  regress anything either, and may benefit future model versions.

## Manual palace observations (per Clarification Q4)

No M2.2.2-iteration `finalize_ticket` succeeded, so no diary
drawers were written by either agent. The `wing_frontend_engineer`
palace contained ~30 prior-test drawers (M2.2.1 integration runs);
haiku iter 2 issued one `mempalace_search` call that returned
those at low similarity (~0.3). No new palace state was created
by the M2.2.2 matrix. The QA-specific calibration in qa-engineer.md
was never exercised because no engineer turn produced a transition
to qa_review.

This is itself a finding: when a trivial ticket isn't being
completed, the palace stays empty for that ticket; the existing
`hall_events` searchability gap (M2.2.1 retro deferred) persists.
