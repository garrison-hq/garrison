## What the plan got wrong (and what we learned from running it)

1. **os/exec pipe-drain race was implicit in the plan.** The real-
   Claude path ran `cmd.Wait` concurrently with `pipeline.Run`
   (reading stdout). Go's StdoutPipe docs explicitly warn: "It is
   thus incorrect to call Wait before all reads from the pipe have
   completed." On long happy-path streams the pipe buffer held all
   events before Wait landed; on shorter failure-path streams the
   race truncated the scanner mid-read with "file already closed"
   and the supervisor recorded `no_result` instead of the correct
   terminal. T016 caught it (TestM21SpawnFailedOnConfigWriteError's
   second ticket). Fix: drain the pipeline to completion *before*
   calling cmd.Wait — select on a pipelineDone channel, then reap.

2. **Mockclaude SIGTERM handling inline in the scanner loop missed
   signals during `#sleep`.** `time.Sleep` is not signal-
   interruptible in Go, so a SIGTERM arriving during a scripted
   sleep was delivered to the signal channel but never drained
   until the scanner woke up — too late to write the marker file
   before WaitDelay escalated to SIGKILL. T017's graceful-shutdown
   test exposed it (mock SIGTERM marker missing, supervisor logged
   "one or more subprocesses required SIGKILL"). Fix: dedicated
   goroutine handling the signal and calling `os.Exit(143)` on
   receipt.

3. **`testdb.Start`'s truncate-only-on-cleanup was order-dependent.**
   The per-test `t.Cleanup(TRUNCATE)` means the first test sees the
   migration seed; subsequent tests see a clean slate. T014's
   channel-rename fix made every M21 test try to insert a row into
   slug=`engineering`, but the seed's `engineering` department was
   still there on first boot — the unique constraint fired. Fix:
   TRUNCATE eagerly at Start too (with CASCADE through the M2.1
   FK graph) so every test starts identically clean.

4. **The `garrison_agent_ro` role has no default password post-
   migration.** The migration ships `CREATE ROLE garrison_agent_ro
   LOGIN;` but no password — operators are expected to run
   `ALTER ROLE … PASSWORD` post-migrate. T018 found this hard
   way: without the ALTER, pgmcp can't authenticate, so the
   chaos test degenerates into a trivial MCP-fail path. Fix:
   `testdb.SetAgentROPassword(t, password)` helper; ops docs will
   need to document this for release (tracked as an M2.2 item).

5. **`cmd.WaitDelay` semantics were used twice without realising
   the interaction.** We set `cmd.Cancel = killProcessGroup(SIGTERM)`
   and `cmd.WaitDelay = 5s`. Per Go docs, after Cancel fires,
   WaitDelay is the grace before pipes are forcibly closed and
   the subprocess killed outright. This is what we wanted, but the
   docs are tricky — it took a careful re-read to confirm the
   escalation does the right thing for the timeout path. Nothing
   to fix; retro-noting the subtlety for whoever touches spawn.go
   next.

6. **T015's cost assertion almost needed float comparison.** The
   mockclaude result event carries `"total_cost_usd":0.003`;
   round-tripping through `json.Number` → `pgtype.Numeric(10,6)`
   → `::text` yields `"0.003000"`. A naive string equality check
   rejected this. Fix: `normaliseCost` helper that trims trailing
   zeros. Cleanly generalises to the real-claude cost
   `"0.040882749999999995"` observed during T018 (which we compare
   numerically, not textually).

## Plan-level open questions — resolved

All resolved via clarify-session or during implementation:

1. **Which Postgres MCP server?** → **In-tree Go** (plan decision,
   spec clarify Q1). `internal/pgmcp` is ~300 LOC plus tests.
2. **How is read-only enforced?** → **Role-based + protocol filter**
   (clarify Q2). Primary guarantee is `GRANT SELECT` on the M2.1
   read surface; `internal/pgmcp/auth.go` is the second-layer
   filter.
3. **Subprocess timeout default?** → **60 seconds** (M2.1 spec
   default; configurable via `GARRISON_SUBPROCESS_TIMEOUT`).
4. **What happens to cost on a no_result exit?** → **NULL**
   (clarify Q3). Only a parsed `result` event produces a cost;
   any earlier terminal writes cost=NULL so the aggregate query
   stays honest about what claude actually billed.
5. **What happens when mcpconfig.Write fails?** → **spawn_failed
   terminal; dispatcher continues** (clarify Q2). The failure is
   legible on the agent_instances row; the supervisor does not
   abort the loop.

## Spike pay-off audit (retro-mandated)

From the M2 spike (`docs/research/m2-spike.md`):

**(a) Did the spike pay off?** Yes, decisively. The spike pinned:
- claude 2.1.117 as the working version (T004 uses it; observed
  identical behaviour in both mock and real runs).
- stream-json event vocabulary (T006 shipped the exact type
  inventory from spike §2.1).
- MCP init-event shape with `mcp_servers[].status` (T006 fail-
  closed health check verified against T018's real-claude run).
- json.Number decoding for `total_cost_usd` (T006 `UseNumber`
  preserved both the mock's "0.003" and the real "0.0408…").
- Subprocess-timeout semantics = from cmd.Start (clarify Q1
  aligned with how `exec.CommandContext` already behaves).

**Issues prevented vs. discovered ratio**: 5 prevented (above) vs.
1 genuinely surprising discovery (§1 in "What the plan got wrong").
Plan pay-off well above the ≥ 50 % validation-rate target.

**(b) Did init-event parsing hold up under real load?** Yes. The
shape matched on the first real-claude run. One empirical
refinement: `rate_limit_info.status` on real claude emitted
`"allowed_warning"` rather than the spike's `"warn"` — observational
only, the supervisor logs whatever value arrives. Struct type still
just `string`.

**(c) Did the in-tree Go Postgres MCP server hold up?** Yes. No
crashes across T008 unit + integration runs, no T015–T018 test
regressions. The role-based read-only grant is the primary guarantee;
the protocol-layer filter caught no real bypasses in testing
(belt-and-braces).

**(d) Did `total_cost_usd` match Anthropic's billing dashboard?**
Not cross-checked in this session (single-run $0.0409 observation).
Worth doing pre-ship when cost dashboard access is available;
adding to the M2.2 "pre-launch ops check" list.

**(e) Did process-group termination work cleanly?** Yes. Three pieces
of evidence:
- T011 unit test spawns a shell that backgrounds a `sleep 30`,
  calls `killProcessGroup(SIGTERM)`, confirms both processes die
  within 1 s.
- T017 TestM21GracefulShutdownWithInflight asserts the mock's
  SIGTERM marker appears (proves the group was signalled, not
  just the top-level PID).
- T018 TestM21ChaosClaudeSigkilledExternally asserts zero running
  rows after the kill settles.
No zombie-process observations during any test run. No grace-period
edge cases hit (the only grace-adjacent concern was the mockclaude
signal-handling fix, which wasn't about the supervisor-side grace
at all).

**(f) Real-claude surprises the spike missed:** two notable ones:
- **Per-run cost ≈13× the spike's fixture**: $0.0409 observed vs.
  $0.003 in the mockclaude fixture. The fixture is a "minimum
  representative" value, not a real-load prediction; the retro
  notes that the spike's cost estimate was inherited from that
  minimum and reality is materially higher. SC-107's $0.50 soft
  gate for 10 runs still holds (10 × $0.04 = $0.40) but the
  headroom is thinner than the spike implied.
- **Claude emits `thinking`/`text`/`tool_use` interleavings** in
  the same assistant event in a pattern the mockclaude fixture
  doesn't reproduce (each event carrying one content type). The
  supervisor handles this correctly — `assistant_seen` + content-
  block-count logging — but every real-claude run emits 5–10
  assistant events vs. the mock's 2. Observational; no code
  change needed.
