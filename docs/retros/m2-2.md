## What the plan got wrong (and what we learned)

### T001 surfaced six implementation-mechanism findings

The pre-implementation validation spike surfaced six deployment-shape findings that invalidated plan-phase assumptions. Each is documented in `specs/004-m2-2-mempalace/research.md` and absorbed into the spec + plan + tasks via surgical edits committed in T001's own commit. Summary:

- **F1** — `mempalace init --yes` is idempotent in 3.3.2; the `chroma.sqlite3` marker-file heuristic Session 2026-04-22 Q4 committed to is wrong (`chroma.sqlite3` appears post-first-write, not post-init). Bootstrap simplified to unconditional init.
- **F2** — `mempalace wake-up` in 3.3.2 takes only `--wing`; `--palace` is a top-level flag; `--max-tokens` doesn't exist. FR-207 + NFR-209 wording corrected; NFR-209 deleted.
- **F3** — `python:3.11-slim` (debian-glibc) is the only viable base for Dockerfile.mempalace; alpine fails because chromadb's Rust-compiled transitives have no musl wheels.
- **F4** — `pip install --require-hashes` requires ALL transitive deps pinned with `==`. Fixed via `pip-compile`-generated `requirements-lock.txt`.
- **F5** — `linuxserver/socket-proxy` listens on TCP :2375 via HAProxy, not a unix socket. `DOCKER_HOST=tcp://garrison-docker-proxy:2375`, not a unix-socket path. Dropped the `docker-proxy-sock` shared named volume entirely.
- **F6** — `python:3.11-slim` has no `ps`/`pgrep`; installed `procps` in Dockerfile.mempalace for human operator convenience; test code uses `/proc/[0-9]*/cmdline` for tool-agnostic robustness.

The plan's `/speckit.analyze` pass (before T001) missed all six because they're empirical facts about external tools, not cross-artefact consistency issues. The spike is the right place for this. Spike pay-off observation below.

### `emit_ticket_transitioned` payload missing `department_id`

During T017 execution, discovered that the trigger I wrote in T003 omitted `department_id` from the NOTIFY payload. spawn.Spawn's payload decoder expects both `ticket_id` + `department_id` — so parseUUID on an empty string failed, the spawn errored silently, the event was never marked processed, and the dispatcher kept re-invoking the qa-engineer handler every 5s (polling cadence).

Fix: the trigger resolves `department_id` via a `SELECT ... FROM tickets WHERE id = NEW.ticket_id` subquery and includes it in the jsonb_build_object. Committed in T017 alongside the test that exposed it. Retro note: the unit tests for the pure Go code caught nothing here because the missing payload field was an invariant the tests didn't assert; the test was exactly the right place to find it.

### T013 widened spawn-test scope later than ideal

T013's test file (`internal/spawn/spawn_test.go`) originally had only 5 tests covering the pure helpers (transitionColumns, ComposeSystemPrompt). The integration-level assertions (wake-up status flows to the row; budget flag passes correctly through argv; role slug dispatches correctly) were all deferred to T017's integration test. That's fine in hindsight — T017 exercises them end-to-end against the real sidecar — but "how much of spawn.go can we unit-test vs. where does integration scope begin" was a judgment call that the plan didn't sharpen.

### M2.1 integration tests legitimately break under M2.2's channel shift

Session 2026-04-23's engineer-listens_for shift (todo → in_dev) means M2.1's supervisor-level integration tests insert tickets at `column_slug='todo'` but nobody listens for the M2.1 channel anymore. The spec explicitly acknowledges this is an architectural change, not a regression. However, the test-harness update required to make them pass against the M2.2 binary wasn't captured as a separate task — T018's `TestM22M21RegressionStillPasses` is a placeholder with documentation. Future operator work to update the M2.1 test fixtures is tracked in acceptance-evidence.md's Criterion 13.

### The `runFakeAgent` path doesn't exercise M2.2 code

Under GARRISON_FAKE_AGENT_CMD, the supervisor takes the M1 `runFakeAgent` flow which uses the original `writeTerminal` (not `writeTerminalCostAndWakeup`), skips palace bootstrap, and skips hygiene goroutines. So fake-agent integration tests exercise M1 dispatcher plumbing only, not M2.2's new runtime. Anyone writing M2.2-specific fake-agent integration tests would miss the actual M2.2 code. T017 uses real-Claude-path-via-mockclaude specifically to avoid this gap.

## Spike pay-off audit

**(a) Did the M2 spike pay off for M2.2 specifically?** Mixed. The spike validated the claude binary, stream-json vocabulary, kill semantics, and the MemPalace MCP tool inventory. But it missed or misjudged five things that landed as T001 findings (palace init marker, wake-up flags, alpine-base, pip --require-hashes, socket-proxy transport). Of those five, four are fixable within M2.2 (the T001 pre-implementation spike was the mechanism); one (NFR-213 `--require-hashes` lockfile discipline) was a soft miss — not spec-breaking, but lockfile-vs-single-line was a cost the plan didn't anticipate. Spike pay-off rate: 5 big issues prevented (claude version pin, stream-json parse shape, MCP init field shape, 29-tool inventory, stdio per-invocation lifecycle) vs. 5 issues missed and caught at T001 instead of implement. Net positive, but the spike-vs-T001-spike boundary shifted some prevention into the pre-impl window.

**(b) Did T001 as a pre-implementation spike pay off?** Yes, decisively. All six findings (F1–F6) surfaced in T001 before any production code was written. Each would have cost 2–6 hours to fix during implement (rewriting bootstrap marker logic, changing Wakeup signature mid-code, debugging the alpine wheel-build failure, researching --require-hashes semantics, unwinding unix-socket assumptions across plan + main.go). T001 cost ~30 minutes of scratch-dir validation.

**(c) Did `total_cost_usd` match Anthropic's billing dashboard within 5%?** Deferred to the live-run append (Criterion 12).

**(d) Did the socket-proxy + sidecar topology hold up under chaos?** Partially — the init-failed path is chaos-tested (T019); the mid-run kill + pause/unpause cycles are live-acceptance items. All unit-level tests of the underlying code paths pass.

**(e) Surprises from running the full two-agent flow.** The `emit_ticket_transitioned` payload omission (missing `department_id`) was the biggest surprise — caught by T017's integration test, not by any unit test. Lesson: triggers with dynamic channel names + JSONB payloads deserve either a targeted pg-level smoke test or a schema pin in the migration's own test.

The `GARRISON_MEMPALACE_WAKEUP_FORCE_FAIL` test hook feels over-engineered in retrospect. A cleaner design would be to swap `mempalace.DockerExec` at wire-up time in main.go — but that's a bigger refactor than the test-env-flag shim.
