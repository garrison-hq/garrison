### Mutation primitives (T003 / T004 / T005 / T006)

## What the spec got wrong

### `hygiene_status` and `vault_access_log.outcome` are TEXT, not ENUM

The plan called for `ALTER TYPE … ADD VALUE` extensions on both;
in reality the M2.1 / M2.3 schema uses TEXT columns with
application-layer enum-like patterns. T001's migration body
adapts; the rollback complexity the plan flagged ("ALTER TYPE
ADD VALUE not reversible") is moot. Cleanly resolvable in-flight;
captured in T001's commit message.

### `vault_access_log.agent_instance_id` was NOT NULL

Designed for agent-driven access only. M4 operator mutations
have no agent_instance_id; T001b lands the nullability
relaxation. Actor identity goes in `metadata.actor_user_id` JSONB
to avoid a cross-domain FK from supervisor-owned
`vault_access_log` to dashboard-owned `users`.

### `garrison_dashboard_app` lacked vault write grants

M3's role split kept the app role off the vault tables. M4 needs
INSERT / UPDATE / DELETE on the three vault tables for
operator-driven mutations. T001c lands those grants; the read
role (`garrison_dashboard_ro`) stays SELECT-only.

### `agents` table had no `updated_at` column

Optimistic locking on agent.md edits (FR-101) needs a version
token; M2.1 didn't ship one. T001d adds the column + auto-bump
trigger.

### `agents.concurrency_cap` (FR-103) conflicts with RATIONALE Principle X

The spec called for per-agent concurrency cap as an editable
field. RATIONALE Principle X is explicit: "Per-department
concurrency caps bound parallelism. … Per-agent caps … are
rejected." M4 drops FR-103. The agent settings editor edits
agentMd / model / listensFor / skills only; concurrency_cap stays
on `departments`.

### M2.1 tickets schema doesn't carry priority / assigned-agent

Spec FR-031 named "title / description / priority / assigned-agent"
as the inline-edit field set. The M2.1 schema has `objective`
(~= title) + `acceptanceCriteria` (~= description) + no priority
+ no assigned-agent. The form maps onto what's there; future
milestones may extend the schema.

### SDK 5.0.2 doesn't expose Infisical-native rotation primitives

Plan T009 distinguished `infisical_native` from `manual_paste`
rotation paths. The SDK's surface treats both the same at the
call layer (`updateSecret(name, { … secretValue })`). M4 keeps
the `rotation_provider` tag for operator-facing UX (it drives
the rotation modal's label copy) but routes both through the
same SDK path. A future milestone may layer in genuine
Infisical-native rotation when the SDK exposes it.

## Dependencies added outside the locked list

One. Justified per AGENTS.md soft-rule:

**`@infisical/sdk@5.0.2`** — TS-side equivalent of the
supervisor's `infisical/go-sdk v0.7.1` (added at M2.3). Used by
the dashboard for vault writes through the parked-since-M2.3
`garrison-dashboard` Machine Identity. Alternatives — raw HTTP
against Infisical's REST API, hand-rolling Universal Auth refresh
— would reproduce the supervisor M2.3 decision in TS without the
SDK's automatic 401-refresh, the typed errors that map cleanly
to `VaultErrorKind`, or the rotation API surface for supported
backends. Rejected for the same reasons the supervisor uses the
Go SDK. MIT-licensed; small footprint.

The supervisor's locked-deps streak (M1 → M2.2.2: zero new deps)
remains broken since M2.3; M4 adds zero supervisor-side deps.

## Threat-model amendment retro questions answered

The threat-model amendment (commit `11b0dfb`) split §7 retro
questions across milestones. M4's three:

### Q4: Did the option-A UI feel coherent now that both halves shipped?

Manual review at retro time. Vault subsystem now has seven
distinct surfaces (list, create, edit, rotation, matrix, audit,
tree); the cumulative UX is dense but coherent. The
`isVaultConfigured()` short-circuit means non-vault surfaces
work without Infisical credentials, which makes onboarding less
brittle than the M2.3 retro feared.

### Q5: Did the operator actually rotate secrets on the surfaced schedule?

Pending operator-week-of-use evaluation. The dedicated
`/vault/rotation` dashboard groups stale + soon-stale rows;
inline `Rotate now` per row dispatches via the rotation modal.
Whether the cadence-driven workflow holds up is a usage
question; the surface is plumbed.

### Q6: Did the Garrison-side audit log prove more useful than Infisical's native log?

The dashboard now writes mutation rows alongside the M2.3 read
rows; both render through the same audit log viewer. The
context-joined queries (filter by ticket / by role / by time)
that motivated the dual-log decision in M2.3 are now exercised
across both read and write paths. Operator review of the
combined log over a week of mutations is the data point that
closes this question.
